## 0) 전역 규칙

### 0-1. 하단 탭(IA)

- **홈(HOME) / 플래너(PLANNER_WEEK) / 팬트리(PANTRY) / 마이페이지(MYPAGE)**

### 0-2. 공통 상태(리스트/데이터 화면)

- **Loading**: 스켈레톤/로딩 인디케이터
- **Empty**: 빈 상태 안내 + 다음 행동 CTA
- **Error**: 오류 안내 + [다시 시도]

### 0-3. 로그인 게이트(전역 패턴)

- 비로그인 사용자가 로그인 필요 기능(좋아요/저장/플래너/팬트리/장보기 완료/요리 완료 등)을 시도하면:
    1. 즉시 로그인 이동 X
    2. 안내 모달 + [로그인] [취소]
    3. 로그인 완료 후 **원래 하려던 작업으로 자동 복귀(return-to-action)**
- `LoginGateModal`의 색상, spacing, radius, sheet geometry는 Wave1 fixed prototype reference를 따르지만, 로그인 완료 후 pending action 복귀 동작은 MVP 서비스 계약을 유지한다.

### 0-4. 핵심 데이터 용어(화면에서 쓰는 개념)

- **Recipe(레시피)**: 재료+스텝+기본 인분
- **Meal(식사)**: 플래너에 등록된 "식사(레시피 + 계획 인분)"
- **MealStatus(식사 상태)**
    - `식사 등록 완료` → `장보기 완료` → `요리 완료`
- **ShoppingList(장보기 리스트)**: 취합 결과 체크리스트(저장됨)
- **Pantry(팬트리)**: 재료 보유 여부만(수량 저장 X)
- **IngredientCategory(재료 카테고리)**: v2 표시 기준은 8대분류/21소분류다. 화면은 shared taxonomy source에서 대분류와 소분류 label을 읽되, migration 동안 v1 canonical 8종(`채소/과일/육류/해산물/양념/유제품/곡류/기타`) 데이터도 fallback으로 표시한다.
    - `곡류/면/떡`: 밥/쌀, 면/파스타, 빵/떡/시리얼
    - `채소/버섯`: 잎/나물채소, 뿌리/줄기채소, 열매채소/버섯
    - `과일/견과`: 과일, 견과/씨앗/건과일
    - `단백질`: 돼지/소/양, 닭/오리, 달걀, 두부/콩류
    - `해산물`: 생선/갑각/조개, 해조/건어물/어묵
    - `유제품/대체유`: 우유/요거트/크림, 치즈/버터/대체유
    - `양념/조미`: 장류/소스, 향신료/허브, 기름/식초/당류/육수
    - `가공/기타`: 김치/절임/통조림, 냉동/간편식/음료/기타

### 0-5. 조리방법 시각 규칙(요리모드 스텝 카드)

- 볶기/볶아주기: **주황 계열**
- 끓이기/국물: **빨간 계열**
- 굽기/오븐/구이: **갈색 계열**
- 찌기/스팀: **파란 계열**
- 튀기기: **노란 계열**
- 데치기: **연두 계열**
- 무치기/버무리기: **초록 계열**
- 섞기/준비/기타: **회색 계열**
- 조리방법 taxonomy v2 표시 기준은 6그룹/20대표 method다. 화면은 `GET /cooking-methods`의 기존 `label`/`color_key`와 shared helper fallback을 계속 사용하고, category는 그룹 표시 또는 선택지 묶음에만 사용한다.
    - `준비/손질`: 썰기, 다지기
    - `전처리`: 해동, 밑간, 절이기
    - `물/수분 조리`: 끓이기, 삶기, 데치기, 찌기
    - `팬/기름 조리`: 볶기, 굽기, 부치기, 튀기기
    - `혼합/조림`: 섞기, 무치기, 조리기, 졸이기
    - `기기 조리`: 전자레인지, 오븐굽기, 에어프라이어
- `씻기`는 대표 method 선택지로 노출하지 않는다. `에어프라이어`는 대표 method 선택지로 노출한다.
- `label`에는 taxonomy 코드/분류 의미를 표시하지 않는다.

### 0-6. 장보기 합산 표시 규칙(변환 불가 단위)

- 변환 가능(kg↔g, L↔ml): 변환 후 합산
- 변환 불가(개 vs g, 망 vs g 등): **라인 분리 없이 "복합 표기만"**
    - 예: `양파 2개 + 200g + 1망`

### 0-7. 장보기 상세 공통 정책 `v1.2 추가`

- 장보기 목록 생성 후 유저는 `SHOPPING_DETAIL`로 즉시 이동한다.
- `SHOPPING_DETAIL`은 **구매 섹션 / 팬트리 제외 섹션** 두 영역으로 구성된다.
- 미완료 리스트에서는:
    - 구매 체크
    - 팬트리 제외/되살리기
    - 카드 순서 변경(드래그&드롭)

    이 가능하다.

- 팬트리 제외 섹션으로 이동한 아이템은 자동으로 **구매 체크 해제**
- 장보기 완료 후 리스트는 **read-only**
    - 재열람은 가능하지만 체크/정렬/제외 수정은 불가
- 공유 텍스트는 **구매 섹션 항목만 포함**하고 제외 섹션은 포함하지 않는다.

- YouTube 등록 레시피의 ingredient/step에 `component_label`이 있으면 재료/스텝 리스트에서 소제목으로 표시한다.

- `component_label`이 있으면 본문 `display_text`, `instruction` 앞의 같은 `[섹션명]` prefix는 중복 표시하지 않는다.

## 10) YT_IMPORT: 유튜브 레시피 가져오기/등록 `v1.3 contract-evolution`

- **Screen ID**: `YT_IMPORT`
- **권한**: 로그인 필요 (feature flag off → 404 FEATURE_DISABLED)
- **목적**: URL → oEmbed 미리보기 → YouTube videos.list 기반 3-way classification + description-first 추출 → 검수/수정 → 세션 기반 원자적 레시피 등록 → 플래너 추가

### 별도 진입면: Recipio-style Quick Import `2026-05-28 addendum`

- **Route**: `/recipes/new/youtube`
- **목적**: 레시피오처럼 링크 입력/추천 영상 선택 후 중복 확인, 영상 미리보기, 진행률 표시, 자동 등록까지 한 화면에서 처리한다.
- **재사용 계약**: 새 저장 API를 만들지 않고 기존 `validate` / `extract` / `register` 계약을 순차 소비한다.
- **중복 상태**:
    - `GET /recipes/youtube/recipio/check` 결과가 duplicate이면 저장된 레시피 카드, "레시피가 이미 존재해요" 문구, [레시피 보러가기] 링크를 표시한다.
    - duplicate 상태에서는 extract/register를 호출하지 않는다.
- **미리보기 상태**:
    - oEmbed 기반 `validate` 결과의 title/channel/thumbnail을 표시한다.
    - [이 영상으로 레시피 가져오기] 클릭 시 quick import를 시작한다.
- **진행률 상태**:
    - checking/extracting/registering/complete 단계별 UI progress를 표시한다.
    - progress는 client-side 상태 표현이며 서버 status endpoint를 새로 만들지 않는다.
- **검수 fallback 상태**:
    - 자동 등록 조건을 만족하지 않으면 추출된 재료/스텝 개수와 blocker 목록을 보여주고 기존 `/menu/add/youtube?youtubeUrl=...` 검수 화면으로 이동시킨다.
    - 다중 요리 후보(`multi_recipe_status=multiple`)는 자동 등록 대상이 아니며, 후보 선택이 필요한 blocker로 표시한다.
- **추천 영상 영역**:
    - 요즘 뜨는 레시피 카드 목록을 표시한다.
    - 추천 카드를 선택하면 해당 YouTube URL을 입력한 것과 동일하게 중복 확인부터 시작한다.
- **정책**:
    - feature flag off 시 `/recipes/new/youtube`도 404다.
    - 로그인/return-to-action 정책은 기존 `YT_IMPORT`와 동일하다.
    - 이 화면 자체는 신규 LLM/이미지 생성 UI 계약을 만들지 않는다. 추출 API는 공식 YouTube 추출 계약의 조건부 Gemini structured fallback을 따를 수 있다.

### 처리 파이프라인 (공개 텍스트 우선)

1. YouTube oEmbed 호출 → URL 유효성, title/channel/thumbnail 미리보기 확인 (YouTube Data API quota 미사용)
2. 추출 단계에서 YouTube Data API `videos.list` 호출 → snippet(title, description, tags, categoryId), contentDetails
3. 설명란 텍스트 파싱 → 재료/스텝 추출 → 표준화
4. 설명란이 불완전하면 공개 작성자 top-level 댓글 후보를 파싱해 누락된 재료/스텝만 보완
5. 설명란/댓글이 불완전하면 공개 caption timedtext를 파싱해 누락된 재료/스텝만 보완
6. 그래도 재료 또는 스텝이 부족하면 env/한도/cache/근거 검증을 통과한 Gemini structured fallback으로 공개 텍스트를 JSON 구조화
7. _(후속)_ OCR, ASR, 추정 레이어 — 현재 scope 밖

### 3-way classification

| classification_status | 의미 | UI 행동 |
| --- | --- | --- |
| `uncertain` | validate 단계의 oEmbed 미리보기만 완료, 요리 여부 미확정 | Step 2 extract 진행 |
| `recipe` | extract 단계에서 레시피 영상 확인 | 검수 화면 진행 |
| `non_recipe` | extract 단계에서 레시피 아님 | 차단 문구 + [다시 입력] |

- `classification_reasons`: string[] — 판정 근거 배열
- `non_recipe` 상태에서 [그래도 진행] 없음 (MVP에서 override 불가)

### UI 흐름

- Step 1) URL 입력 + [가져오기] → validate API
- Step 1.5) oEmbed 미리보기 결과
    - title/channel/thumbnail을 확인하고 Step 2로 자동 진행
    - 요리 영상 여부는 이 단계에서 확정하지 않음
- Step 2) 추출 진행 (indeterminate spinner/progress)
    - extract API 호출 → YouTube `videos.list` 기반 3-way classification → youtube_extraction_session 생성 (서버 draft 상태)
    - `non_recipe` 판정 시 추출 차단 + 다시 입력 안내
    - extraction_methods 표시: `description`은 “설명란”, `comment`는 “댓글”, `caption`은 “자막”으로 표시
    - Gemini structured fallback을 쓴 경우에도 source label은 실제 원천 텍스트 기준으로 표시하고, 필요 시 보조 배지 “AI 구조화 보조”만 별도로 표시한다.
    - 한 영상 안에서 여러 요리 후보가 감지되면 `multi_recipe_status=multiple`로 Step 3에 진입한다. 이때 top-level 재료/스텝은 비어 있고 후보 선택 전 저장은 비활성화된다.
    - 미분류 조리방법은 이 단계에서 즉시 생성 → 선택지 활용
- Step 3) 결과 검수/수정
    - 다중 후보가 있으면 검수 상단에 요리 후보 목록을 표시한다.
        - 각 후보는 후보명, 재료 개수, 만들기 단계 수를 표시한다.
        - 후보 선택 시 `POST /recipes/youtube/candidate-drafts`를 호출해 해당 후보를 단일 draft로 승격하고, 화면의 재료/스텝을 교체한다.
        - 후보 선택 중 loading 상태와 실패 error 상태를 표시한다.
        - `multi_parent` 세션 자체는 register/ingredient-registration 대상이 아니며, 후보 선택 후 생성된 `candidate_child` 세션만 등록 가능하다.
    - 레시피명 (편집 가능)
    - 기본 인분 (필수, 편집 가능)
    - 재료 리스트 (추가/삭제/수정)
        - 각 재료에 `resolution_status` 표시: `resolved` / `needs_review` / `unresolved`
        - `needs_review`: 경고 배지, candidate 선택/검색 교체/새 재료 등록 전 저장 차단
        - `unresolved`: 에러 배지(저장 차단 — 사용자가 검색 교체 또는 새 재료 등록으로 해소)
        - 서버 추출 row는 `draft_ingredient_id`를 가진다. 검수 중 row가 수정/재정렬되어도 새 재료 등록 요청에는 이 값을 사용한다.
        - [재료 검색으로 교체]: 기존 `GET /ingredients` 검색으로 표준 재료 선택
        - [새 재료로 등록]: 표준명, v1 canonical 8종 카테고리, optional synonym을 확인하는 sheet/modal을 열고 `POST /recipes/youtube/ingredient-registration` 호출
        - 등록 성공 시 응답의 `ingredient_id`와 canonical `standard_name`으로 현재 row를 `resolved`로 전환한다. 수량/단위/원문 표시는 유지한다.
        - 새 등록 실패 시 row는 기존 `needs_review` / `unresolved` 상태를 유지하고 error 메시지 + 재시도 액션을 표시한다.
        - `component_label`이 있으면 인접 항목 label 변경 지점에만 섹션 소제목을 표시한다.
        - 검수 중 수정/재정렬/등록 body 생성 시 `component_label`을 보존한다.
        - 본문 `display_text`에 같은 `[섹션명]` prefix를 중복 표시하지 않는다.
    - 스텝 리스트에도 같은 `component_label` 소제목 규칙을 적용한다.
    - 스텝 리스트 (추가/삭제/수정)
        - 각 스텝에 `is_incomplete` / `missing_fields` 표시
        - blocking(`instruction`, `cooking_method`): 빈값이면 저장 차단
        - warning(`duration`, `ingredients_used`): 빈값이면 경고만
    - 조리방법 자동 분류 결과 표시
    - 새로 생성된 조리방법은 `is_new: true` 라벨 구분
- **저장 활성화 조건**: 모든 재료 resolved + blocking incomplete 스텝 0건
- Step 3.5) 썸네일/태그 미리보기 `v1.5.11 추가`
    - 추출 세션의 썸네일 URL을 검수 화면 상단에 이미지로 표시한다. 로드 실패 시 placeholder를 표시한다.
    - 서버가 생성한 태그를 읽기 전용 chip으로 표시한다. 사용자는 태그를 수정/추가/삭제할 수 없다(MVP).
    - 태그가 비어 있으면 태그 영역을 표시하지 않는다.
    - 썸네일과 태그는 사용자가 편집할 수 없다. 등록 시 서버가 세션 값을 사용한다.
- **수량 provenance 배지** `v1.5.12 추가`:
    - 추출된 재료 행에 수량 출처 배지를 표시한다: `텍스트 확인`(text_explicit), `화면 확인`(visual_explicit), `단위 변환`(unit_normalized), `기본값`(ingredient_default), `추정`(recipe_inferred).
    - `quantity_review_required=true`인 재료 행은 확인 필요 표시와 함께 confirm/edit/clear 인터랙션을 제공한다.
    - confirm: 기존 제안 수량을 명시적으로 확인한다. edit: 수량을 직접 수정한다. clear: `맛에 맞게(TO_TASTE)`로 전환한다.
    - 모든 review-required 수량이 해결되기 전까지 저장 버튼은 비활성화된다.
- **quick import 수량 리뷰 차단** `v1.5.12 추가`:
    - 레시피오형 빠른 가져오기에서 `quantity_review_required=true`인 재료가 하나라도 있으면 auto-register를 차단하고 기존 YT_IMPORT 검수 화면으로 fallback한다.
- Step 4) [레시피 등록] → register API (세션 기반 원자적 RPC)
    - 완료 후 "이 끼니에 추가"(계획 인분 입력) → Meal 생성

### 세션 정책

- 서버가 extract 시 `youtube_extraction_sessions` 레코드 생성 (status=`draft`, 24h TTL)
- extract 응답과 session `draft_json.ingredients[]`에는 각 재료의 `draft_ingredient_id`가 포함된다.
- register 시 extraction_id 기반 소유권 검증 (cross-user → 404, expired → 410, consumed → 409)
- Provenance(recipe_sources)는 세션에서 복사 — 클라이언트 body 아님
- 미등록 재료 등록 API는 session 소유권과 `draft_ingredient_id`를 검증하지만 `draft_json`을 수정하지 않는다. 검수 화면 row resolved 전환은 클라이언트 로컬 상태 업데이트다.

### 정책

- 새 표준 재료 등록은 자동 실행하지 않고, 반드시 사용자 확인 후 수행한다.
- 사용자가 직접 추가한 client-only 재료 row는 이 흐름의 대상이 아니다. 이 슬라이스의 새 등록은 서버 추출 row(`draft_ingredient_id` 보유)에 한정한다.

### 1-a) 사용자 진도 표시 (compact) `user-progress 예정`

- **데이터 소스**: `GET /users/me/progress`
- **표시 위치**: 계정 섹션 내 닉네임/프로필 아래, 탭 영역 위
- **표시 내용**:
    - 현재 레벨 + 등급명 (예: `Steel · Lv.8`)
    - 진행도 바 (`progress_percent` 기반)
    - 다음 레벨까지 남은 XP (`xp_to_next_level`)
    - v2.1 등급 band는 `Clay`, `Wood`, `Steel`, `Silver`, `Gold`, `Diamond`, `Titanium`을 사용한다.
- **상태**:
    - Loading: progress 영역만 skeleton. MYPAGE 전체 loading에 연동하지 않음
    - Error (5xx/network): progress 영역만 숨김 또는 fallback 문구. MYPAGE 전체 error가 아님
    - Unauthorized (401): 기존 auth gate와 동일하게 fatal
    - 0 XP: 레벨 1, 진행도 바 0%
- **금지**:
    - 기존 하드코딩 레벨 문구 (`Wood · 레벨 5` 등)는 제거한다
    - 기존 MYPAGE 상단 stats(식사 수 등)와 progress event counts를 같은 수치로 혼동 표시하지 않는다
    - 33b에서 badge grid, quest list, XP toast는 표시하지 않는다
    - compact UI가 아닌 대형 progress card는 390px/320px evidence + authority pass 없이 도입하지 않는다

### 1-b) 사용자 성장/업적 앨범 표시 `growth-achievement-album`

- **데이터 소스**:
    - 레벨/XP compact display: `GET /users/me/progress`
    - 배지/업적 앨범/튜토리얼/notification/등급/archive preview: `GET /users/me/gamification`
    - 최근 성장 기록 전체: `GET /users/me/gamification/archive`
- **표시 위치**:
    - 계정 섹션 내부. 별도 대형 growth card가 아니라 프로필 header 안에 통합한다.
    - avatar / nickname / provider / 설정 버튼과 같은 프로필 맥락 안에서 grade, level, XP bar, 주요 기록, action buttons를 함께 보여준다.
    - 등급/업적/튜토리얼/알림 상세는 profile header 내부 버튼으로 여는 modal 또는 bottom sheet로 둔다.
    - 레시피북/장보기 기록 탭 진입을 첫 viewport 밖으로 과도하게 밀어내지 않는다.
- **표시 내용**:
    - 등급명 + 현재 레벨 + XP progress: 프로필 header의 핵심 정보로 표시
    - 주요 기록: 요리 기록, 플래너 기록, 장보기 기록 같은 compact count 3개 이내
    - Action buttons: `등급`, `업적`, `튜토리얼`, `알림`
    - 등급 modal: 7단계 등급 목록, 현재 등급 highlight, 각 등급 level range, spoon icon/character
    - 업적 앨범 modal: category tab, stamp grid, earned/active/locked 상태, locked hint
    - 튜토리얼 modal: 6개 튜토리얼 단계와 전체 완료 stamp
    - 알림 modal: 성장/업적/system filter, 최신순 archive list
    - XP/level/achievement/badge/quest toast stack: source action 성공 후 앱 shell 안전 영역 안에 표시
    - 잠긴 업적/배지 hint: category별 짧은 next-action 문구. 압박/경쟁/loot 표현 금지.
- **상태**:
    - Loading: gamification 영역만 skeleton. 33b progress와 MYPAGE core loading에 연동하지 않음
    - Empty: 0 XP/신규 사용자에게 `Clay · Lv.1`과 첫 튜토리얼 업적 표시
    - Error (5xx/network): gamification 영역만 soft-fail. MYPAGE core와 33b progress는 유지
    - Read-only: 업적/배지/튜토리얼 목록은 조회 중심이며 claim reward 버튼 없음
    - Unauthorized (401): 기존 auth gate와 동일하게 fatal
- **XP toast / notification trigger**:
    - 레시피 저장, 커스텀 레시피북 생성, 장보기 완료, 요리 완료, 플래너 등록 성공 후 notification refresh를 호출한다
    - toast 생성/seen 실패는 원래 action success를 실패로 바꾸지 않는다
    - 알림 우선순위는 `level_up > achievement_unlocked/badge_unlocked > quest_completed > xp_awarded`
    - 모바일은 동시에 최대 2개, 데스크톱은 동시에 최대 3개를 visible toast로 표시한다
    - 렌더링되지 않은 unseen notification은 다음 표시/보관함을 위해 queue에 남긴다
    - source action 성공 UI와 겹치지 않도록 모바일 하단 탭 위 safe area 또는 desktop lower-right에 표시한다
- **장보기 안내 문구**:
    - SHOPPING_FLOW intro/empty/preview surface 중 하나 이상에 `여러 끼니를 한번에 장보기할 수 있어요`를 표시한다.
    - `장보기 완료 N회`처럼 기준이 모호한 문구는 피하고, 리스트 완료와 끼니 묶음 완료를 구분한다.
- **배지/등급 visual**:
    - 배지 shape family는 `plate`, `shield`, `ribbon`, `bookmark`, `pot`, `leaf`, `bowl`을 기본 후보로 한다.
    - 등급 visual은 `Clay`, `Wood`, `Steel`, `Silver`, `Gold`, `Diamond`, `Titanium` spoon asset을 design source로 사용한다.
    - Runtime 적용 시 source PNG는 production 크기로 최적화하거나 equivalent CSS/SVG component로 대체할 수 있다. 단, grade key와 level band는 바꾸지 않는다.
    - Concept board는 다중 상태를 한 장에 보여주는 reference이며, 35c production UI는 실제 responsive user path로 재해석한다.
- **금지**:
    - leaderboard, competitive rank, 전체 사용자 순위
    - pressure streak, 실패/미접속 패널티
    - season reset, XP decay
    - loot/reward box, 랜덤 보상, 보상 claim CTA
    - locked badge 대형 grid를 MYPAGE 첫 viewport에 노출
    - 클라이언트 측 achievement unlock/badge unlock/quest 완료/XP 계산

## 20) RECIPEBOOK_DETAIL: 레시피북 상세

- **Screen ID**: `RECIPEBOOK_DETAIL`
- **권한**: 로그인 필요
- **목적**: 레시피북 안의 레시피를 책처럼 넘겨 읽고, 필요한 관리 액션을 수행

### UI 구성

- 웹:
    - 왼쪽 목차/책 정보 rail
    - 오른쪽 레시피북 리더
    - `책`/`목록` 전환
    - 책 모드 카드 정보: 썸네일, 제목, 태그, 조회수, 조리시간, 기본 인분, 재료, 만들기
- 모바일:
    - 상단 앱바 제목은 현재 레시피북 이름
    - 앱바 아래 공간은 레시피 목차 중심으로 사용
    - 현재 레시피 카드에는 썸네일, 제목, 메타, 재료, 만들기, 저장하기, 요리하기, 플래너 추가 액션을 표시
- 액션:
    - 레시피 탭 → `RECIPE_DETAIL`
    - 레시피 제거
    - 플래너 추가/저장하기가 직접 실행되지 않는 경우 기존 레시피 상세 진입으로 연결할 수 있다.

### 데이터

- 목록: `GET /recipe-books/{book_id}/recipes`
- 리더 상세: `GET /recipe-books/{book_id}/recipes/{recipe_id}`
- 리더 상세 조회는 `view_count`를 증가시키지 않는다.

### 제거 정책

- **좋아요한 레시피북(liked)** 에서 제거
    - 좋아요 해제 의미
- **저장한 레시피북(saved)** / **커스텀 레시피북(custom)**
    - 레시피북에서 제거 가능
- **내가 추가한 레시피북(my_added)**
    - 레시피북에서 제거 불가
    - 필요 시 별도 레시피 삭제 정책으로 처리(Phase 1 범위 밖 가능)

## 21) ADMIN_DASHBOARD: 관리자 대시보드

- **Screen ID**: `ADMIN_DASHBOARD`

- **권한**: 관리자 전용 (admin_members 테이블 등록 필수)
- **목적**: 내부 운영 관리 대시보드 — 관리자 전용 진입점. 일반 사용자 네비게이션에 노출되지 않음

- 사용자 통계 요약 (총 사용자 수, 최근 가입 등)
- 운영 이벤트 요약 (최근 에러 카운트 등)

- Loading: 스켈레톤/로딩 인디케이터
- Empty: 빈 상태 안내
- Error: 오류 안내 + [다시 시도]
- Read-only: 읽기 전용
- Unauthorized: 인증 실패 안내

### 정책

- 진입 시 `admin_page_view` 감사 로그 기록; 감사 기록 실패 시 fail closed (제어된 관리자 에러 상태)
- API: 관리자 인증은 서버 측 `requireAdminUser` 가드 사용
