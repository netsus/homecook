## 0) 전역 규칙

### 0-1. 하단 탭(IA)

- **홈(HOME) / 플래너(PLANNER_WEEK) / 팬트리(PANTRY) / 마이페이지(MYPAGE)**

### 0-2. 공통 상태(리스트/데이터 화면)

- **Loading**: 스켈레톤/로딩 인디케이터
- **Empty**: 빈 상태 안내 + 다음 행동 CTA
- **Error**: 오류 안내 + [다시 시도]

### 0-3. 로그인 게이트(전역 패턴)

- 비로그인 사용자가 로그인 필요 기능(좋아요/저장/플래너/팬트리/장보기 완료/요리 완료 등)을 시도하면:
    1. 즉시 로그인 이동 X
    2. 안내 모달 + [로그인] [취소]
    3. 로그인 완료 후 **원래 하려던 작업으로 자동 복귀(return-to-action)**
- `LoginGateModal`의 색상, spacing, radius, sheet geometry는 Wave1 fixed prototype reference를 따르지만, 로그인 완료 후 pending action 복귀 동작은 MVP 서비스 계약을 유지한다.

### 0-4. 핵심 데이터 용어(화면에서 쓰는 개념)

- **Recipe(레시피)**: 재료+스텝+기본 인분
- **Meal(식사)**: 플래너에 등록된 "식사(레시피 + 계획 인분)"
- **MealStatus(식사 상태)**
    - `식사 등록 완료` → `장보기 완료` → `요리 완료`
- **ShoppingList(장보기 리스트)**: 취합 결과 체크리스트(저장됨)
- **Pantry(팬트리)**: 재료 보유 여부만(수량 저장 X)
- **IngredientCategory(재료 카테고리)**: freeze 기간의 화면 표시/선택 기준은 legacy 7종(`채소/육류/해산물/양념/유제품/곡류/기타`)이다. 신규 taxonomy는 화면 label을 대체하지 않고 shared mapping source로만 해석한다.

### 0-5. 조리방법 시각 규칙(요리모드 스텝 카드)

- 볶기/볶아주기: **주황 계열**
- 끓이기/국물: **빨간 계열**
- 굽기/오븐/구이: **갈색 계열**
- 찌기/스팀: **파란 계열**
- 튀기기: **노란 계열**
- 데치기: **연두 계열**
- 무치기/버무리기: **초록 계열**
- 섞기/준비/기타: **회색 계열**
- 조리방법 category는 화면 필수 그룹이 아니다. 화면은 `GET /cooking-methods`의 `label`/`color_key`와 shared helper fallback을 사용하고, `label`에 taxonomy 코드/분류 의미를 표시하지 않는다.

### 0-6. 장보기 합산 표시 규칙(변환 불가 단위)

- 변환 가능(kg↔g, L↔ml): 변환 후 합산
- 변환 불가(개 vs g, 망 vs g 등): **라인 분리 없이 "복합 표기만"**
    - 예: `양파 2개 + 200g + 1망`

### 0-7. 장보기 상세 공통 정책 `v1.2 추가`

- 장보기 목록 생성 후 유저는 `SHOPPING_DETAIL`로 즉시 이동한다.
- `SHOPPING_DETAIL`은 **구매 섹션 / 팬트리 제외 섹션** 두 영역으로 구성된다.
- 미완료 리스트에서는:
    - 구매 체크
    - 팬트리 제외/되살리기
    - 카드 순서 변경(드래그&드롭)

    이 가능하다.

- 팬트리 제외 섹션으로 이동한 아이템은 자동으로 **구매 체크 해제**
- 장보기 완료 후 리스트는 **read-only**
    - 재열람은 가능하지만 체크/정렬/제외 수정은 불가
- 공유 텍스트는 **구매 섹션 항목만 포함**하고 제외 섹션은 포함하지 않는다.

## 1) HOME: 홈(레시피 탐색)

- **Screen ID**: `HOME`
- **권한**: 비로그인 가능
- **목적**: 레시피 탐색 → 레시피 상세 진입

### UI 구성

1. 공통 브랜드 헤더 (`HOMECOOK` 로고, `/` 링크)
2. discovery panel
    - 상단 검색바(placeholder: "레시피 제목 검색")
    - [재료로 검색] 버튼 → `INGREDIENT_FILTER_MODAL` (검색바 아래 단독 행)
3. 테마 carousel strip
    - compact horizontal carousel strip (높이 ~130px, 카드 1.5개 peek)
    - 가로 스크롤 affordance: 마지막 카드 0.5장 노출 + 우측 gradient hint
    - 헤더: "이번 주 인기" + "더보기 >" (tertiary CTA)
    - v1.3.1까지의 2열 카드 그리드 방식은 이 버전에서 폐기
4. `모든 레시피` 섹션
    - 카드 수
    - 정렬 컨트롤(기본: 조회수순)
5. 레시피 그리드(카드)

- 정렬 컨트롤은 검색바 아래가 아니라 `모든 레시피` 섹션 헤더에 둔다.

### 레시피 카드 정보

- 썸네일, source badge, 태그, 제목, 기본 인분 pill, 조회수, 좋아요 수, 저장 수

### 상호작용

- 카드 탭 → `RECIPE_DETAIL`

### 개별 식사 [요리하기] 버튼 `v1.5.4 추가`

- `shopping_done` 상태인 식사 카드에만 `[요리하기]` 버튼을 노출한다
- `registered` 상태 식사에는 노출하지 않는다 — 장보기를 우회하여 `cook_done`으로 전이할 수 없다
- `cook_done` 상태 식사에는 노출하지 않는다
- `[요리하기]` 클릭 시:
    1. 해당 식사 1건으로 planner 요리 세션을 생성한다 (`POST /cooking/sessions`, `meal_ids=[해당 meal_id]`)
    2. `COOK_MODE`로 진입한다
    3. 요리 완료 시: 해당 식사만 `shopping_done` → `cook_done` 전이. 남은요리 추가, 소진 재료 팬트리 반영은 기존 세션 완료와 동일
    4. 취소/뒤로가기 시: 상태 변경 없이 `MEAL_SCREEN`으로 복귀
- 이 경로는 기존 `COOK_READY_LIST` 일괄 요리와 독립적으로 동작하는 **planner 세션 단축 경로**이다
- 독립 요리(RECIPE_DETAIL → COOK_MODE)와 혼합되지 않는다 — cooking_session을 생성하며 meal 상태를 전이한다

### 인분/상태 변경 주의

- MealStatus가 `장보기 완료` 또는 `요리 완료`인 식사의 인분 변경 시:
    - 확인 모달: "상태가 진행된 식사입니다. 인분 변경 시 다시 장보기/요리 흐름이 필요할 수 있어요."
    - MVP 처리: 변경 허용 + 상태는 그대로 유지

### 정책

- 식사 카드는 상세 조회용 진입 역할이다.
- `shopping_done` 상태 식사에 한해 개별 `[요리하기]` 단축 경로를 제공한다 `v1.5.4 변경`

## 10) YT_IMPORT: 유튜브 레시피 가져오기/등록 `v1.3 contract-evolution`

- **Screen ID**: `YT_IMPORT`
- **권한**: 로그인 필요 (feature flag off → 404 FEATURE_DISABLED)
- **목적**: URL → oEmbed 미리보기 → YouTube videos.list 기반 3-way classification + description-first 추출 → 검수/수정 → 세션 기반 원자적 레시피 등록 → 플래너 추가

### 처리 파이프라인 (MVP description-first)

1. YouTube oEmbed 호출 → URL 유효성, title/channel/thumbnail 미리보기 확인 (YouTube Data API quota 미사용)
2. 추출 단계에서 YouTube Data API `videos.list` 호출 → snippet(title, description, tags, categoryId), contentDetails
3. 설명란 텍스트 파싱 → 재료/스텝 추출 → 표준화
4. _(후속)_ OCR, ASR, LLM, 추정 레이어 — MVP scope 밖 (extraction_methods 배열에 `description`만 기록)

### 3-way classification

| classification_status | 의미 | UI 행동 |
| --- | --- | --- |
| `uncertain` | validate 단계의 oEmbed 미리보기만 완료, 요리 여부 미확정 | Step 2 extract 진행 |
| `recipe` | extract 단계에서 레시피 영상 확인 | 검수 화면 진행 |
| `non_recipe` | extract 단계에서 레시피 아님 | 차단 문구 + [다시 입력] |

- `classification_reasons`: string[] — 판정 근거 배열
- `non_recipe` 상태에서 [그래도 진행] 없음 (MVP에서 override 불가)

### UI 흐름

- Step 1) URL 입력 + [가져오기] → validate API
- Step 1.5) oEmbed 미리보기 결과
    - title/channel/thumbnail을 확인하고 Step 2로 자동 진행
    - 요리 영상 여부는 이 단계에서 확정하지 않음
- Step 2) 추출 진행 (indeterminate spinner/progress)
    - extract API 호출 → YouTube `videos.list` 기반 3-way classification → youtube_extraction_session 생성 (서버 draft 상태)
    - `non_recipe` 판정 시 추출 차단 + 다시 입력 안내
    - extraction_methods 표시 (MVP: `["description"]`)
    - 미분류 조리방법은 이 단계에서 즉시 생성 → 선택지 활용
- Step 3) 결과 검수/수정
    - 레시피명 (편집 가능)
    - 기본 인분 (필수, 편집 가능)
    - 재료 리스트 (추가/삭제/수정)
        - 각 재료에 `resolution_status` 표시: `resolved` / `needs_review` / `unresolved`
        - `needs_review`: 경고 배지, candidate 선택/검색 교체/새 재료 등록 전 저장 차단
        - `unresolved`: 에러 배지(저장 차단 — 사용자가 검색 교체 또는 새 재료 등록으로 해소)
        - 서버 추출 row는 `draft_ingredient_id`를 가진다. 검수 중 row가 수정/재정렬되어도 새 재료 등록 요청에는 이 값을 사용한다.
        - [재료 검색으로 교체]: 기존 `GET /ingredients` 검색으로 표준 재료 선택
        - [새 재료로 등록]: 표준명, legacy 7종 카테고리, optional synonym을 확인하는 sheet/modal을 열고 `POST /recipes/youtube/ingredient-registration` 호출
        - 등록 성공 시 응답의 `ingredient_id`와 canonical `standard_name`으로 현재 row를 `resolved`로 전환한다. 수량/단위/원문 표시는 유지한다.
        - 새 등록 실패 시 row는 기존 `needs_review` / `unresolved` 상태를 유지하고 error 메시지 + 재시도 액션을 표시한다.
    - 스텝 리스트 (추가/삭제/수정)
        - 각 스텝에 `is_incomplete` / `missing_fields` 표시
        - blocking(`instruction`, `cooking_method`): 빈값이면 저장 차단
        - warning(`duration`, `ingredients_used`): 빈값이면 경고만
    - 조리방법 자동 분류 결과 표시
    - 새로 생성된 조리방법은 `is_new: true` 라벨 구분
- **저장 활성화 조건**: 모든 재료 resolved + blocking incomplete 스텝 0건
- Step 4) [레시피 등록] → register API (세션 기반 원자적 RPC)
    - 완료 후 "이 끼니에 추가"(계획 인분 입력) → Meal 생성

### 세션 정책

- 서버가 extract 시 `youtube_extraction_sessions` 레코드 생성 (status=`draft`, 24h TTL)
- extract 응답과 session `draft_json.ingredients[]`에는 각 재료의 `draft_ingredient_id`가 포함된다.
- register 시 extraction_id 기반 소유권 검증 (cross-user → 404, expired → 410, consumed → 409)
- Provenance(recipe_sources)는 세션에서 복사 — 클라이언트 body 아님
- 미등록 재료 등록 API는 session 소유권과 `draft_ingredient_id`를 검증하지만 `draft_json`을 수정하지 않는다. 검수 화면 row resolved 전환은 클라이언트 로컬 상태 업데이트다.

### 정책

- 수동 입력은 **검수 단계(Step 3)에서만** 허용
- 새 표준 재료 등록은 자동 실행하지 않고, 반드시 사용자 확인 후 수행한다.
- 사용자가 직접 추가한 client-only 재료 row는 이 흐름의 대상이 아니다. 이 슬라이스의 새 등록은 서버 추출 row(`draft_ingredient_id` 보유)에 한정한다.
- 미분류 조리방법은 추출 단계에서 먼저 생성되어 선택지로 활용 가능
- 식약처/농식품올바로 등 외부 데이터는 화면에서 production 재료로 자동 등록하지 않는다. 외부 데이터는 별도 staging/review/approved seed 경로를 통과한 뒤 검색/선택지에 반영된다.
- feature flag off 시 `/menu/add/youtube` 라우트 자체가 404 반환

### D. SETTINGS: 설정(계정 섹션 톱니바퀴로 진입)

- **Screen/Modal ID**: `SETTINGS`
- **진입**: MYPAGE 계정 섹션의 ⚙️ 버튼
- **형태**: 별도 화면 또는 바텀시트

### 설정 항목

- **요리모드 화면 꺼짐 방지** 토글(ON/OFF)
- **끼니 컬럼 관리**
    - 현재 컬럼 목록 표시
    - 컬럼 이름 변경
    - 컬럼 추가
    - 컬럼 삭제
    - 컬럼 수: 최소 1개, 최대 5개
    - 삭제 제한: 해당 컬럼에 등록된 식사가 있으면 삭제 불가 안내
    - 순서 변경은 1차 구현 범위 밖

### 계정 관련

- 로그아웃
- 회원탈퇴
    - 확인 다이얼로그 필수
    - 안내 문구: 레시피북, 플래너, 장보기, 팬트리 기록은 삭제되며 되돌릴 수 없음
    - 안내 문구: 직접 등록한 레시피는 작성자 정보 없이 남을 수 있음
    - 성공 후 HOME 이동
    - 같은 소셜 계정으로 다시 로그인해도 이전 개인 기록은 보이지 않음
- 닉네임 변경

## 21) ADMIN_DASHBOARD: 관리자 대시보드

- **Screen ID**: `ADMIN_DASHBOARD`
- **경로**: `/admin`
- **권한**: 관리자 전용 (admin_members 테이블 등록 필수)
- **목적**: 내부 운영 관리 대시보드 — 관리자 전용 진입점. 일반 사용자 네비게이션에 노출되지 않음

### UI 구성

- 관리자 인증 확인 (admin_members 테이블 조회)
- 미등록 시 403 Forbidden 표시 (fail closed)
- 사용자 통계 요약 (총 사용자 수, 최근 가입 등)
- 운영 이벤트 요약 (최근 에러 카운트 등)
- 각 관리 화면 네비게이션 링크 (ADMIN_USERS, ADMIN_EVENTS, ADMIN_AUDIT_LOGS)
- 후속 커뮤니티/신고/제재 네비게이션은 미구현 또는 disabled placeholder; 미구현 관리 페이지로 네비게이션하지 않음

### 상태

- Loading: 스켈레톤/로딩 인디케이터
- Empty: 빈 상태 안내
- Error: 오류 안내 + [다시 시도]
- Read-only: 읽기 전용
- Unauthorized: 인증 실패 안내
- forbidden(403): 선택적 추가 상태 (미등록 관리자 구분용)

### 정책

- 진입 시 `admin_page_view` 감사 로그 기록; 감사 기록 실패 시 fail closed (제어된 관리자 에러 상태)
- API: 관리자 인증은 서버 측 `requireAdminUser` 가드 사용
