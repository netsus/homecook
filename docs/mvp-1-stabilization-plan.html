합치기·분리 결정

표의 항목 중 같이 처리할 것과 따로 떼야 할 것을 명확히 나눕니다.

원래 항목 처리 이유
전체 회귀 검증 주간 + 완성도 높이기 + 내부 beta/UAT 묶기 같은 안정화 루프입니다. 순서는 내부 QA, 수정, RC, beta가 좋습니다.
모바일 디자인 수정 + 모바일·웹 용어 통일 순서 분리 용어 통일을 먼저 해야 이후 디자인 polish 중 문구가 다시 흔들리지 않습니다.
다시열기 기능 삭제 계약 변경 현재 공식 기능이므로 UI 수정처럼 삭제하면 안 됩니다. 문서 변경 승인 후 진행합니다.
Manual Only 정리 P0/P1 체크에 포함 출시 검증 구멍이므로 별도 후순위가 아니라 release checklist에 넣습니다.
레시피 DB 추가 P2로 이동 데이터 품질과 라이선스 검토가 있어 MVP 안정화 전에 넣으면 scope가 커집니다.
직접등록·유튜브 사진 등록 Post-MVP storage, 이미지 권한, 압축, 삭제 정책까지 필요해서 별도 v2 workpack이 맞습니다.
마케팅 기획·마케팅 실행 Beta 이후 실제로 써본 피드백이 있어야 메시지와 타깃을 정확히 잡을 수 있습니다.

Release Candidate 체크

RC 단계에서 확인할 세부 기준입니다. 별도 완료 버튼을 두지 않고 `Release Candidate 브랜치·태그` 시행 순서에서 한 번에 완료 처리합니다.

기능·계약

  • 문서 대비 route/API 누락 없음
  • 문서 밖 기능은 deferred 또는 contract-evolution 후보로 기록
  • 유튜브 핵심 흐름 실제 URL smoke 통과
  • Manual Only 항목의 처리 방침 기록

품질·검증

  • 전체 frontend verification 통과
  • 모바일 작은 화면 CTA 가림 없음
  • 로그인과 return-to-action 회귀 없음
  • known issues 문서 작성

운영·배포

  • Supabase migration/seed 확인
  • env와 OAuth 설정 문서화
  • 보안 smoke와 rollback 메모 준비
  • 피드백 수집 채널 준비
