## 0. 서비스 개요

- **서비스 한줄 설명**: ‘요리할 메뉴정하기 → 식단 구성 → 장보기목록 생성 → 장보기 → 요리’의 집밥 과정을 도와주는 **웹뷰앱 서비스**.
- **핵심 가치**
    1. 식단을 계획하면 장보기 목록이 자동 생성된다.
    2. 팬트리(집에 있는 재료)는 장보기 목록에서 자동 제외되며, 사용자가 되살릴 수 있다.
    3. 요리는 요리모드에서 한 눈에 보고, 완료 시 남은요리 기록과 팬트리 업데이트까지 연결된다.

## 1. MVP 범위(화면/기능 요구사항)

### 1-1. 홈(레시피 탐색)

- **진입**: 앱 실행/재방문 기본 진입 화면은 홈이다.
- **권한**: 비로그인 가능
- **기능**
    - 레시피 목록 + 테마 섹션 노출
    - **검색(기본)**: 제목 중심 검색
    - **재료 검색(추가)**: 검색창 아래 "재료 버튼" → 재료 검색 팝업 → 재료 다중 선택
        - 선택 재료들이 **모두 포함된 레시피**를 노출
    - **정렬**: 조회수순(기본), 최신순, 저장순, 플래너 등록순

---

### 1-2. 레시피 상세

- **진입**: 홈/레시피북/플래너(끼니 화면)에서 레시피 선택
- **권한**
    - 상세 조회/공유/요리하기 진입: 비로그인 가능
    - 좋아요/저장/플래너 추가: 로그인 필요
- **표시 정보**
    - 레시피명, 태그(AI 자동 생성), 재료 목록, 단계(스텝), 기본 인분
    - YouTube 등록 레시피에 섹션 라벨이 있으면 재료/단계 목록에 소제목을 표시한다. 소제목은 인접 항목의 `component_label`이 바뀌는 지점에만 표시한다.
    - `component_label`이 있으면 본문 `display_text`/`instruction`에는 같은 `[섹션명]` prefix를 중복 표시하지 않는다.
    - 지표: 조회수(상세 진입), 저장 수, 좋아요 수, 플래너 등록 수, 요리완료 수
- **액션**
    - 공유: Web Share + 링크 복사
    - 좋아요(토글)
    - 저장(레시피북 선택/생성)
        - 저장 대상은 **저장한 레시피북(saved)** 또는 **커스텀 레시피북(custom)** 이다.
        - 한 번에 여러 `saved/custom` 레시피북을 선택해 저장할 수 있다.
    - 플래너 추가(날짜/끼니 선택)
- **인분 조절**
    - 레시피 상세에서 인분 조절 가능(실시간으로 재료량 반영)
    - 조절한 인분은 "요리하기 진입 시 기본값"으로 사용될 수 있다(정책은 공통 정책 참조)

### 1-3. 요리하기(요리모드)

- **진입**: 레시피 상세 / 플래너 요리하기 리스트에서 "요리하기" / 끼니 화면(MEAL_SCREEN)에서 개별 식사 "요리하기" `v1.6.7 추가`
- **권한**
    - 요리하기 화면 진입: 비로그인 가능
    - 요리완료 기록 저장/팬트리 업데이트: 로그인 필요

### UI 구성

- **전체화면 모드**: 요리모드 진입 시 전체화면으로 전환
- **화면 꺼짐 방지**: 요리모드 진입 시 화면이 자동으로 꺼지지 않음 (설정에서 on/off 가능)
- **좌우 스와이프**: 유일한 주요 터치 인터랙션
- **왼쪽 페이지**: 레시피에 사용되는 재료 전체 목록 + 인분 표시
- **오른쪽 페이지**: 스텝 카드가 아래로 쭉 이어짐 (스크롤 최소화)
- YouTube 등록 레시피의 재료/단계에 `component_label`이 있으면 요리모드에서도 같은 소제목 규칙을 유지한다.

### 스텝 카드

- **카드 1개 = 스텝 1개**
- 카드 내용: 넣을 재료/양, 썰기 크기, 조리법, 불세기(🔥), 조리시간
- **조리방법 색상 구분**: 스텝 카드의 테두리 또는 배경 색상으로 조리방법을 한눈에 구분
    - 볶기: 주황 계열
    - 끓이기: 빨간 계열
    - 굽기: 갈색 계열
    - 찌기/스팀: 파란 계열
    - 튀기기: 노란 계열
    - 데치기: 연두 계열
    - 무치기/버무리기: 초록 계열
    - 섞기/준비/기타: 회색 계열

### 인분 조절 정책

- **요리하기 화면에서는 인분 조절 불가**
- 요리하기는 "진입 전 설정된 인분(플래너 인분 또는 상세에서 설정한 인분)"으로 진행한다.

### 요리완료

- "요리 완료" / "취소" 버튼 존재

### 플래너(위클리)

- **주간 범위 바 + 하루 카드 리스트** 구조
- **주간 이동**: 요일 스트립을 좌우로 넘겨 이전 주 / 다음 주로 이동하고, 필요 시 `이번주로 가기`로 기본 범위로 복귀
- 하루 카드 1개 안에 사용자가 설정한 끼니 컬럼을 모두 노출
- 신규 사용자의 기본 끼니 컬럼은 **아침 / 점심 / 저녁 3개**이다
- 끼니 컬럼은 설정 화면에서 이름 변경, 추가, 삭제할 수 있다
- 끼니 컬럼은 최소 1개, 최대 5개까지 허용한다
- 끼니 컬럼 삭제는 해당 컬럼에 등록된 식사가 없을 때만 허용한다
- 기존 사용자에게 이미 생성된 컬럼은 자동 삭제하지 않고, 설정 화면에서 사용자가 직접 정리할 수 있게 한다
- 같은 날짜 식단은 한 카드 안에서 함께 읽힌다
- Baemin prototype planner reference와 동일한 스크롤/탐색 모델로 일주일 식단을 탐색한다
- planner 내부 localized horizontal scroll, swipe, peek affordance는 prototype reference와 일치하는 경우 허용한다
- 범위 메타데이터는 한 번만 노출하고, 개발용 상태 문자열은 사용자 화면에 노출하지 않는다
- 작은 모바일에서도 같은 레이아웃 비율을 유지한 채 폰트/여백/카드가 함께 축소되어야 한다
- 끼니 칸 클릭 → **끼니 화면** 진입
- 남은요리는 색으로 구분 표시
- **식사 상태 표시**: 각 끼니 칸에 식사 상태(등록/장보기/요리)를 시각적으로 구분
- **상단 버튼**: 장보기, **요리하기, 남은요리**

### 장보기 진입 flow

플래너 상단 "장보기" 버튼 클릭 시:

1. **대상 자동 선택**: 오늘~마지막 등록일 중 **"식사 등록 완료" 상태이면서 아직 다른 장보기 리스트에 묶이지 않은 식사**의 인분을 자동 합산
2. **합산 리스트 표시**: 인분이 합산된 레시피들이 선택된 상태로 리스트에 노출 → 사용자가 선택 해제 가능
3. **장보기 기준 인분 조정**: 레시피별 장보기 기준 인분을 조절 가능
4. **장보기 목록 생성**: 선택 확정 → 재료 취합/합산 → 팬트리 자동 제외 처리 → 장보기 리스트 생성
5. **목록 생성 후 자동 이동**: 생성 즉시 `SHOPPING_DETAIL`로 이동
6. **장보기 완료 처리**:
    - 해당 식사들의 상태가 **"식사 등록 완료" → "장보기 완료"**로 변경
    - 완료 직전 “팬트리에 추가할 아이템 선택” 팝업을 통해 팬트리 반영 여부를 고를 수 있음

### 요리하기 진입 flow

#### A. 플래너 상단 "요리하기" 버튼 (일괄/레시피 그룹 요리)

플래너 상단 "요리하기" 버튼 클릭 시:

1. **대상 자동 선택**: 오늘~마지막 등록일 중 **"장보기 완료"** 상태인 식사들을 인분 합산하여 리스트로 표시
2. **레시피별 요리하기**: 리스트의 각 레시피 옆에 "요리하기" 버튼 존재
3. **요리하기 클릭** → 해당 레시피 1개 단위 세션 생성 → 요리모드(1-3) 진입
4. **요리 완료 시**: 해당 레시피의 "장보기 완료" 상태 식사들이 **"요리 완료"** 상태로 변경 → 남은요리 리스트에 자동 추가
    - 장본 인분보다 적거나 많은 인분으로 요리해도 동일하게 처리

#### B. 끼니 화면(MEAL_SCREEN) 개별 식사 "요리하기" 버튼 (단축 경로) `v1.6.7 추가`

끼니 화면에서 **"장보기 완료(shopping_done)"** 상태인 개별 식사에 `[요리하기]` 버튼을 제공한다.

1. **대상 조건**: 해당 식사의 상태가 `shopping_done`인 경우에만 `[요리하기]` 버튼을 노출한다
2. **`[요리하기]` 클릭** → 해당 식사 1건 + 해당 식사의 recipe_id로 planner 요리 세션을 생성한다
    - 기존 `POST /cooking/sessions` 계약을 `meal_ids=[선택된_meal_id]`로 호출하는 단축 경로이다
3. **요리모드(COOK_MODE) 진입** → 기존 세션 기반 요리모드와 동일하게 동작한다
4. **요리 완료 시**: 해당 식사만 `shopping_done` → `cook_done`으로 전이한다. 남은요리 자동 추가, 소진 재료 팬트리 반영은 기존과 동일하다
5. **취소/뒤로가기**: 상태 변경 없이 끼니 화면(MEAL_SCREEN)으로 복귀한다

**제한 사항**:
- `식사 등록 완료(registered)` 상태 식사는 `[요리하기]` 대상이 아니다. 장보기를 우회하여 `cook_done` 상태로 전이할 수 없다
- `요리 완료(cook_done)` 상태 식사에는 `[요리하기]` 버튼을 노출하지 않는다
- 이 경로는 **planner 요리 세션의 단축 경로**이며, 독립 요리(⑧)와 혼합되지 않는다

### 1-5. 식사추가(끼니 화면 내)

- **공통**
    - 식사 등록 시 **인분 입력 필수**
    - 사용자가 많이 쓸 순서(우선순위): `유튜브 → 검색 → 남은요리 → 레시피북 → 팬트리만 → 직접등록`
- **끼니 화면 UI 배치**
    - 상단: 검색창(검색해서 추가)
    - 버튼 영역(바둑판): 유튜브 링크 / 레시피북 / 남은요리 / 팬트리만 이용
    - 직접 등록: 작게 별도 노출
- **식사 추가 옵션**
    1. **유튜브 링크로 추가(MVP 필수)**
    2. 팬트리 재료만 이용(추천: 1차는 DB 필터링/스코어링)
    3. 레시피 검색해서 추가
    4. 직접 레시피 등록(수동)
    5. 레시피북에서 추가
    6. 남은요리에서 추가

### 1-6. 장보기 `v1.6 업데이트`

- **권한**: 로그인 필요
- **진입**: 플래너 상단 "장보기" 버튼 (1-4 장보기 진입 flow 참조)

### 핵심 기능

- 플래너에서 "식사 등록 완료" 상태 식사들의 인분 자동 합산 → 선택/해제 → 장보기 목록 생성
- 재료 취합/합산 → 팬트리 자동 제외 처리
- 장보기 목록은 **저장**되며 마이페이지에서 재열람 가능
- 쿠팡/컬리 버튼: "검색 링크" 수준으로 제공

### 장보기 상세 화면 정책

장보기 목록 생성 후 진입하는 상세 화면에서는 다음이 가능해야 한다.

1. **구매 섹션**
    - 실제 장볼 아이템 카드 리스트
    - 체크박스로 구매 완료 표시
    - 카드 드래그&드롭으로 순서 변경 가능
2. **팬트리 제외 섹션**
    - 팬트리에 있어 자동 제외된 재료를 별도 섹션으로 표시
    - 유저가 되살리고 싶으면 구매 섹션으로 다시 이동 가능
3. **섹션 간 이동 규칙**
    - 구매 섹션 → 팬트리 제외 섹션 이동 시 해당 항목은 자동으로 구매 체크 해제
    - 팬트리 제외 섹션 → 구매 섹션 이동 시 다시 장보기 대상으로 복귀
4. **공유**
    - 장보기 공유 텍스트는 **구매 섹션 항목만 포함**
    - 팬트리 제외 섹션 항목은 공유 대상에서 제외

### 장보기 완료 처리

- 장보기 목록에서 사용자가 **실제로 구매한 항목을 체크**
- [장보기 완료] 버튼 클릭 시:
    1. 해당 식사들의 상태 → **"장보기 완료"**로 변경
    2. "팬트리에 추가할 아이템 선택" 팝업 노출
        - 기본값: 구매 체크된 항목 전체 선택
        - 사용자가 체크 해제로 팬트리 반영 제외 가능
    3. 선택된 항목만 팬트리에 추가
- 팬트리에 아무것도 추가하지 않고 장보기 완료만 하는 것도 가능
- 완료 후 장보기 리스트는 **read-only**
    - 재열람은 가능하지만 체크/정렬/제외 수정은 불가

### 1-11. 내부 운영 관리 (Admin Foundation) `v1.7.2 추가`

- **접근 경로**: `/admin` (내부 운영자 전용, 일반 사용자 네비게이션에 노출되지 않음)
- **권한**: OAuth 로그인 + `admin_members` 테이블 등록 확인. 미등록 시 403 Forbidden
- **최초 admin 등록**: Supabase SQL 또는 service-role API로 직접 등록. 환경변수 허용목록(allowlist) 우회 없음
- **MVP 범위**: 읽기 전용
    - 관리 대시보드 (사용자 통계 요약, 운영 이벤트 요약, 관리 화면 네비게이션)
    - 사용자 목록 조회 (승인된 요약 정보만)
    - 운영 이벤트 로그 조회
    - 관리자 감사 로그 조회
- **금지 동작**: 사용자 정지·삭제·수정, 레시피 삭제, 비밀번호 재설정 등 파괴적 관리 동작은 MVP 범위 밖
- **PII 최소화**: 사용자 목록은 승인된 요약 정보만 표시 (user id, 마스킹된 이메일, provider, nickname, created_at, 고수준 카운트/상태). 원문 소스 텍스트, OAuth 토큰, YouTube URL, 비공개 장보기/팬트리 상세, 관리자 검색어는 로그에 기록하지 않음
- **운영 이벤트 로깅**: `operational_events` 테이블에 시스템 수준 이벤트 기록. 최소 이벤트 소스: OAuth/인증 실패, YouTube 프로바이더 실패, 계정 삭제 성공/실패, Admin API service-role 누락 실패, 선별된 라우트 핸들러 미처리 서버 에러. `request_path`는 pathname만 저장하고 쿼리스트링은 저장하지 않음
- **감사 로깅**: `admin_audit_logs` 테이블에 관리자 행위 기록. 모든 `/api/v1/admin/*` 읽기는 감사 기록 작성. `/admin` 페이지 진입 시 `admin_page_view` 감사 기록 작성; 실패 시 fail closed. 관리자 검색어는 저장하지 않음 (target_type='user_search', target_id=null 사용). `ip_hash`, `user_agent_hash`는 해시 값만 저장
- **API 패턴**: `/api/v1/admin/*` 엔드포인트는 `createServiceRoleClient()` 사용 필수. service role 없을 시 fail closed (제어된 서버 에러). `routeClient` 폴백 금지. 응답 형식은 기존 `{ success, data, error }` 래퍼 유지
- **화면 상태**: loading / empty / error / read-only / unauthorized 필수. forbidden(403)은 선택적 추가 상태
- **Scope 밖**: 사용자 삭제/정지/위장, 역할 관리 UI, 콘텐츠 CRUD, 커뮤니티 신고/제재, 외부 모니터링 의존성 도입

## 2. 공통 정책(데이터/UX/로직)

### 2-1. 용어

- **레시피(Recipe)**: 재료(이름+양+단위)와 스텝을 가진 단위
- **식사(Meal)**: 플래너에 등록된 식사 인스턴스. 레시피인데, 상태를 가진다(2-8 참조)
- **팬트리(Pantry)**: 재료 "보유 여부"만 저장
- **장보기 목록(ShoppingList)**: 선택 식사로부터 재료를 취합한 체크리스트(저장됨)

---

### 2-2. 온보딩/첫 사용

- 팬트리 입력은 **선택사항**이며 로그인 시 제공 가치(장보기 제외 등)를 어필한다.
- 튜토리얼은 MVP 제외(필요 시 2차)

---

### 2-3. 로그인 게이트(안내 + 작업 복귀)

- 비로그인 사용자가 로그인 필요 기능에 접근하면:
    1. 즉시 로그인 화면 이동 X
    2. 안내 메시지 + 선택 버튼 제공
    3. 로그인 완료 후 **원래 하려던 작업으로 자동 복귀**
- 비로그인 상태에서 둘러본 기록(좋아요/저장 등)은 저장하지 않는다.

---

### 2-4. 유튜브 레시피 추출 정책

- 유튜브 레시피는 **등록 시점에 재료/양이 확정된 상태**로 저장한다.
- **추출 파이프라인 (순서대로 적용)**:
    1. **설명란 파싱** → 재료/양/단계 표준화
    2. **공개 작성자 댓글 후보 파싱** → 설명란이 불완전할 때 작성자 top-level 댓글만 보조 source로 사용
    3. **공개 자막/timedtext 파싱** → 설명란/댓글이 불완전할 때 공개 caption track 텍스트를 보조 source로 사용
    4. **OCR(재료표 화면)** → 영상 내 재료표 이미지에서 텍스트 추출
    5. **ASR 전사** → 자막이 부실한 영상 커버
    6. **추정 레이어 고도화** → 위 단계에서 누락/불완전한 수량 보완
        - 1차: 관용량 테이블(통계 기반)
        - 2차: 유사 레시피 Retrieval
        - 3차: 수량 예측 모델
- **URL 검증 단계 (3-way classification)** `v1.6.8`:
    - 유튜브 URL 입력 직후, 해당 영상이 요리/레시피 영상인지 3단계로 판정한다.
    - `recipe`: 설명란에 재료/조리 단계가 명확 → 추출 진행, 경고 없음
    - `uncertain`: 음식 관련이지만 구조화된 레시피가 불분명 → 추출 진행 가능, careful-review 경고 표시
    - `non_recipe`: 요리와 무관한 강한 증거 → 추출 차단, 다른 URL 입력 요청 (MVP에서 override 없음)
    - Classifier는 보수적: 약한/혼합 증거는 `uncertain`, 강한 비요리 증거만 `non_recipe`
- **추출 세션 관리** `v1.6.8`:
    - 추출은 서버에서 세션(`youtube_extraction_sessions`)을 생성하여 관리한다.
    - 세션은 24시간 후 만료된다. 같은 URL이어도 매번 새 세션을 생성한다(MVP).
    - 등록은 세션 소유권 검증 + Postgres RPC 원자적 트랜잭션으로 수행한다.
- **다중 레시피 후보 선택** `2026-05-30 addendum`:
    - 한 영상 안에 여러 요리가 있는 경우 공개 설명란/작성자 댓글/caption timedtext에서 일반 규칙으로 후보를 분리한다.
    - 추출 결과는 `multi_parent` 세션과 `recipe_candidates[]`를 반환하고, 사용자가 저장할 후보를 먼저 선택해야 한다.
    - 후보 선택 시 `candidate_child` 세션을 생성해 기존 검수/미등록 재료 등록/register 흐름을 재사용한다.
    - 특정 video_id별 하드코딩, 레시피오 결과 반환, 유료 provider/LLM 호출은 금지한다.
    - 공개 텍스트에서도 후보별 재료/단계를 만들 수 없으면 임의 생성하지 않고 검수 필요/지원 불가 상태로 남긴다.
- **재료 해결 상태** `v1.6.8`:
    - 추출된 재료는 `resolved`, `needs_review`, `unresolved` 상태를 가진다.
    - 모든 재료가 `resolved`이고 blocking step field가 채워져야 등록이 가능하다.
- **미등록 재료 등록** `v1.6.9`:
    - 추출된 재료가 DB 표준명/동의어로 매칭되지 않으면 검수 화면에서 `unresolved`로 표시한다.
    - 사용자는 기존 재료 검색으로 교체하거나, 서버 추출 row(`draft_ingredient_id` 보유)에 한해 새 표준 재료로 등록할 수 있다.
    - 새 재료 등록은 자동 실행하지 않고 반드시 사용자 확인을 거친다.
    - 등록 시 표준명, 카테고리, optional synonym을 검증하고 `ingredients` / `ingredient_synonyms`를 RPC로 처리한다.
    - slice27 전 freeze 기간의 등록 카테고리는 기존 7종 문자열(`채소`, `육류`, `해산물`, `양념`, `유제품`, `곡류`, `기타`)만 허용한다.
    - 이미 같은 `standard_name`이 있으면 새 row를 만들지 않고 기존 ingredient를 재사용한다.
    - 같은 synonym이 다른 ingredient에 이미 있으면 ambiguous 확산을 막기 위해 synonym 저장을 skip하고 warning만 반환한다.
    - 등록 성공 후 session `draft_json`은 원본 추출 snapshot으로 유지하고, 클라이언트가 현재 검수 row를 `resolved`로 전환한다.
- **레시피오형 빠른 가져오기 모드** `2026-05-28 addendum`:
    - `/recipes/new/youtube`는 기존 `YT_IMPORT`와 별도 진입면으로 제공한다.
    - URL 입력 또는 추천 영상 선택 시 기존 레시피 중복 여부를 먼저 확인한다.
    - 중복이면 크레딧/추출 비용이 드는 작업 없이 저장된 레시피 카드와 상세 이동을 제공한다.
    - 중복이 아니면 기존 `validate` → `extract` → `register` 계약을 순차 호출해 자동 등록을 시도한다.
    - 모든 재료가 `resolved`이고 blocking step field가 없을 때만 자동 등록한다.
    - `needs_review`/`unresolved` 재료, 필수 step 누락, `non_recipe`, provider error는 자동 등록하지 않고 기존 검수/재입력 흐름으로 fallback한다.
    - 진행률은 실제 비동기 단계 상태를 보여주는 UI 값이며, 서버에 새 progress/status endpoint를 만들지 않는다.
    - 신규 LLM/이미지 생성/외부 유료 provider는 이 모드의 계약에 포함하지 않는다.
    - 특정 YouTube video_id에 대해 레시피오에서 생성된 레시피 데이터를 코드 fixture로 반환하지 않는다. 중복 확인을 제외한 신규 추출은 항상 provider/parser 경로를 거친다.
- **섹션 라벨 영속화** `v1.7.3`:
    - description parser가 감지한 재료/단계 섹션명은 extract 응답의 `component_label`로 반환한다.
    - YouTube register는 `component_label`을 `recipe_ingredients`, `recipe_steps`에 nullable 값으로 저장한다.
    - `component_label`이 있으면 저장되는 `display_text`, `instruction`은 같은 섹션명 prefix를 포함하지 않는다.
    - 기존 legacy row에서 `component_label` 없이 `[섹션명]` prefix만 있는 경우에는 현상 유지한다.
    - 수동 레시피 작성 계약은 이번 범위에서 섹션 라벨 입력을 지원하지 않는다.
- **YouTube Data API** `v1.6.8`:
    - URL 입력 직후 미리보기는 YouTube oEmbed로 title/channel/thumbnail만 조회한다(YouTube Data API quota 미사용).
    - 실제 추출 단계에서만 `videos.list` (part=snippet,contentDetails)를 호출한다: 1 quota unit, title/description/tags/category/thumbnails/duration/caption flag 조회
    - 요리 영상 여부와 공개 텍스트 레시피 존재 여부는 `videos.list` 응답과 이후 공개 텍스트 source를 확인한 추출 단계에서 판정한다.
    - `captions.list`/`captions.download`: 비용/권한 제약으로 현재 미사용
    - 공개 caption track의 timedtext 조회는 설명란/작성자 댓글 보조 source로 허용한다. 로그인/소유자 권한이 필요한 caption download, 오디오 ASR, 비디오 OCR은 이 범위에서 사용하지 않는다.
- **사용자 수동 입력**: 위 파이프라인으로 추론 완료 후, 결과를 사용자에게 보여주고 수정/보완을 받는다.
- (참고: OCR, ASR, 추정 레이어는 단계적으로 구현. 현재 YouTube 추출은 설명란 + 공개 작성자 댓글 + 공개 timedtext 자막 + 수동 보완)

### 2-6. 재료 표준화/단위/타입

- 재료 표준명은 재료 마스터 사전(표준명+동의어)로 매핑
- 재료 카테고리 v1 canonical은 freeze 기간 동안 기존 7종 문자열(`채소/육류/해산물/양념/유제품/곡류/기타`)이다.
- 신규 ingredient taxonomy는 기존 `ingredients.category`를 대체하지 않는다. 코드/seed 기반 shared mapping source나 additive shadow metadata로만 해석 계층을 둔다.
- `GET /ingredients?category=<legacy label>`와 YouTube 미등록 재료 등록의 category validation은 legacy 7종 label 계약을 유지한다.
- 단위 표준 그룹: g/kg, ml/L, 개/ea, 큰술(tbsp), 작은술(tsp), TEXT_UNIT(컵/꼬집/줌/망 등)
- 재료 타입: QUANT(정량), TO_TASTE(비정량)
- UNKNOWN 타입은 사용하지 않는다(정량 재료는 등록 시점에 반드시 수량/단위 확정)
- 조리방법 category는 현재 MVP 필수 계약이 아니라 optional additive metadata다. `cooking_methods.label`은 짧은 표시 라벨 용도이며 taxonomy 코드/분류 저장소로 사용하지 않는다.
- 외부 공공/상용 데이터(식약처, 농식품올바로 등)는 production canonical ingredient로 직접 넣지 않는다. raw import → normalization → review → 승인 seed 경로를 거친 뒤에만 반영한다.

### 2-7. 장보기 취합/합산 규칙

- 합산 규칙:
    1. 같은 재료(표준명) + 같은 단위 → 수치 합산
    2. 변환 가능 단위(kg↔g, L↔ml) → 변환 후 합산
    3. 변환 불가 단위(개 vs g, 망 vs g 등) → **복합 표기만 사용**
        - 예: `양파 2개 + 200g + 1망`
    4. TEXT_UNIT은 같은 문자열끼리만 합산(예: 1줌+2줌=3줌)

### 2-8. 식사 상태 & 상태 전이

### 식사(Meal) 상태 정의

플래너에 레시피를 등록하면 "식사(Meal)"가 생성되며, 아래 3가지 상태를 가진다.

| 상태 | 설명 | 전이 조건 |
| --- | --- | --- |
| **식사 등록 완료** | 플래너에 식사가 등록된 초기 상태 | 식사 등록 시 자동 |
| **장보기 완료** | 해당 식사의 재료를 장봤음 | 장보기 완료 처리 시 |
| **요리 완료** | 해당 식사의 요리를 했음(= 남은요리 상태) | 요리완료 처리 시 |

### 상태 전이 flow

```
[식사 등록] → 식사 등록 완료
                  │
                  ▼  (장보기 완료 클릭)
             장보기 완료
                  │
                  ▼  (요리 완료 클릭)
              요리 완료 → 남은요리 목록에 자동 추가
```

### 장보기 대상

- 장보기 버튼 클릭 시 **"식사 등록 완료"** 상태인 식사들만 대상
- 단, **이미 다른 장보기 리스트에 묶인 식사(shopping_list_id 존재)는 제외**
- 장보기 목록 생성 시 선택된 식사는 해당 장보기 리스트에 먼저 연결된다.
- 장보기 완료 처리 시 대상 식사들이 일괄 **"장보기 완료"**로 전이된다.

### 요리하기 대상

- 요리하기 버튼 클릭 시 **"장보기 완료"** 상태인 식사들만 대상
- 요리 완료 처리 시 해당 레시피의 "장보기 완료" 식사들이 일괄 **"요리 완료"**로 전이
- 장본 인분과 실제 요리 인분이 달라도 동일하게 처리(상태 전이는 인분 무관)
- `MEAL_SCREEN` 개별 식사 `[요리하기]`도 동일한 `shopping_done` 조건을 적용한다. `registered` 상태 식사는 장보기를 거치지 않고 `cook_done`으로 전이할 수 없다 `v1.6.7 추가`

### 팬트리/남은요리 연동

- **장보기 완료 시**
    - 식사는 `shopping_done`으로 전이
    - 팬트리 반영은 팝업에서 선택된 항목에 한해 수행
- **요리 완료 시**
    - 소진 재료 체크리스트(기본 체크 해제) → 체크된 재료만 팬트리에서 제거
    - 남은요리 목록에 자동 추가

### 2-12. 장보기 상세 화면 정책 `v1.6 추가`

- 장보기 목록 생성 후 유저는 **즉시 장보기 상세 화면으로 이동**
- 장보기 상세 화면은 **구매 섹션 / 팬트리 제외 섹션** 두 영역으로 구성된다.
- 미완료 장보기 리스트에서는 다음이 가능하다:
    - 구매 체크
    - 팬트리 제외/되살리기
    - 카드 순서 변경(드래그&드롭)
- 팬트리 제외 섹션으로 이동한 항목은 자동으로 구매 체크가 해제된다.
- 장보기 완료 후 리스트는 **read-only**
    - 기록은 `다시열기`로 재열람 가능하지만 수정은 불가
    - 완료된 기록은 완료 시각을 함께 표시할 수 있다.

### 2-13. 레시피북 정책 `v1.6 추가`

- 시스템 레시피북은 다음 3개이다.
    - 내가 추가한 레시피
    - 저장한 레시피
    - 좋아요한 레시피
- 커스텀 레시피북은 유저가 생성/수정/삭제할 수 있다.
- 저장 액션은 **저장한 레시피북(saved)** 또는 **커스텀 레시피북(custom)** 에만 수행한다.
- 저장 액션은 여러 `saved/custom` 레시피북을 한 번에 선택할 수 있다.
- 레시피북 상세 카드는 태그, 조회수, 조리시간, 기본 인분 정보를 표시한다.
- 좋아요한 레시피북은 좋아요 상태를 기반으로 구성된다.
- 내가 추가한 레시피북은 직접 등록/유튜브 등록한 레시피를 기반으로 구성된다.

### 2-14. 관리자 접근 정책 `v1.7.2 추가`

- 관리자 기능은 내부 운영 전용이며, 일반 사용자에게 노출되지 않는다.
- `admin_members` 테이블이 관리자 신원의 단일 진실 소스이다.
- 관리자 API는 반드시 `createServiceRoleClient()`를 사용하며, service role 부재 시 fail closed 한다. `routeClient` 폴백은 관리자/교차 사용자 읽기에 금지된다.
- 관리자 행위는 예외 없이 `admin_audit_logs`에 기록한다.
- PII 최소화 원칙: 승인된 요약 정보만 표시하며, 민감 정보(OAuth 토큰, YouTube URL, 검색어, 비공개 장보기/팬트리 상세)는 로그에 기록하지 않는다.
- 운영 이벤트(`operational_events`)와 감사 로그(`admin_audit_logs`)의 `request_path`는 pathname만 저장한다. 쿼리스트링은 첫 버전에서 저장하지 않는다.
