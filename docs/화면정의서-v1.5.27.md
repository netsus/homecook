> **2026-07-21 contract-evolution — 레시피 재료 혼합 표기**
>
> | # | 변경 내용 | 영향 범위 |
> | --- | --- | --- |
> | 1 | RECIPE_DETAIL과 COOK_MODE는 원본 수량을 먼저 표시하고 승인된 부피→질량 실측 근거가 있을 때만 `1큰술 · 약 18g`처럼 보조 g을 병기한다 | 재료 목록 |
> | 2 | 내부 소수 실측값은 유지하되 화면은 1g 단위 반올림, 0~1g은 `<1g`으로 표시한다 | 표시 정밀도 |
> | 3 | 원본 `g/kg`, `TO_TASTE`, 근거 없는 부피, exact size 근거 없는 `개/장`에는 보조 g을 표시하지 않는다 | fail-closed display |
> | 4 | 보조 환산 조회가 실패해도 원본 수량·재료·조리 순서와 기존 action은 정상 표시하며 보조 g만 생략한다 | soft degrade |
>
> 이 화면 결정은 아래 같은 날의 `recipe 재료에 보조 g을 병기하지 않는다`는 임시 경계를 대체한다. 보조 g은 작은 보조 정보이며 원본 수량보다 시각적 위계를 높이지 않고, 내부 evidence code·밀도·legacy `VOLUME_G*` 식별자를 노출하지 않는다.

> **2026-07-16 contract-evolution — Recipe Detail 영양 가용성 상태 분리**
>
> | # | 변경 내용 | 영향 범위 |
> | --- | --- | --- |
> | 1 | nutrition `availability_reason='missing'`은 current snapshot row가 실제로 없는 정상 준비 상태로 표시한다 | RECIPE_DETAIL unavailable |
> | 2 | `availability_reason='temporarily_unavailable'`은 snapshot 조회 자체가 일시 실패한 상태로 표시하고 기존 상세 본문은 유지한 채 영양 영역에만 재시도를 제공한다 | RECIPE_DETAIL soft error |
> | 3 | snapshot row를 정상 조회했으면 `availability_reason=null`이며, 그 row의 `calculation_status=complete/partial/unavailable`을 별도로 표시한다 | 상태 의미 분리 |

> 이 변경은 새 화면·endpoint·HTTP status·error code를 만들지 않는다. 기존 `GET /recipes/{id}` 성공 payload의 additive nullable field만 소비하며, 영양 조회 실패가 제목·CTA·재료·조리 단계를 막지 않게 한다. 정량 재료의 `amount + unit`은 실제 투입 가식부 사용량이고 조리 손실은 미모델링 한계로 안내한다.

## 0) 전역 규칙

### 0-0. 서비스명과 이미지 브랜드 자산

- 법적 고지, SEO metadata, 서비스 최초 설명에는 정식명 `무엇을 먹든`을 사용한다.
- 공식 이미지 심볼은 파란색 둥근 사각형 안의 흰색 `무먹` 글자이며 선택 완료 source와 그 파생 export만 사용한다.
- AppBar와 좁은 내비게이션에는 공식 심볼과 짧은명 `무먹`을 함께 사용한다.
- HOME만 예외로 모바일 `HomeAppBar`와 desktop HOME `WebTopNav` 브랜드 영역에 공식 심볼 옆으로 `무먹` 아래 작은 `무엇을 먹든`을 세로 2단 표시한다. 다른 화면의 공통 AppBar/WebTopNav에는 정식명 보조 줄을 추가하지 않는다.
- 심볼 옆에 시각 텍스트가 있으면 이미지 `alt`는 비워 중복 낭독을 막고 링크/heading의 접근성 이름은 시각 텍스트 계약과 일치시킨다.
- favicon은 투명 외곽 export를 사용한다. 설치/PWA·Apple touch 아이콘은 흰색 외곽 모서리가 없는 full-bleed 파란 배경 export를 사용하며, OG/Twitter·가로형·흑백 버전은 기존 생성 완료본을 유지한다.
- `homecook:*`, `HOMECOOK_*`, cookie/header/event/storage/package/repository/Supabase/OMO/stored key는 화면 copy 변경 대상이 아니다.
- 사용자 콘텐츠와 일반명사 `집밥`은 일괄 치환하지 않는다.
- screenshot, 미선택 후보, 따옴표·점·장식·마스코트가 포함된 변형은 사용하지 않는다.

### 0-1. 하단 탭(IA)

### 0-2. 공통 상태(리스트/데이터 화면)

- **Loading**: 스켈레톤/로딩 인디케이터
- **Empty**: 빈 상태 안내 + 다음 행동 CTA
- **Error**: 오류 안내 + [다시 시도]
- **Not Found**: `페이지를 찾을 수 없어요` 제목, `홈으로`/`플래너로` 복구 CTA, 짧은 인라인 피드백 입력 영역을 제공한다.
  - 피드백 영역은 모달로 선노출하지 않고 본문 아래 작은 카드로 둔다.
  - 사용자는 `어떤 버튼이나 링크를 눌렀는지`를 600자 안으로 남길 수 있다.
  - 화면은 전송 중/성공/실패 상태를 같은 자리에서 보여준다.
  - 개인정보 입력을 유도하지 않으며, 현재 path/referrer path/로그인 여부/사용자 또는 익명 식별자/user agent hash/발생 시각은 서버가 자동 기록한다.

### 0-3. 로그인 게이트(전역 패턴)

- 비로그인 사용자가 로그인 필요 기능(좋아요/저장/플래너/팬트리/장보기 완료/요리 완료 등)을 시도하면:
    1. 즉시 로그인 이동 X
    2. 안내 모달 + [로그인] [취소]
    3. 로그인 완료 후 **원래 하려던 작업으로 자동 복귀(return-to-action)**
- `LoginGateModal`의 색상, spacing, radius, sheet geometry는 Wave1 fixed prototype reference를 따르지만, 로그인 완료 후 pending action 복귀 동작은 MVP 서비스 계약을 유지한다.

### 0-4. 핵심 데이터 용어(화면에서 쓰는 개념)

- **Recipe(레시피)**: 재료+스텝+기본 인분
- **Meal(식사)**: 플래너에 등록된 "식사(레시피 + 계획 인분)"
- **FoodProduct(완제품)**: 별도 조리 없이 먹는 포장식품·간편식. 공공 검수 제품, 다른 사용자가 공개한 `사용자 등록` 제품, 또는 본인 legacy private manual 제품
- **ProductPlannerEntry(완제품 계획 항목)**: 날짜/끼니/수량과 당시 영양 version을 고정한 항목. MealStatus 없음
- **PlannerEntry**: 화면에서만 Recipe Meal과 완제품 항목을 `recipe / product` 유형으로 합쳐 표시하는 read model
- **계획 영양**: 고정 snapshot 합산 예상치. 실제 섭취나 의료 조언이 아님
- **MealStatus(식사 상태)**
    - `식사 등록 완료` → `장보기 완료` → `요리 완료`
- **ShoppingList(장보기 리스트)**: 취합 결과 체크리스트(저장됨)
- **Pantry(팬트리)**: 재료 보유 여부만(수량 저장 X)
- **IngredientCategory(재료 카테고리)**: v2 표시 기준은 8대분류/21소분류다. 화면은 shared taxonomy source에서 대분류와 소분류 label을 읽되, migration 동안 v1 canonical 8종(`채소/과일/육류/해산물/양념/유제품/곡류/기타`) 데이터도 fallback으로 표시한다.
    - `곡류/면/떡`: 밥/쌀, 면/파스타, 빵/떡/시리얼
    - `채소/버섯`: 잎/나물채소, 뿌리/줄기채소, 열매채소/버섯
    - `과일/견과`: 과일, 견과/씨앗/건과일
    - `단백질`: 돼지/소/양, 닭/오리, 달걀, 두부/콩류
    - `해산물`: 생선/갑각/조개, 해조/건어물/어묵
    - `유제품/대체유`: 우유/요거트/크림, 치즈/버터/대체유
    - `양념/조미`: 장류/소스, 향신료/허브, 기름/식초/당류/육수
    - `가공/기타`: 김치/절임/통조림, 냉동/간편식/음료/기타

### 0-5. 조리방법 시각 규칙(요리모드 스텝 카드)

- 볶기/볶아주기: **주황 계열**
- 끓이기/국물: **빨간 계열**
- 굽기/오븐/구이: **갈색 계열**
- 찌기/스팀: **파란 계열**
- 튀기기: **노란 계열**
- 데치기: **연두 계열**
- 무치기/버무리기: **초록 계열**
- 섞기/준비/기타: **회색 계열**
- 조리방법 taxonomy v2 표시 기준은 6그룹/20대표 method다. 화면은 `GET /cooking-methods`의 기존 `label`/`color_key`와 shared helper fallback을 계속 사용하고, category는 그룹 표시 또는 선택지 묶음에만 사용한다.
    - `준비/손질`: 썰기, 다지기
    - `전처리`: 해동, 밑간, 절이기
    - `물/수분 조리`: 끓이기, 삶기, 데치기, 찌기
    - `팬/기름 조리`: 볶기, 굽기, 부치기, 튀기기
    - `혼합/조림`: 섞기, 무치기, 조리기, 졸이기
    - `기기 조리`: 전자레인지, 오븐굽기, 에어프라이어
- `씻기`는 대표 method 선택지로 노출하지 않는다. `에어프라이어`는 대표 method 선택지로 노출한다.
- `label`에는 taxonomy 코드/분류 의미를 표시하지 않는다.

### 0-6. 장보기 합산 표시 규칙(변환 불가 단위)

- 변환 가능(kg↔g, L↔ml): 변환 후 합산
- 변환 불가(개 vs g, 망 vs g 등): **라인 분리 없이 "복합 표기만"**
    - 예: `양파 2개 + 200g + 1망`

### 0-7. 장보기 상세 공통 정책 `v1.2 추가`

- 장보기 목록 생성 후 유저는 `SHOPPING_DETAIL`로 즉시 이동한다.
- `SHOPPING_DETAIL`은 **구매 섹션 / 팬트리 제외 섹션** 두 영역으로 구성된다.
- 미완료 리스트에서는:
    - 구매 체크
    - 팬트리 제외/되살리기
    - 카드 순서 변경(드래그&드롭)

    이 가능하다.

- 팬트리 제외 섹션으로 이동한 아이템은 자동으로 **구매 체크 해제**
- 장보기 완료 후 리스트는 **read-only**
    - 재열람은 가능하지만 체크/정렬/제외 수정은 불가
- 공유 텍스트는 **구매 섹션 항목만 포함**하고 제외 섹션은 포함하지 않는다.

### 0-8. 영양 표시 공통 규칙 `v1.5.21 추가`

- 우선 표시 영양소는 열량·탄수화물·단백질·지방·나트륨 5종이다. 당류·포화지방·식이섬유는 값이 있을 때만 보조 표시한다.
- 계산과 표시에 사용하는 원천은 pinned predecessor가 승인하고 current로 지정한 source/profile/assignment뿐이다. 미승인·철회·과거 version이나 runtime public API 응답을 화면 계산에 섞지 않는다.
- `complete`: 예상값을 표시한다. `calculation_quality`가 `estimated` 또는 `mixed`이면 `예상`을 함께 표시한다.
- `partial`: `known_amount`만 `최소 X`로 표시하고 반영 수/대상 수와 누락 이유를 상세에서 제공한다.
- `unavailable`: amount는 `null`이며 `0` 대신 `정보 준비 중`을 표시하고 계산 품질 배지를 만들지 않는다.
- 대표 환산 등급으로 변환한 g은 정수 반올림하고 `약`을 붙인다. 소수점 실측 밀도나 `VOLUME_G*` 내부 code는 사용자 화면에 노출하지 않는다.
- source 표시는 기관명·dataset·version/기준일·license·원본 링크의 안전한 attribution만 제공한다. API key, 인증 query, raw provider response는 표시하거나 DOM/client state에 포함하지 않는다.
- 모든 합계에는 `계획 영양` label을 사용한다. `cook_done`을 실제 섭취로 해석하거나 목표 달성률·질환 조언을 추가하지 않는다.

### 섹션 라벨 표시 `v1.5.10`

- YouTube 등록 레시피의 ingredient/step에 `component_label`이 있으면 재료/스텝 리스트에서 소제목으로 표시한다.
- 소제목은 현재 항목의 label이 이전 인접 항목 label과 다를 때만 표시한다.
- 기존 순서를 재정렬하지 않는다.
- `component_label`이 없으면 기존 flat list를 유지한다.
- `component_label`이 있으면 본문 `display_text`, `instruction` 앞의 같은 `[섹션명]` prefix는 중복 표시하지 않는다.

### 인분 조절 규칙

- 레시피 상세에서 인분을 변경하면 **재료량 표기가 즉시 변경**
- **요리모드에서는 인분 조절 불가** → 요리모드 진입 전 확정된 값으로 진행
- 예상 영양 카드의 1인분 기준값은 고정하고, 선택 인분 전체값은 snapshot의 `scalable vector × selected_servings / base_servings + fixed vector`로만 갱신한다. `1인분 값 × selected_servings` shortcut으로 fixed vector를 잘못 늘리지 않는다.

### 예상 영양 카드 `v1.5.21 추가`

- 카드 제목은 `1인분 기준 예상 영양`이다.
- 핵심 5종을 우선 표시하고 optional 영양소는 값이 있는 경우만 펼침 영역에 표시한다.
- `complete`: 숫자를 표시하되 `estimated/mixed`이면 `예상` 배지를 유지한다.
- `partial`: `최소 X`, `8/10개 재료 반영` 같은 reflected count, 누락 사유를 함께 표시한다.
- `availability_reason='missing'`: current snapshot이 없는 정상 상태다. 숫자 대신 `영양 정보 준비 중`을 표시하고 0 kcal 또는 조회 오류로 표현하지 않는다.
- `availability_reason='temporarily_unavailable'`: snapshot 조회가 일시 실패한 상태다. 기존 Recipe Detail 전체를 오류 처리하지 않고 영양 카드 안에만 안내와 `[다시 시도]`를 제공해 같은 영양 조회를 재실행한다.
- `availability_reason=null`: snapshot row를 정상 조회한 상태다. 이때 `calculation_status='unavailable'`이면 snapshot의 결측 사유·warning에 따른 `영양 정보 준비 중`을 표시하며 일시 조회 실패로 오인시키지 않는다.
- generic 승인 계량값 사용 시 `재료 투입량과 승인된 계량값을 기준으로 계산한 예상치예요` 설명을 제공한다.
- 정량 재료의 양은 조리에 실제 투입한 가식부 사용량 기준임을 안내한다. 조리 손실·국물 폐기·튀김유 흡수 등 오차가 큰 경우 immutable snapshot의 `warnings[]`를 세부 설명에 표시한다.

    - `homecook:last-auth-provider:v1`의 유효한 provider를 우선 표시하고, 값이 없을 때만 기존 `homecook-last-auth-provider` cookie를 fallback으로 사용한다.
    - provider 버튼에 `최근 로그인` 보조 라벨과 시각 강조를 표시할 수 있으나 이메일·닉네임 등 개인 정보는 표시하지 않는다.
    - 안내 문구는 “이 브라우저에서 최근 사용” 의미이며 현재 사용자나 계정 소유를 증명하지 않는다.
- 다른 provider 확인창:
    - 선택 provider가 기억한 provider와 다를 때만 OAuth 전에 표시한다.
    - primary action: 기억한 provider로 로그인
    - secondary action: 선택한 provider로 다른 계정 로그인 계속
    - 취소, ESC, backdrop 닫기 시 OAuth 호출 없음. 닫힌 뒤 선택했던 provider 버튼으로 focus 복귀
    - 선택 provider와 기억 provider가 같으면 확인창 없이 OAuth를 시작한다.
- 로그인 오류 상태:
    - `email_required`: “이 서비스는 이메일 제공 동의가 필요해요. 제공자 설정에서 이메일 동의 후 다시 시도해주세요.”
    - `account_conflict`: “계정을 안전하게 확인하지 못했어요. 기존 로그인 방법으로 다시 시도해주세요.”
    - 오류 문구에는 이메일, Supabase user id, token, authorization code, provider payload를 표시하지 않는다.
- 닉네임 입력(신규 회원인 경우). 입력이 없거나 trim 후 빈 값이면 `무먹러`를 fallback으로 표시하고, 기존 저장 nickname은 변경하지 않는다.
- 로그인 성공 후: 원래 하려던 액션으로 자동 복귀
- 마지막 성공 provider는 일반 로그인 완료 후에만 갱신한다. 버튼 클릭, 확인창 열기/취소, OAuth 실패, 수동 provider 연결은 값을 바꾸지 않는다.
- 실제 provider는 `app_metadata.provider` 단독 값이 아니라 검증된 provider attempt와 callback user identity의 sign-in evidence가 일치할 때만 성공값으로 사용한다. 판정이 모호하면 최근 provider 안내를 새로 쓰지 않는다.

## 8-a) FOOD_PRODUCT_PICKER: 완제품 검색해서 추가

- **Component ID**: `FOOD_PRODUCT_PICKER`
- **권한**: 로그인 필요
- **형태**: `MENU_ADD`에서 진입하는 modal/sheet 또는 nested full-height sheet
- **목적**: 승인 `공공 영양DB` 제품, `사용자 등록` public 제품, 본인 legacy private 제품을 검색하고 수량을 정해 플래너에 추가

### UI

- 검색 입력 + source filter(`전체 / 공공 영양DB / 사용자 등록`) + cursor pagination 결과 목록
- 결과 범위는 `공공 영양DB + 사용자 등록 public + current user legacy private`이며 다른 사용자의 private 제품은 존재 여부도 노출하지 않는다.
- 제품 카드: 제품명, 브랜드, source(`공공 영양DB` / `사용자 등록` / `비공개 보관`), 비교 기준량, 원 라벨 기준량, 핵심 영양 상태, 예상 열량
- 제품 선택 후 수량 입력: `100g`, `101g`, `250mL`, `1회`처럼 현재 immutable nutrition version과 호환되는 단위만 선택한다. g/mL 기준 제품은 기본값 `100`, step `1g/1mL`를 사용한다.
- 동일 basis 안의 배수 계산은 허용한다. `serving/package/g/ml` 간 교차 변환은 현재 version의 승인된 `basis_relations[]`에 정확한 `{ from: { amount, unit }, to: { amount, unit } }` 관계가 있을 때만 선택지로 노출한다. 관계가 없으면 추정 환산하지 않고 `422 NUTRITION_BASIS_MISMATCH`를 `이 기준으로는 수량을 바꿀 수 없어요` validation으로 표시한다.
- 추가 완료 후 `MEAL_SCREEN`으로 복귀하며 Recipe Meal 상태를 만들지 않는다.
- 사용자 등록 public 제품 카드에는 `신고` action을 제공한다. 본인 제품, legacy private 제품, `공공 영양DB` 제품에는 `신고`를 노출하지 않는다.
- 검색 결과가 없으면 `[직접 등록]`으로 `FOOD_PRODUCT_CREATE`를 연다.

### 상태

- loading: 결과 skeleton
- empty: `찾는 완제품이 없어요` + `[직접 등록]`
- error: inline retry
- unauthorized: 로그인 게이트 후 검색어/선택 날짜/끼니 return-to-action 복귀
- `공공 영양DB` 제품과 다른 사용자의 `사용자 등록` public 제품은 read-only다. 본인 visible manual 제품만 수정/삭제 진입 가능하다.
- 신고 또는 운영 숨김된 `사용자 등록` 제품은 새 검색/추가에서 제외하되 기존 planner entry는 pin된 이름·영양을 계속 표시한다.
- soft-deleted 제품도 새 검색/추가에서 제외하되 기존 planner entry는 pin된 이름·영양을 계속 표시한다.

## 10) YT_IMPORT: 유튜브 레시피 가져오기/등록 `v1.3 contract-evolution`

- **Screen ID**: `YT_IMPORT`
- **권한**: 로그인 필요 (feature flag off → 404 FEATURE_DISABLED)
- **목적**: URL → oEmbed 미리보기 → YouTube videos.list 기반 3-way classification + description-first 추출 → 검수/수정 → 세션 기반 원자적 레시피 등록 → 플래너 추가

### 별도 진입면: Recipio-style Quick Import `2026-05-28 addendum`

- **Route**: `/recipes/new/youtube`
- **목적**: 레시피오처럼 링크 입력/추천 영상 선택 후 중복 확인, 영상 미리보기, 진행률 표시, 자동 등록까지 한 화면에서 처리한다.
- **재사용 계약**: 새 저장 API를 만들지 않고 기존 `validate` / `extract` / `register` 계약을 순차 소비한다.
- **중복 상태**:
    - `GET /recipes/youtube/recipio/check` 결과가 duplicate이면 저장된 레시피 카드, "레시피가 이미 존재해요" 문구, [레시피 보러가기] 링크를 표시한다.
    - duplicate 상태에서는 extract/register를 호출하지 않는다.
- **미리보기 상태**:
    - oEmbed 기반 `validate` 결과의 title/channel/thumbnail을 표시한다.
    - [이 영상으로 레시피 가져오기] 클릭 시 quick import를 시작한다.
- **진행률 상태**:
    - checking/extracting/registering/complete 단계별 UI progress를 표시한다.
    - progress는 client-side 상태 표현이며 서버 status endpoint를 새로 만들지 않는다.
- **검수 fallback 상태**:
    - 자동 등록 조건을 만족하지 않으면 추출된 재료/스텝 개수와 blocker 목록을 보여주고 기존 `/menu/add/youtube?youtubeUrl=...` 검수 화면으로 이동시킨다.
    - 다중 요리 후보(`multi_recipe_status=multiple`)는 자동 등록 대상이 아니며, 후보 선택이 필요한 blocker로 표시한다.
- **추천 영상 영역**:
    - 요즘 뜨는 레시피 카드 목록을 표시한다.
    - 추천 카드를 선택하면 해당 YouTube URL을 입력한 것과 동일하게 중복 확인부터 시작한다.
- **정책**:
    - feature flag off 시 `/recipes/new/youtube`도 404다.
    - 로그인/return-to-action 정책은 기존 `YT_IMPORT`와 동일하다.
    - 이 화면 자체는 신규 LLM/이미지 생성 UI 계약을 만들지 않는다. 추출 API는 공식 YouTube 추출 계약의 조건부 Gemini structured fallback을 따를 수 있다.

### 처리 파이프라인 (공개 텍스트 우선)

1. YouTube oEmbed 호출 → URL 유효성, title/channel/thumbnail 미리보기 확인 (YouTube Data API quota 미사용)
2. 추출 단계에서 YouTube Data API `videos.list` 호출 → snippet(title, description, tags, categoryId), contentDetails
3. 설명란 텍스트 파싱 → 재료/스텝 추출 → 표준화
4. 설명란이 불완전하면 공개 작성자 top-level 댓글 후보를 파싱해 누락된 재료/스텝만 보완
5. 설명란/댓글이 불완전하면 공개 caption timedtext를 파싱해 누락된 재료/스텝만 보완
6. 그래도 재료 또는 스텝이 부족하면 env/한도/cache/근거 검증을 통과한 Gemini structured fallback으로 공개 텍스트를 JSON 구조화
7. _(후속)_ OCR, ASR, 추정 레이어 — 현재 scope 밖

### 3-way classification

| classification_status | 의미 | UI 행동 |
| --- | --- | --- |
| `uncertain` | validate 단계의 oEmbed 미리보기만 완료, 요리 여부 미확정 | Step 2 extract 진행 |
| `recipe` | extract 단계에서 레시피 영상 확인 | 검수 화면 진행 |
| `non_recipe` | extract 단계에서 레시피 아님 | 차단 문구 + [다시 입력] |

- `classification_reasons`: string[] — 판정 근거 배열
- `non_recipe` 상태에서 [그래도 진행] 없음 (MVP에서 override 불가)

### UI 흐름

- Step 1) URL 입력 + [가져오기] → validate API
- Step 1.5) oEmbed 미리보기 결과

- Step 2) 추출 진행 (indeterminate spinner/progress)

### 1) 상단: 계정(Account) 섹션

- 닉네임

    - `public.users.social_provider` 기반의 “가입 provider” 표시와 Supabase identity 기반의 “연결된 provider” 목록은 의미를 구분한다.

### 1-a) 사용자 진도 표시 (compact) `user-progress 예정`

    - 33b에서 badge grid, quest list, XP toast는 표시하지 않는다
    - compact UI가 아닌 대형 progress card는 390px/320px evidence + authority pass 없이 도입하지 않는다

    - 주요 기록: 요리 기록, 플래너 기록, 장보기 기록 같은 compact count 3개 이내
    - Action buttons: `등급`, `업적`, `알림`
    - 등급 modal: 7단계 등급 목록, 현재 등급 highlight, 각 등급 level range, spoon icon/character
    - 업적 앨범 modal: 묶인 category tab, 같은 누적 `track_key`별 card, card 내부 milestone badge row, earned/active/locked 상태, locked hint
    - 튜토리얼 진행: 업적 앨범의 `tutorial` 카테고리에서 6개 튜토리얼 단계와 전체 완료 stamp 표시

- **XP toast / notification trigger**:
    - 레시피 저장, 커스텀 레시피북 생성, 장보기 완료, 요리 완료, 플래너 등록 성공 후 notification refresh를 호출한다
    - system notification의 과거 브랜드 고정 copy는 승인된 exact mapping에 한해 렌더 직전에 canonicalize한다. 저장 row/payload, nickname, 사용자 콘텐츠, 일반명사 `집밥`은 바꾸지 않는다.
    - toast 생성/seen 실패는 원래 action success를 실패로 바꾸지 않는다
    - 알림 우선순위는 `level_up > achievement_unlocked/badge_unlocked > xp_awarded`
    - 퀘스트 완료는 업적 앨범의 튜토리얼 카테고리 상태로만 표시하고 별도 `quest_completed` toast를 띄우지 않는다.
    - 모바일은 동시에 최대 2개, 데스크톱은 동시에 최대 3개를 visible toast로 표시한다
    - 렌더링되지 않은 unseen notification은 다음 표시/보관함을 위해 queue에 남긴다
    - source action 성공 UI와 겹치지 않도록 모바일 하단 탭 위 safe area 또는 desktop lower-right에 표시한다
- **장보기 안내 문구**:
    - SHOPPING_FLOW intro/empty/preview surface 중 하나 이상에 `여러 끼니를 한번에 장보기할 수 있어요`를 표시한다.
    - `장보기 완료 N회`처럼 기준이 모호한 문구는 피하고, 리스트 완료와 끼니 묶음 완료를 구분한다.
- **배지/등급 visual**:
    - 배지 shape family는 `plate`, `shield`, `ribbon`, `bookmark`, `pot`, `leaf`, `bowl`을 기본 후보로 한다.
    - 등급 visual은 `Clay`, `Wood`, `Steel`, `Silver`, `Gold`, `Diamond`, `Titanium` spoon asset을 design source로 사용한다.
    - Runtime 적용 시 source PNG는 production 크기로 최적화하거나 equivalent CSS/SVG component로 대체할 수 있다. 단, grade key와 level band는 바꾸지 않는다.
    - Concept board는 다중 상태를 한 장에 보여주는 reference이며, 35c production UI는 실제 responsive user path로 재해석한다.
- **금지**:
    - leaderboard, competitive rank, 전체 사용자 순위
    - pressure streak, 실패/미접속 패널티
    - season reset, XP decay
    - loot/reward box, 랜덤 보상, 보상 claim CTA
    - locked badge 대형 grid를 MYPAGE 첫 viewport에 노출
    - 클라이언트 측 achievement unlock/badge unlock/quest 완료/XP 계산

## 20) RECIPEBOOK_DETAIL: 레시피북 상세

- **Screen ID**: `RECIPEBOOK_DETAIL`
- **권한**: 로그인 필요
- **목적**: 레시피북 안의 레시피를 책처럼 넘겨 읽고, 필요한 관리 액션을 수행

### UI 구성

- 웹:
    - 왼쪽 목차/책 정보 rail
    - 오른쪽 레시피북 리더
    - `책`/`목록` 전환
    - 책 모드 카드 정보: 썸네일, 제목, 태그, 조회수, 조리시간, 기본 인분, 재료, 만들기
- 모바일:
    - 상단 앱바 제목은 현재 레시피북 이름
    - 앱바 아래 공간은 레시피 목차 중심으로 사용
    - 현재 레시피 카드에는 썸네일, 제목, 메타, 재료, 만들기, 저장하기, 요리하기, 플래너 추가 액션을 표시
- 액션:
    - 레시피 탭 → `RECIPE_DETAIL`
    - 레시피 제거
    - 플래너 추가/저장하기가 직접 실행되지 않는 경우 기존 레시피 상세 진입으로 연결할 수 있다.

### 데이터

- 목록: `GET /recipe-books/{book_id}/recipes`
- 리더 상세: `GET /recipe-books/{book_id}/recipes/{recipe_id}`
- 리더 상세 조회는 `view_count`를 증가시키지 않는다.

### 제거 정책

- **좋아요한 레시피북(liked)** 에서 제거
    - 좋아요 해제 의미
- **저장한 레시피북(saved)** / **커스텀 레시피북(custom)**
    - 레시피북에서 제거 가능
- **내가 추가한 레시피북(my_added)**
    - 레시피북에서 제거 불가
    - 필요 시 별도 레시피 삭제 정책으로 처리(Phase 1 범위 밖 가능)
