## 0. 서비스 개요

- **서비스 한줄 설명**: ‘요리할 메뉴정하기 → 식단 구성 → 장보기목록 생성 → 장보기 → 요리’의 집밥 과정을 도와주는 **웹뷰앱 서비스**.
- **핵심 가치**
    1. 식단을 계획하면 장보기 목록이 자동 생성된다.
    2. 팬트리(집에 있는 재료)는 장보기 목록에서 자동 제외되며, 사용자가 되살릴 수 있다.
    3. 요리는 요리모드에서 한 눈에 보고, 완료 시 남은요리 기록과 팬트리 업데이트까지 연결된다.

## 1. MVP 범위(화면/기능 요구사항)

### 1-1. 홈(레시피 탐색)

- **진입**: 앱 실행/재방문 기본 진입 화면은 홈이다.
- **권한**: 비로그인 가능
- **기능**
    - 레시피 목록 + 테마 섹션 노출
    - **검색(기본)**: 제목 + 승인된 태그 검색
    - **재료 검색(추가)**: 검색창 아래 "재료 버튼" → 재료 검색 팝업 → 재료 다중 선택
        - 선택 재료들이 **모두 포함된 레시피**를 노출
    - **정렬**: 조회수순(기본), 최신순, 저장순, 플래너 등록순
    - **태그 필터**: 태그 chip 또는 테마 진입 시 `normalized_key` 기준 정확 필터를 적용할 수 있다. P0에서는 한글 key를 그대로 사용하고 자동 로마자 변환은 하지 않는다.
    - **테마 생성**: HOME 테마는 `theme_eligible=true`, `visibility='public'`, `review_status='approved'`인 시스템 의미/source 태그만 seed로 사용한다. 사용자 자유 입력 태그는 표시와 검색에는 사용할 수 있지만 테마 seed가 되지 않는다.

### UI 구성

- **전체화면 모드**: 요리모드 진입 시 전체화면으로 전환
- **화면 꺼짐 방지**: 설정에서 ON이고 브라우저 wake lock이 실제 활성화되면 `화면 안 꺼짐` 상태를 표시한다. 미지원/실패 환경에서는 상태를 과장해 표시하지 않는다.
- **whole-board 모드**: 사용자가 이전/다음 단계를 넘기지 않아도 전체 재료와 전체 조리순서를 동시에 볼 수 있어야 한다.

- **조리순서 보드**: 모든 스텝을 번호 순서대로 표시한다. 긴 레시피는 보드 내부 세로 스크롤을 허용한다.
- YouTube 등록 레시피의 재료/단계에 `component_label`이 있으면 요리모드에서도 같은 소제목 규칙을 유지한다.

### 스텝 카드

- **카드 1개 = 스텝 1개**
- 카드 내용: 정사각형 번호 박스, 조리방법 태그, 조리법 본문
- 조리방법 태그는 조리법 본문 위에 표시한다.
- 전체 재료가 별도 보드에 항상 보이므로 단계별 투입 재료/양/썰기 크기는 기본 UI에서 표시하지 않는다.
- 불세기와 조리시간은 COOK_MODE 기본 스텝 카드에 표시하지 않는다.
- 이전/다음 단계 이동, 현재 단계만 크게 보는 UI, 현재 단계 전용 재료량 보드는 제공하지 않는다.
- **조리방법 색상 구분**: 스텝 카드의 테두리 또는 배경 색상으로 조리방법을 한눈에 구분
    - 볶기: 주황 계열
    - 끓이기: 빨간 계열
    - 굽기: 갈색 계열
    - 찌기/스팀: 파란 계열
    - 튀기기: 노란 계열
    - 데치기: 연두 계열
    - 무치기/버무리기: 초록 계열
    - 섞기/준비/기타: 회색 계열

### 장보기 진입 flow

플래너 상단 "장보기" 버튼 클릭 시:

1. **대상 자동 선택**: 오늘~마지막 등록일 중 **"식사 등록 완료" 상태이면서 아직 다른 장보기 리스트에 묶이지 않은 식사**의 인분을 자동 합산
2. **합산 리스트 표시**: 인분이 합산된 레시피들이 선택된 상태로 리스트에 노출 → 사용자가 선택 해제 가능
3. **장보기 기준 인분 조정**: 레시피별 장보기 기준 인분을 조절 가능
4. **장보기 목록 생성**: 선택 확정 → 재료 취합/합산 → 팬트리 자동 제외 처리 → 장보기 리스트 생성
5. **목록 생성 후 자동 이동**: 생성 즉시 `SHOPPING_DETAIL`로 이동
6. **장보기 완료 처리**:
    - 해당 식사들의 상태가 **"식사 등록 완료" → "장보기 완료"**로 변경
    - 완료 직전 “팬트리에 추가할 아이템 선택” 팝업을 통해 팬트리 반영 여부를 고를 수 있음

### 요리하기 진입 flow

#### A. 플래너 상단 "요리하기" 버튼 (일괄/레시피 그룹 요리)

플래너 상단 "요리하기" 버튼 클릭 시:

1. **대상 자동 선택**: 오늘~마지막 등록일 중 **"장보기 완료"** 상태인 식사들을 인분 합산하여 리스트로 표시
2. **레시피별 요리하기**: 리스트의 각 레시피 옆에 "요리하기" 버튼 존재
3. **요리하기 클릭** → 해당 레시피 1개 단위 세션 생성 → 요리모드(1-3) 진입
4. **요리 완료 시**: 해당 레시피의 "장보기 완료" 상태 식사들이 **"요리 완료"** 상태로 변경 → 남은요리 리스트에 자동 추가
    - 장본 인분보다 적거나 많은 인분으로 요리해도 동일하게 처리

#### B. 끼니 화면(MEAL_SCREEN) 개별 식사 "요리하기" 버튼 (단축 경로) `v1.6.7 추가`

끼니 화면에서 **"장보기 완료(shopping_done)"** 상태인 개별 식사에 `[요리하기]` 버튼을 제공한다.

1. **대상 조건**: 해당 식사의 상태가 `shopping_done`인 경우에만 `[요리하기]` 버튼을 노출한다
2. **`[요리하기]` 클릭** → 해당 식사 1건 + 해당 식사의 recipe_id로 planner 요리 세션을 생성한다
    - 기존 `POST /cooking/sessions` 계약을 `meal_ids=[선택된_meal_id]`로 호출하는 단축 경로이다
3. **요리모드(COOK_MODE) 진입** → 기존 세션 기반 요리모드와 동일하게 동작한다
4. **요리 완료 시**: 해당 식사만 `shopping_done` → `cook_done`으로 전이한다. 남은요리 자동 추가, 소진 재료 팬트리 반영은 기존과 동일하다
5. **취소/뒤로가기**: 상태 변경 없이 끼니 화면(MEAL_SCREEN)으로 복귀한다

**제한 사항**:
- `식사 등록 완료(registered)` 상태 식사는 `[요리하기]` 대상이 아니다. 장보기를 우회하여 `cook_done` 상태로 전이할 수 없다
- `요리 완료(cook_done)` 상태 식사에는 `[요리하기]` 버튼을 노출하지 않는다
- 이 경로는 **planner 요리 세션의 단축 경로**이며, 독립 요리(⑧)와 혼합되지 않는다

### 1-6. 장보기 `v1.6 업데이트`

### 핵심 기능

- 플래너에서 "식사 등록 완료" 상태 식사들의 인분 자동 합산 → 선택/해제 → 장보기 목록 생성
- 재료 취합/합산 → 팬트리 자동 제외 처리
- 장보기 목록은 **저장**되며 마이페이지에서 재열람 가능
- 쿠팡/컬리 버튼: "검색 링크" 수준으로 제공

### 장보기 상세 화면 정책

장보기 목록 생성 후 진입하는 상세 화면에서는 다음이 가능해야 한다.

1. **구매 섹션**
    - 실제 장볼 아이템 카드 리스트
    - 체크박스로 구매 완료 표시
    - 카드 드래그&드롭으로 순서 변경 가능
2. **팬트리 제외 섹션**
    - 팬트리에 있어 자동 제외된 재료를 별도 섹션으로 표시
    - 유저가 되살리고 싶으면 구매 섹션으로 다시 이동 가능
3. **섹션 간 이동 규칙**
    - 구매 섹션 → 팬트리 제외 섹션 이동 시 해당 항목은 자동으로 구매 체크 해제
    - 팬트리 제외 섹션 → 구매 섹션 이동 시 다시 장보기 대상으로 복귀
4. **공유**
    - 장보기 공유 텍스트는 **구매 섹션 항목만 포함**
    - 팬트리 제외 섹션 항목은 공유 대상에서 제외

### 장보기 완료 처리

- 장보기 목록에서 사용자가 **실제로 구매한 항목을 체크**
- [장보기 완료] 버튼 클릭 시:
    1. 해당 식사들의 상태 → **"장보기 완료"**로 변경
    2. "팬트리에 추가할 아이템 선택" 팝업 노출
        - 기본값: 구매 체크된 항목 전체 선택
        - 사용자가 체크 해제로 팬트리 반영 제외 가능
    3. 선택된 항목만 팬트리에 추가
- 팬트리에 아무것도 추가하지 않고 장보기 완료만 하는 것도 가능
- 완료 후 장보기 리스트는 **read-only**
    - 재열람은 가능하지만 체크/정렬/제외 수정은 불가

## 2. 공통 정책(데이터/UX/로직)

    2. 안내 메시지 + 선택 버튼 제공

### 2-4. 유튜브 레시피 추출 정책

- 유튜브 레시피는 **등록 시점에 재료/양이 확정된 상태**로 저장한다.
- **추출 파이프라인 (순서대로 적용)**:
    1. **설명란 파싱** → 재료/양/단계 표준화
    2. **공개 작성자 댓글 후보 파싱** → 설명란이 불완전할 때 작성자 top-level 댓글만 보조 source로 사용
    3. **공개 자막/timedtext 파싱** → 설명란/댓글이 불완전할 때 공개 caption track 텍스트를 보조 source로 사용
    4. **OCR(재료표 화면)** → 영상 내 재료표 이미지에서 텍스트 추출
    5. **ASR 전사** → 자막이 부실한 영상 커버
    6. **추정 레이어 고도화** → 위 단계에서 누락/불완전한 수량 보완
        - 1차: 관용량 테이블(통계 기반)
        - 2차: 유사 레시피 Retrieval
        - 3차: 수량 예측 모델
- **URL 검증 단계 (3-way classification)** `v1.6.8`:
    - 유튜브 URL 입력 직후, 해당 영상이 요리/레시피 영상인지 3단계로 판정한다.
    - `recipe`: 설명란에 재료/조리 단계가 명확 → 추출 진행, 경고 없음
    - `uncertain`: 음식 관련이지만 구조화된 레시피가 불분명 → 추출 진행 가능, careful-review 경고 표시
    - `non_recipe`: 요리와 무관한 강한 증거 → 추출 차단, 다른 URL 입력 요청 (MVP에서 override 없음)
    - Classifier는 보수적: 약한/혼합 증거는 `uncertain`, 강한 비요리 증거만 `non_recipe`
- **추출 세션 관리** `v1.6.8`:
    - 추출은 서버에서 세션(`youtube_extraction_sessions`)을 생성하여 관리한다.
    - 세션은 24시간 후 만료된다. 같은 URL이어도 매번 새 세션을 생성한다(MVP).
    - 등록은 세션 소유권 검증 + Postgres RPC 원자적 트랜잭션으로 수행한다.
- **다중 레시피 후보 선택** `2026-05-30 addendum`:
    - 한 영상 안에 여러 요리가 있는 경우 공개 설명란/작성자 댓글/caption timedtext에서 일반 규칙으로 후보를 분리한다.
    - 추출 결과는 `multi_parent` 세션과 `recipe_candidates[]`를 반환하고, 사용자가 저장할 후보를 먼저 선택해야 한다.
    - 후보 선택 시 `candidate_child` 세션을 생성해 기존 검수/미등록 재료 등록/register 흐름을 재사용한다.
    - 특정 video_id별 하드코딩, 레시피오 결과 반환, 고정 fixture를 실제 추출 결과처럼 반환하는 shortcut은 금지한다.
    - 공개 텍스트에서도 후보별 재료/단계를 만들 수 없으면 임의 생성하지 않고 검수 필요/지원 불가 상태로 남긴다.
- **Gemini 구조화 fallback** `2026-06-01 addendum`:
    - 설명란, 공개 작성자 댓글, 공개 caption/timedtext parser 이후에도 재료 또는 조리 단계가 부족하거나, 공개 자막/전사 parser 결과가 대화 파편/노이즈 중심으로 저품질일 때만 Gemini structured fallback을 호출할 수 있다.
    - fallback은 기본 비활성화이며 `YOUTUBE_RECIPE_LLM_ENABLED=true`, API key, 일일/사용자 한도, cache miss 조건을 모두 만족해야 한다.
    - Gemini에는 이미 수집한 공개 텍스트만 전달한다. 원문 근거(`source`, `line_index`, 시간 범위)가 없는 재료/단계는 폐기한다.
    - `extraction_methods`에는 새 LLM 값을 넣지 않고 실제 공개 source인 `description` / `comment` / `caption`만 기록한다.
    - `source_providers`에는 `gemini_structured_extractor` 또는 `gemini_structured_extractor_cache`를 기록하고, `extraction_meta_json.llm_extractor`에 provider/model/cache/status/token/reason 정보와 parser 품질 판정(`parser_quality`)을 남긴다.
    - 자막/전사 parser 결과가 저품질인데 Gemini가 비활성/실패/한도 초과인 경우, 대화 파편을 재료·단계처럼 확정 노출하지 않고 검수 필요 상태로 남긴다.
    - API key, provider raw response, secret, 레시피오 결과는 `raw_source_text`, `extraction_meta_json`, event table에 저장하지 않는다.
- **재료 해결 상태** `v1.6.8`:
    - 추출된 재료는 `resolved`, `needs_review`, `unresolved` 상태를 가진다.
    - 모든 재료가 `resolved`이고 blocking step field가 채워져야 등록이 가능하다.
- **미등록 재료 등록** `v1.6.9`:
    - 추출된 재료가 DB 표준명/동의어로 매칭되지 않으면 검수 화면에서 `unresolved`로 표시한다.
    - 사용자는 기존 재료 검색으로 교체하거나, 서버 추출 row(`draft_ingredient_id` 보유)에 한해 새 표준 재료로 등록할 수 있다.
    - 새 재료 등록은 자동 실행하지 않고 반드시 사용자 확인을 거친다.
    - 등록 시 표준명, 카테고리, optional synonym을 검증하고 `ingredients` / `ingredient_synonyms`를 RPC로 처리한다.
    - 등록 카테고리는 v1 canonical 8종 문자열(`채소`, `과일`, `육류`, `해산물`, `양념`, `유제품`, `곡류`, `기타`)만 허용한다.
    - 이미 같은 `standard_name`이 있으면 새 row를 만들지 않고 기존 ingredient를 재사용한다.
    - 같은 synonym이 다른 ingredient에 이미 있으면 ambiguous 확산을 막기 위해 synonym 저장을 skip하고 warning만 반환한다.
    - 등록 성공 후 session `draft_json`은 원본 추출 snapshot으로 유지하고, 클라이언트가 현재 검수 row를 `resolved`로 전환한다.
- **레시피오형 빠른 가져오기 모드** `2026-05-28 addendum`:
    - `/recipes/new/youtube`는 기존 `YT_IMPORT`와 별도 진입면으로 제공한다.
    - URL 입력 또는 추천 영상 선택 시 기존 레시피 중복 여부를 먼저 확인한다.
    - 중복이면 크레딧/추출 비용이 드는 작업 없이 저장된 레시피 카드와 상세 이동을 제공한다.
    - 중복이 아니면 기존 `validate` → `extract` → `register` 계약을 순차 호출해 자동 등록을 시도한다.
    - 모든 재료가 `resolved`이고 blocking step field가 없을 때만 자동 등록한다.
    - `needs_review`/`unresolved` 재료, 필수 step 누락, `non_recipe`, provider error는 자동 등록하지 않고 기존 검수/재입력 흐름으로 fallback한다.
    - 진행률은 실제 비동기 단계 상태를 보여주는 UI 값이며, 서버에 새 progress/status endpoint를 만들지 않는다.
    - 이 화면 자체는 별도 LLM/이미지 생성 계약을 만들지 않는다. 다만 extract 단계는 공식 YouTube 추출 계약의 Gemini structured fallback 조건을 그대로 따른다.
    - 특정 YouTube video_id에 대해 레시피오에서 생성된 레시피 데이터를 코드 fixture로 반환하지 않는다. 중복 확인을 제외한 신규 추출은 항상 provider/parser 경로를 거친다.
- **레시피 썸네일/태그 영속화** `v1.7.11 갱신`:
    - YouTube 등록 시 서버가 추출 세션의 `thumbnail_url`을 `recipes.thumbnail_url`에 복사한다. 클라이언트는 썸네일을 제공하거나 오버라이드할 수 없다.
    - YouTube 썸네일은 원본 URL을 그대로 보존한다. 다운로드/리호스팅/크롭/압축을 하지 않는다.
    - YouTube/직접 등록 시 서버가 공유 태그 추천기로 추천 태그를 만든다. 태그 생성은 제목, 재료, 조리 과정, 조리방법, source_type에서 추론하며 YouTube 등록은 provider `snippet.tags`를 필터링해 참고한다.
    - 사용자가 태그를 수정하지 않으면 서버 추천값을 저장한다. 사용자가 태그를 추가/삭제/수정하면 서버는 이를 사용자 검수 완료 태그로 정규화해 저장한다.
    - canonical tag 관계는 `tags` / `recipe_tags`이며, `recipes.tags`는 카드/레거시 응답용 projection이다.
    - 사용자 자유 tag는 승인 전 HOME theme seed로 자동 승격하지 않는다.
- **섹션 라벨 영속화** `v1.7.3`:
    - description parser가 감지한 재료/단계 섹션명은 extract 응답의 `component_label`로 반환한다.
    - YouTube register는 `component_label`을 `recipe_ingredients`, `recipe_steps`에 nullable 값으로 저장한다.
    - `component_label`이 있으면 저장되는 `display_text`, `instruction`은 같은 섹션명 prefix를 포함하지 않는다.
    - 기존 legacy row에서 `component_label` 없이 `[섹션명]` prefix만 있는 경우에는 현상 유지한다.
    - 수동 레시피 작성 계약은 이번 범위에서 섹션 라벨 입력을 지원하지 않는다.
- **YouTube Data API** `v1.6.8`:
    - URL 입력 직후 미리보기는 YouTube oEmbed로 title/channel/thumbnail만 조회한다(YouTube Data API quota 미사용).
    - 실제 추출 단계에서만 `videos.list` (part=snippet,contentDetails)를 호출한다: 1 quota unit, title/description/tags/category/thumbnails/duration/caption flag 조회
    - 요리 영상 여부와 공개 텍스트 레시피 존재 여부는 `videos.list` 응답과 이후 공개 텍스트 source를 확인한 추출 단계에서 판정한다.
    - `captions.list`/`captions.download`: 비용/권한 제약으로 현재 미사용
    - 공개 caption track의 timedtext 조회는 설명란/작성자 댓글 보조 source로 허용한다. 로그인/소유자 권한이 필요한 caption download, 오디오 ASR, 비디오 OCR은 이 범위에서 사용하지 않는다.
- **사용자 수동 입력**: 위 파이프라인으로 추론 완료 후, 결과를 사용자에게 보여주고 수정/보완을 받는다.
- (참고: OCR, ASR, 추정 레이어는 단계적으로 구현. 현재 YouTube 추출은 설명란 + 공개 작성자 댓글 + 공개 timedtext 자막 + 조건부 Gemini 구조화 fallback + 수동 보완)
- **Visual Quantity Enrichment** `2026-06-02 addendum`:
    - 공개 텍스트 추출과 기존 Gemini text structured fallback 이후에도 재료 수량이 부족한 경우, 조건부로 `visual_quantity_extractor`를 실행할 수 있다.
    - visual enrichment는 기본 비활성화이며 `YOUTUBE_RECIPE_VISUAL_QUANTITY_ENABLED=true`, API key/provider config, 사용자/일일 한도, cache miss, timeout budget 조건을 모두 만족해야 한다.
    - visual enrichment는 원천 extraction method가 아니라 보조 enrichment이므로 `extraction_methods`에는 추가하지 않는다. `source_providers`에 `visual_quantity_extractor` 또는 `visual_quantity_extractor_cache`를 기록하고, `extraction_meta_json.visual_quantity_extractor`에 provider/model/cache/status/enriched_count 정보를 남긴다.
    - 수량 출처는 `YoutubeQuantitySource` enum으로 추적한다: `text_explicit | visual_explicit | unit_normalized | ingredient_default | recipe_inferred | user_entered | unknown`.
    - `text_explicit`과 `visual_explicit`만 explicit fact이다. `unit_normalized`는 raw explicit evidence가 있을 때만 허용한다. `ingredient_default`는 명시 count evidence가 있을 때만 허용한다(예: `두부 1모` → 300g).
    - `recipe_inferred`는 review-only이며, human confirmation/edit/clear 없이는 register 또는 quick auto-register를 unblock하지 못한다.
    - register 시 `quantity_confirmation_status`(`not_required | confirmed_suggestion | edited_quantity | cleared_to_taste`)를 서버가 draft 기준으로 검증한다. 클라이언트가 보낸 `quantity_review_required`는 신뢰하지 않는다.
    - 실패 시 공개 텍스트 전용 결과로 fallback한다. visual enrichment 에러가 전체 추출을 실패시키지 않는다.
    - raw video, raw frame, raw provider response, API key, secret, 레시피오 data는 저장/반환하지 않는다.
    - selected-frame OCR, Cloud Vision, ASR/STT는 v1 out of scope이다.

### 2-6. 재료 표준화/단위/타입

- 재료 표준명은 재료 마스터 사전(표준명+동의어)로 매핑
- 재료 taxonomy v2 목표는 사용자-facing 8대분류와 내부 21소분류다.
    - `곡류/면/떡`: 밥/쌀, 면/파스타, 빵/떡/시리얼
    - `채소/버섯`: 잎/나물채소, 뿌리/줄기채소, 열매채소/버섯
    - `과일/견과`: 과일, 견과/씨앗/건과일
    - `단백질`: 돼지/소/양, 닭/오리, 달걀, 두부/콩류
    - `해산물`: 생선/갑각/조개, 해조/건어물/어묵
    - `유제품/대체유`: 우유/요거트/크림, 치즈/버터/대체유
    - `양념/조미`: 장류/소스, 향신료/허브, 기름/식초/당류/육수
    - `가공/기타`: 김치/절임/통조림, 냉동/간편식/음료/기타
- 재료 카테고리 v1 canonical은 `과일`을 포함한 8종 문자열(`채소/과일/육류/해산물/양념/유제품/곡류/기타`)이다.
- 신규 ingredient taxonomy는 기존 `ingredients.category`를 대체하지 않는다. `category_code` 같은 코드/seed 기반 shared mapping source나 additive shadow metadata로만 해석 계층을 둔다.
- `GET /ingredients?category=<category label>`와 YouTube 미등록 재료 등록의 category validation은 v1 canonical 8종 label 계약을 유지한다.
- 단위 표준 그룹: g/kg, ml/L, 개/ea, 큰술(tbsp), 작은술(tsp), TEXT_UNIT(컵/꼬집/줌/망 등)
- 재료 타입: QUANT(정량), TO_TASTE(비정량)
- UNKNOWN 타입은 사용하지 않는다(정량 재료는 등록 시점에 반드시 수량/단위 확정)
- 조리방법 taxonomy v2 목표는 6그룹/20대표 method다.
    - `준비/손질`: 썰기, 다지기
    - `전처리`: 해동, 밑간, 절이기
    - `물/수분 조리`: 끓이기, 삶기, 데치기, 찌기
    - `팬/기름 조리`: 볶기, 굽기, 부치기, 튀기기
    - `혼합/조림`: 섞기, 무치기, 조리기, 졸이기
    - `기기 조리`: 전자레인지, 오븐굽기, 에어프라이어
- `씻기`, `채썰기`, `재우기`, `핏물빼기`, `지지기`, `중탕`, `압력솥`, `간보기`, `토핑`, `담기`, `식히기`, `숙성`은 canonical method가 아니라 synonym 또는 자유 step text 후보로 둔다.
- 조리방법 category는 optional additive metadata다. `cooking_methods.label`은 짧은 표시 라벨 용도이며 taxonomy 코드/분류 저장소로 사용하지 않는다.
- 외부 공공/상용 데이터(식약처, 농식품올바로 등)는 production canonical ingredient로 직접 넣지 않는다. raw import → normalization → review → 승인 seed 경로를 거친 뒤에만 반영한다.

### 2-7. 장보기 취합/합산 규칙

- 합산 규칙:
    1. 같은 재료(표준명) + 같은 단위 → 수치 합산
    2. 변환 가능 단위(kg↔g, L↔ml) → 변환 후 합산
    3. 변환 불가 단위(개 vs g, 망 vs g 등) → **복합 표기만 사용**
        - 예: `양파 2개 + 200g + 1망`
    4. TEXT_UNIT은 같은 문자열끼리만 합산(예: 1줌+2줌=3줌)

### 2-8. 식사 상태 & 상태 전이

### 식사(Meal) 상태 정의

플래너에 레시피를 등록하면 "식사(Meal)"가 생성되며, 아래 3가지 상태를 가진다.

| 상태 | 설명 | 전이 조건 |
| --- | --- | --- |
| **식사 등록 완료** | 플래너에 식사가 등록된 초기 상태 | 식사 등록 시 자동 |
| **장보기 완료** | 해당 식사의 재료를 장봤음 | 장보기 완료 처리 시 |
| **요리 완료** | 해당 식사의 요리를 했음(= 남은요리 상태) | 요리완료 처리 시 |

### 상태 전이 flow

```
[식사 등록] → 식사 등록 완료
                  │
                  ▼  (장보기 완료 클릭)
             장보기 완료
                  │
                  ▼  (요리 완료 클릭)
              요리 완료 → 남은요리 목록에 자동 추가
```

### 장보기 대상

- 장보기 버튼 클릭 시 **"식사 등록 완료"** 상태인 식사들만 대상
- 단, **이미 다른 장보기 리스트에 묶인 식사(shopping_list_id 존재)는 제외**
- 장보기 목록 생성 시 선택된 식사는 해당 장보기 리스트에 먼저 연결된다.
- 장보기 완료 처리 시 대상 식사들이 일괄 **"장보기 완료"**로 전이된다.

### 요리하기 대상

- 요리하기 버튼 클릭 시 **"장보기 완료"** 상태인 식사들만 대상
- 요리 완료 처리 시 해당 레시피의 "장보기 완료" 식사들이 일괄 **"요리 완료"**로 전이
- 장본 인분과 실제 요리 인분이 달라도 동일하게 처리(상태 전이는 인분 무관)
- `MEAL_SCREEN` 개별 식사 `[요리하기]`도 동일한 `shopping_done` 조건을 적용한다. `registered` 상태 식사는 장보기를 거치지 않고 `cook_done`으로 전이할 수 없다 `v1.6.7 추가`

### 팬트리/남은요리 연동

- **장보기 완료 시**
    - 식사는 `shopping_done`으로 전이
    - 팬트리 반영은 팝업에서 선택된 항목에 한해 수행
- **요리 완료 시**
    - 소진 재료 체크리스트(기본 체크 해제) → 체크된 재료만 팬트리에서 제거
    - 남은요리 목록에 자동 추가

### 2-12. 장보기 상세 화면 정책 `v1.6 추가`

- 장보기 완료 후 리스트는 **read-only**
    - 기록은 카드 탭으로 재열람 가능하지만 수정은 불가
    - 완료된 기록은 완료 시각을 함께 표시할 수 있다.

### 2-13. 레시피북 정책 `v1.6 추가`

- 시스템 레시피북은 다음 3개이다.
    - 내가 추가한 레시피
    - 저장한 레시피
    - 좋아요한 레시피
- 커스텀 레시피북은 유저가 생성/수정/삭제할 수 있다.
- 저장 액션은 **저장한 레시피북(saved)** 또는 **커스텀 레시피북(custom)** 에만 수행한다.
- 저장 액션은 여러 `saved/custom` 레시피북을 한 번에 선택할 수 있다.
- 레시피북 상세 카드는 태그, 조회수, 조리시간, 기본 인분 정보를 표시한다.
- 레시피북 리더는 사용자가 소유한 레시피북 안의 레시피에 한해 재료와 만들기 내용을 읽기 전용으로 표시할 수 있다.
- 레시피북 리더용 상세 조회는 `GET /recipes/{id}`의 상세 화면 조회 의미와 분리하며, `view_count`를 증가시키지 않는다.
- 좋아요한 레시피북은 좋아요 상태를 기반으로 구성된다.
- 내가 추가한 레시피북은 직접 등록/유튜브 등록한 레시피를 기반으로 구성된다.

### 2-14. 관리자 접근 정책 `v1.7.2 추가`

- 관리자 기능은 내부 운영 전용이며, 일반 사용자에게 노출되지 않는다.
- `admin_members` 테이블이 관리자 신원의 단일 진실 소스이다.
- 관리자 API는 반드시 `createServiceRoleClient()`를 사용하며, service role 부재 시 fail closed 한다. `routeClient` 폴백은 관리자/교차 사용자 읽기에 금지된다.
- 관리자 행위는 예외 없이 `admin_audit_logs`에 기록한다.
- PII 최소화 원칙: 승인된 요약 정보만 표시하며, 민감 정보(OAuth 토큰, YouTube URL, 검색어, 비공개 장보기/팬트리 상세)는 로그에 기록하지 않는다.
- 운영 이벤트(`operational_events`)와 감사 로그(`admin_audit_logs`)의 `request_path`는 pathname만 저장한다. 쿼리스트링은 첫 버전에서 저장하지 않는다.

### 2-16. 사용자 진도/성장 추적 정책 `growth-leveling-v2`

- 사용자의 요리/장보기/레시피 저장/레시피북 생성 활동을 기반으로 XP와 레벨을 부여한다.
- XP와 레벨 계산은 서버가 단일 권한자이다. 클라이언트는 표시만 한다.
- 진도 조회는 전용 API(`GET /users/me/progress`)를 통한다. `GET /users/me`는 프로필/설정-only를 유지한다.
- 34 시리즈부터 XP source는 아래 5개이다. 33c의 “새 XP source 없음 / XP 배점 변경 없음” 제약은 이 계약으로 대체한다:
  - `cooking_completed`: `leftover_dishes` row 확정 시
  - `shopping_completed`: `shopping_lists.is_completed=true` and `completed_at` 전환 시
  - `recipe_saved`: user+recipe saved/custom membership `0 → ≥1` transition 시 (distinct-ever, 현재 수가 아님)
  - `custom_book_created`: `recipe_books.book_type='custom'` INSERT 성공 시 (bootstrap system books 제외)
  - `planner_registered`: `meals` row INSERT 성공 시. 최초 등록 XP와 반복 등록 XP를 분리하고 반복 XP에는 cap을 둔다.
- XP source별 배점은 아래 표를 기준으로 한다.

| XP source | 첫 경험치 | 반복 경험치 | 반복/악용 방지 |
| --- | ---: | ---: | --- |
| `recipe_saved` | 15 XP | 8 XP | user+recipe distinct-ever. unsave/resave 재적립 금지 |
| `custom_book_created` | 25 XP | 10 XP | custom book only. KST 2/day repeat cap |
| `shopping_completed` | 40 XP | 25 XP | shopping list idempotency. 완료 retry 재적립 금지 |
| `cooking_completed` | 60 XP | 45 XP | leftover_dish source row 기준 1회 |
| `planner_registered` | 25 XP | 5 XP | KST 3/day, 12/week repeat cap. first는 repeat cap 미소비 |

- 중복 award는 전용 ledger의 idempotency key로 방지한다.
- XP source별 첫 경험치와 반복 경험치는 분리한다. 플래너 등록은 최초 25 XP, 반복 5 XP이며 반복은 KST 기준 3/day, 12/week cap을 적용한다. 최초 XP는 반복 cap을 소비하지 않는다.
- XP source action 재시도, 삭제/재생성, PATCH, 상태 전이 재호출은 idempotency key와 cap으로 반복 악용을 막는다.
- XP가 없는 업적/퀘스트 활동은 `user_growth_activity_events`에 기록한다. 이 ledger는 팬트리 추가, 장보기 끼니 묶음 준비, 남은요리 다먹음, 다양한 식사 추가 경로, 레시피북 생성/추가/제거 같은 활동 truth이며 XP 지급 ledger가 아니다.
- `operational_events`는 운영 로그이며, 사용자 보상 truth로 재사용하지 않는다.
- legacy/backfill은 surviving rows 기준 lower-bound이다. 삭제된 활동은 복원됐다고 주장하지 않는다. 기존 유저 backfill은 XP/레벨/등급/배지/퀘스트 상태를 조용히 반영하되 과거 toast 대상 notification row를 만들지 않는다.
- Historical/backfill recompute는 `level_up`, `achievement_unlocked`, `badge_unlocked`, `xp_awarded` notification row를 만들지 않는다. 최근 성장 기록/알림 보관함은 v1에서 cutover 이후 live event만 포함한다.
- MYPAGE 기존 stats(식사 수 등)와 progress event counts는 다른 개념이다. 같은 수치로 표시하지 않는다.
- progress 조회 실패 시 MYPAGE 전체가 아닌 progress 영역만 soft-fail한다. 401은 기존 auth gate와 동일하게 fatal.
- 33a(foundation): progress ledger, read model, `GET /users/me/progress`만 포함한다.
- 33a에 배지, 퀘스트, 토스트, 튜토리얼, 타임라인, backfill API는 포함하지 않는다.
- 33b(MYPAGE UI): 하드코딩 레벨 문구를 실제 progress 기반 compact UI로 교체한다.
- 33c(gamification): 배지, 퀘스트, XP toast, 튜토리얼 퀘스트를 별도 계약으로 추가한다.
- 34a(contract-evolution): 새 XP source/배점/레벨 곡선/등급/활동 ledger/알림 보관함/프로필 통합 UI 방향을 공식 문서로 잠근다.
- leaderboard, competitive rank, pressure streak, season reset, loot-box style rewards는 별도 승인 전까지 scope 밖이다.

#### 2-16-a. 사용자 성장/배지/퀘스트 정책 `growth-leveling-v2`

- `GET /users/me/progress`는 33a response shape를 유지한다. 배지, 퀘스트, 토스트, 튜토리얼 필드는 별도 gamification API에서 조회한다.
- badge/quest/tutorial/notification 상태는 서버가 단일 권한자이다. 클라이언트는 unlock 조건, quest 완료 여부, XP/level을 계산하지 않는다.
- 배지는 조건을 만족하면 자동으로 unlock된다. 같은 badge는 사용자별로 한 번만 획득하며, 중복 호출이나 source action retry로 중복 생성되지 않는다.
- 퀘스트는 신규/초기 사용자의 자연스러운 사용 흐름을 돕는 튜토리얼 전용 surface이다. 35 시리즈 이후 standard quest expansion은 중단한다.
- 튜토리얼은 업적 앨범의 `tutorial` 카테고리로도 표시한다. 숨김/dismiss는 UX 상태만 바꾸며 XP, level, badge award를 변경하지 않는다.
- XP toast/notification은 source action 성공 후 “이번 행동이 성장에 반영됐다”는 feedback이다. notification 생성/조회/seen 실패는 원래 source action 성공을 실패로 바꾸지 않는다.
- toast는 stack을 지원한다. 알림 우선순위는 `level_up > achievement_unlocked/badge_unlocked > xp_awarded`이며, 모바일은 최대 2개, 데스크톱은 최대 3개 visible toast를 기본으로 한다.
- 퀘스트 완료 상태는 `user_quest_progress`와 업적 앨범의 튜토리얼 카테고리로 표시하며, 별도 `quest_completed` notification row나 toast를 만들지 않는다.
- `level_up` notification은 live XP write 후 `previous_level < next_level`일 때만 생성한다. backfill/recompute는 level-up toast를 만들지 않는다.
- gamification projection은 `user_progress_events` / `user_progress_summary`와 `user_growth_activity_events`에서 파생한다. `operational_events`를 reward truth로 사용하지 않는다.
- 장보기 관련 quest count는 리스트 기준과 끼니 묶음 기준을 분리한다. `shopping_list_completed_count`는 완료된 `shopping_lists` row 수이고, `shopping_meal_bundle_completed_count`는 distinct `shopping_bundle_prepared` activity 수이며, `shopping_meals_covered_count`는 장보기 prep action으로 cover된 meal id 수이다.
- 장보기 안내 문구는 “여러 끼니를 한번에 장보기할 수 있어요”를 사용한다. UI는 shopping list 1개가 meal 1개와 같다는 오해를 만들면 안 된다.
- 업적 카테고리는 `tutorial`, `recipe`, `planner`, `shopping`, `cooking`, `pantry`, `leftovers`, `recipebook`을 기본으로 한다. `recipe` 카테고리 안에서 레시피 보관 track과 레시피 등록 track은 분리한다.
- 잠긴 배지는 짧은 next-action hint를 제공할 수 있다. hint는 압박 streak, 경쟁 순위, loot 보상 표현을 쓰지 않는다.
- 배지는 `plate`, `shield`, `ribbon`, `bookmark`, `pot`, `leaf`, `bowl` 같은 shape family를 가진다. 업적 stamp는 badge metadata를 공유하되 achievement unlock과 XP 지급은 분리한다.
- 레벨 곡선은 만렙 중심이 아니라 장기 성장 곡선이다. v2 공식은 `levelStartXp(level) = 40 * (level - 1) ** 2 + 60 * (level - 1)`이다.
- 등급명은 레벨 band로 제공한다: 1-3 `Clay`, 4-7 `Wood`, 8-12 `Steel`, 13-20 `Silver`, 21-34 `Gold`, 35-49 `Diamond`, 50+ `Titanium`.
- MYPAGE에서는 별도 대형 성장 card보다 프로필 header 안에 등급/레벨/XP/주요 기록/action buttons를 통합한다. 등급/업적/튜토리얼/알림 상세는 버튼으로 여는 modal 또는 bottom sheet로 둔다.
- legacy/backfill 사용자에게는 surviving rows 기준 lower-bound만 반영한다. 삭제된 과거 활동이 badge/quest로 복원됐다고 주장하지 않는다.
- 사용자에게 압박을 주는 경쟁 랭킹, 전체 순위, streak 실패/패널티, season reset, XP decay, loot-box/random reward, reward claim CTA는 scope 밖이다.

#### 2-16-b. 업적 앨범/튜토리얼 카테고리 정책 `growth-achievement-album`

- 업적은 장기 사용 목표와 수집 상태를 표시하는 read-only progression이다. 업적 달성은 XP를 추가 지급하지 않는다.
- 업적 정의는 MVP에서 서버 코드 상수로 관리한다. 운영 중 편집이 필요하면 후속 definition table 또는 admin editor로 분리한다.
- 업적 item은 `achievement_key`, `category_key`, `track_key`, `title`, `description`, `target`, `current`, `status`, `badge`, `locked_hint`를 가진다.
- `status`는 `earned`, `active`, `locked` 중 하나다. 클라이언트는 unlock을 계산하지 않는다.
- 튜토리얼 업적 기준은 다음 6단계와 전체 완료 1개다.
    1. 첫 레시피 저장
    2. 첫 플래너 끼니 등록

- 레시피 보관 count는 저장한 레시피와 사용자가 등록한 레시피를 dedupe한 library 기준이다.
