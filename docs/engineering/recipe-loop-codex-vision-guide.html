recipe-loop beginner guide

Codex Vision은 영상을 어떻게 레시피 JSON으로 바꾸나요?

기존 기본 경로는 Gemini입니다. 새로 추가된 --provider codex-vision은 Gemini를 지우는 기능이 아니라, 같은 입력을 Codex Vision 방식으로도 돌려서 비교할 수 있게 만든 실험용 옆길입니다.

Gemini 기본 유지 Codex Vision 선택 실행 캐시와 실패 로그 보존
1. source.json 읽기 제목, 설명란, 자막, 댓글, YouTube URL을 입력으로 만든다.
2. 영상 프레임 뽑기 영상에서 중요한 장면 이미지를 고르고 파일로 저장한다.
3. Codex가 이미지 묶음을 본다 프레임을 여러 묶음으로 나누어 화면 속 재료와 동작을 메모한다.
4. 최종 recipes JSON 생성 텍스트 소스와 시각 메모를 합쳐 구조화된 레시피를 만든다.

먼저 큰 그림

recipe-loop는 YouTube 레시피 추출 품질을 실험하는 오프라인 도구입니다. 입력은 각 영상 폴더의 source.json이고, 출력은 실행 태그별 result.json입니다.

Provider

“어떤 AI 경로로 추출할지”를 고르는 이름입니다. 지금은 gemini와 codex-vision 두 가지가 있습니다.

Gemini는 그대로 기본값

--provider를 안 쓰면 여전히 Gemini 경로가 실행됩니다. 기존 비교 기준을 깨지 않기 위해서입니다.

Codex Vision은 비교용 새 경로

--provider codex-vision을 붙였을 때만 영상 프레임을 뽑고 Codex CLI에 이미지를 넘깁니다.

실행 명령어는 이렇게 다릅니다

Gemini와 Codex Vision은 같은 run-extraction.mjs를 씁니다. 차이는 --provider 옵션입니다.

기존 Gemini 기본 경로

옵션을 생략하면 이쪽입니다.

node scripts/recipe-loop/run-extraction.mjs \
  --split train \
  --ids 5NSTRKouSWs \
  --out-tag gemini-smoke

새 Codex Vision 경로

영상 프레임 기반으로 비교 실행합니다.

node scripts/recipe-loop/run-extraction.mjs \
  --split train \
  --ids 5NSTRKouSWs \
  --out-tag codex-vision-smoke \
  --provider codex-vision \
  --max-frames 4 \
  --batch-size 4

Codex Vision 전체 흐름

아래 순서대로 보면 됩니다. 핵심은 “영상을 바로 한 번에 이해한다”가 아니라, “영상에서 이미지를 뽑고, 이미지 메모를 만든 뒤, 그 메모로 JSON을 만든다”입니다.

1

source.json 읽기

영상 제목, 설명란, 자막, 작성자 댓글, YouTube URL을 읽습니다.

2

입력 객체 만들기

sourceToInput()이 추출 모듈이 이해하는 형태로 바꿉니다.

3

Provider 선택

--provider codex-vision이면 Codex Vision client를 만듭니다.

4

프롬프트 생성

extractRecipeFromSources()가 레시피 추출 지시문을 만듭니다.

5

프레임 추출

extract-video-frames.py가 중요한 장면 이미지를 저장합니다.

6

이미지 배치 분석

프레임을 batch-size만큼 나눠 Codex CLI에 이미지로 보냅니다.

7

시각 메모 저장

각 배치 결과를 합쳐 visual_notes.md에 저장합니다.

8

최종 JSON 생성

원래 프롬프트와 시각 메모를 함께 주고 { recipes: [...] }를 받습니다.

9

파싱과 1회 복구

JSON 파싱 실패 시 한 번만 “JSON만 복구해줘”라고 다시 요청합니다.

10

결과와 캐시 저장

final.json, run_meta.json, 최종 result.json이 남습니다.

프레임 추출은 어떤 방식인가요?

Codex Vision은 영상 전체를 그대로 AI에 넣지 않습니다. 먼저 Python helper가 영상에서 대표 장면 이미지를 뽑습니다.

방식 무슨 뜻인가요? 관련 옵션
scene 화면 변화가 큰 지점을 장면 후보로 잡습니다. 기본값입니다. --scene-detail, --scene-selection, --max-frames
interval 일정 시간 간격으로 이미지를 뽑는 방식입니다. --interval, --max-frames
storyboard:fallback YouTube 영상 파일 다운로드가 막히면 storyboard 이미지를 받아 대체합니다. 자동 fallback이며 실패 기록은 video_download_failure.json에 남습니다.

캐시와 로그가 남는 위치

Codex Vision은 비교 실험용이라 “왜 실패했는지”, “다시 돌렸을 때 캐시를 썼는지”가 중요합니다. 그래서 중간 파일을 많이 남깁니다.

프레임 캐시

  • notebooks/recipe_loop_data/cache/codex-vision/_frames/<key>/frames/
  • frames.json
  • extraction_stats.json
  • extract-video-frames.log

Codex Vision 결과 캐시

  • batch_reports/
  • visual_notes.md
  • final.raw.md
  • final.json
  • run_meta.json

실패 기록

  • failure.json
  • parse_failure_before_repair.json
  • video_download_failure.json
  • 명령 로그에는 실패한 CLI 명령과 stderr가 남습니다.

중요한 안전 규칙

golden.json은 추출 중 읽지 않습니다.

golden.json은 정답지입니다. 추출 단계가 정답지를 보면 실험이 오염됩니다. 그래서 run-extraction.mjs는 source.json만 읽고, 채점은 별도 grade-extraction.mjs 같은 단계에서 수행합니다.

Gemini와 Codex Vision 차이

항목 Gemini 기본 경로 Codex Vision 경로
실행 방법 --provider 생략 또는 --provider gemini --provider codex-vision
영상 처리 Gemini client가 기존 방식으로 처리합니다. 프레임 추출 후 Codex CLI에 이미지 묶음을 넘깁니다.
실패 시 동작 기존 Gemini 경로의 실패 처리 방식을 따릅니다. Gemini로 fallback하지 않고 failure.json을 남깁니다.
비교 목적 기존 기준선입니다. 프레임 기반 추출이 더 나은지 나란히 비교하기 위한 새 경로입니다.

자주 만지는 옵션

--provider codex-vision
Codex Vision 경로를 켭니다. 없으면 Gemini가 기본입니다.
--max-frames 4
영상에서 최대 몇 장의 이미지를 볼지 정합니다. 숫자가 크면 더 자세하지만 느립니다.
--batch-size 4
Codex CLI 한 번에 몇 장의 이미지를 보낼지 정합니다.
--scene-detail dense
장면 변화를 얼마나 촘촘히 찾을지 정합니다. normal, dense, exhaustive가 있습니다.
--scene-selection balanced
후보 장면이 많을 때 영상 전체에 고르게 퍼지도록 고릅니다.
--timeout-ms 900000
외부 명령이 너무 오래 걸릴 때 끊는 시간입니다. 단위는 밀리초입니다.
--no-cache
기존 Codex Vision 결과 캐시를 쓰지 않고 새로 생성합니다.
--model, --codex-effort
Codex CLI에 쓸 모델과 추론 강도를 조정합니다.

초보자용 비유

Codex Vision은 요리 영상을 “처음부터 끝까지 통째로 보는 사람”이라기보다, “영상에서 사진 여러 장을 뽑아 메모한 뒤, 그 메모와 설명란을 합쳐 레시피 카드를 작성하는 사람”에 가깝습니다.

사진 뽑기

중요해 보이는 순간을 캡처합니다. 재료가 들어가거나 장면이 바뀌는 지점입니다.

메모하기

각 사진 묶음을 보고 “보이는 재료, 도구, 조리 동작, 자막”만 적습니다.

레시피 쓰기

메모와 텍스트 소스를 합쳐 ingredients와 steps가 있는 JSON을 만듭니다.
