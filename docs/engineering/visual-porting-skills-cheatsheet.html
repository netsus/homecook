Homecook visual porting

디자인 포팅과 화면 비교를 한 번에 이해하기

Figma나 prototype을 우리 프론트엔드에 옮길 때는 세 단계를 나누면 쉽습니다. 먼저 화면을 구현하고, 브라우저에서 실제로 확인하고, 마지막으로 참조 이미지와 점수로 비교합니다.

1. 구현 frontend-design으로 Figma/prototype을 코드에 옮깁니다.
2. 브라우저 확인 playwright-visual-review-loop로 모바일/데스크톱 화면을 확인합니다.
3. 비교 판정 visual-verdict로 참조 이미지와 구현 스크린샷을 비교합니다.

Playwright visual comparison이란?

Playwright가 실제 브라우저에서 화면을 PNG로 찍고, 저장된 기준 이미지와 비교하는 테스트입니다. 버튼 위치, 모달 크기, 카드 간격, 하단 탭 겹침처럼 눈으로 보던 차이를 자동으로 잡는 데 씁니다.

기준 이미지

정답으로 삼을 스크린샷입니다. 의도한 디자인이 확정됐을 때 저장합니다.

현재 이미지

테스트 실행 시점의 실제 프론트엔드 화면입니다.

diff 이미지

두 이미지가 다른 부분을 보여줍니다. 큰 차이는 레이아웃 문제일 가능성이 큽니다.

우리 저장소에서 쓰는 명령어

이미 Homecook에는 Playwright visual test가 설정되어 있습니다. 테스트 파일은 tests/e2e/qa-visual.spec.ts입니다.

비교 실행

현재 화면이 기준과 같은지 확인

pnpm test:e2e:visual
기준 갱신

의도한 디자인 변경일 때만 사용

pnpm test:e2e:visual:update
기준 갱신은 “테스트를 통과시키기 위해” 누르는 버튼이 아닙니다. 바뀐 PNG를 직접 보고 이 화면이 새 정답이라고 판단될 때만 사용해야 합니다.

가장 작은 테스트 예시

아래 코드는 개념 설명용입니다. 실제 우리 저장소 테스트는 폰트 고정, 애니메이션 제거, mock route까지 더 넣어서 스크린샷이 흔들리지 않게 합니다. 

import { test, expect } from "@playwright/test";

test("플래너 화면이 기준 이미지와 맞는다", async ({ page }) => {
  await page.goto("/planner");
  await expect(page.getByRole("heading", { name: "플래너" })).toBeVisible();

  await expect(page).toHaveScreenshot("planner-default.png", {
    fullPage: true,
    animations: "disabled",
    maxDiffPixels: 150,
  });
});
옵션 뜻 언제 쓰나
fullPage 스크롤 가능한 전체 페이지 캡처 페이지 전체 레이아웃을 보고 싶을 때 씁니다.
animations 애니메이션 처리 방식 움직임 때문에 매번 다른 사진이 나오지 않게 보통 끕니다.
maxDiffPixels 허용할 다른 픽셀 수 폰트 렌더링 같은 작은 차이를 일부 허용할 때 씁니다.
mask 특정 요소 가리기 날짜, 랜덤 이미지처럼 매번 바뀌는 영역을 비교에서 뺄 때 씁니다.
stylePath 스크린샷 전용 CSS 적용 커서, 개발 배너, 외부 위젯 같은 변동 요소를 숨길 때 씁니다.

바로 쓸 수 있는 3가지 스킬

처음에는 한 화면, 한 상태만 맡기는 것이 좋습니다. 예: 플래너 기본 화면, 식사 추가 모달 열린 상태.

frontend-design

구현/포팅

Figma나 prototype을 보고 우리 Next.js 컴포넌트에 맞게 실제 화면을 구현할 때 씁니다. 기존 API 계약과 상태 전이는 유지하고, visual만 맞추는 식으로 요청하면 안전합니다.

화면 구현 모달 구조 상태 UI 
예시 요청:
"이 Figma 플래너 frame을 frontend-design 방식으로 우리 플래너 화면에 포팅해줘.
동작 계약은 바꾸지 말고 visual만 맞춰줘."

기대효과: 버튼, 카드, 모달, empty/loading/error 상태를 코드 구조에 맞춰 정리합니다.

playwright-visual-review-loop

브라우저 QA

로컬 프론트엔드를 띄우고 모바일/데스크톱 스크린샷, 콘솔 에러, 페이지 에러, 실패한 요청을 함께 확인합니다.

모바일 깨짐 모달 위치 콘솔 에러 
예시 요청:
"localhost 플래너 화면을 playwright-visual-review-loop로 확인해줘.
390px, 320px, desktop에서 스크린샷 찍고 깨진 부분을 찾아줘."

기대효과: 실제 브라우저에서 버튼 가림, 모달 잘림, 가로 스크롤을 빠르게 찾습니다.

visual-verdict

비교 판정

참조 이미지와 현재 구현 스크린샷을 비교해서 0~100 점수, 다른 점, 다음 수정 제안을 냅니다.

레이아웃 차이 간격 차이 폰트/색상 차이 
예시 요청:
"이 참조 이미지와 현재 플래너 스크린샷을 visual-verdict로 비교해줘.
90점 미만이면 어떤 부분을 고쳐야 하는지 알려줘."

기대효과: “느낌이 다르다”를 “카드 간격이 넓다”, “버튼 위치가 낮다”처럼 바꿉니다.

추천 작업 순서

1. 기준 준비 Figma frame 또는 prototype 스크린샷을 정합니다.
2. 구현 frontend-design으로 한 화면을 포팅합니다.
3. 브라우저 확인 playwright-visual-review-loop로 실제 화면을 찍습니다.
4. 비교 판정 visual-verdict로 참조 이미지와 비교합니다.
5. 반복 수정 90점 이상 또는 눈에 띄는 차이가 없어질 때까지 반복합니다.
한 번에 전체 앱을 똑같이 만들기보다, 플래너 기본 화면, 식사 추가 모달 열린 상태처럼 작은 단위로 나누면 성공률이 높습니다.

실패했을 때 판단 기준

실패는 “화면이 달라졌다”는 신호입니다. 의도한 변경인지, 실제 버그인지 나누는 것이 중요합니다.

상황 해석 다음 행동
모달 위치나 카드 구조가 크게 다름 구현 버그일 가능성이 큽니다. 컴포넌트 구조나 CSS를 고친 뒤 다시 캡처합니다.
1~2px, 폰트 렌더링만 조금 다름 환경 차이일 수 있습니다. 허용치, 고정 폰트, 애니메이션 끄기를 확인합니다.
의도한 리디자인으로 전체가 달라짐 새 기준 이미지가 필요합니다. pnpm test:e2e:visual:update 후 PNG를 직접 확인합니다.

Playwright report를 엽니다

기준 이미지, 현재 이미지, diff 이미지를 보고 차이가 큰 영역부터 확인합니다.

레이아웃 차이인지 스타일 차이인지 나눕니다

버튼 위치, 모달 위치, 카드 개수 차이는 버그 가능성이 큽니다. 작은 색상 차이는 허용치 조정 대상일 수 있습니다.

의도한 변경이면 기준을 갱신합니다

pnpm test:e2e:visual:update 실행 후 바뀐 PNG를 직접 보고 커밋합니다.

의도하지 않은 변경이면 코드를 고칩니다

CSS나 컴포넌트 구조를 수정하고 다시 pnpm test:e2e:visual을 실행합니다.

자주 막히는 지점

1~2px씩 계속 달라짐

폰트, OS, 브라우저 렌더링 차이일 수 있습니다. 테스트 전용 폰트와 애니메이션 제거가 도움이 됩니다.

날짜/이미지 때문에 실패

날짜, 랜덤 추천, 외부 이미지는 변동 요소입니다. mock data, mask, stylePath를 씁니다.

모달 비교가 부정확함

전체 페이지 대신 dialog 같은 특정 locator만 캡처하면 더 안정적입니다.

Figma 캡처와 코드가 다름

Figma HTML 캡처는 raw frame입니다. Figma screenshot, node context, Playwright 결과를 같이 봐야 합니다.
