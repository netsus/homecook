# Mobile UX Rules

## 목적

이 문서는 집밥 서비스가 "웹처럼 보이는 모바일 화면"이 아니라
"익숙한 모바일 앱처럼 자연스럽게 느껴지는 화면"이 되기 위한 최소 규칙을 정리한다.

토큰 문서가 색과 간격을 다룬다면,
이 문서는 스크롤 구조, CTA 위계, 정보 밀도, 모바일 패턴 적합성을 다룬다.

## 기본 원칙

- 화면 전체가 아니라 **의도된 영역만** 스크롤되어야 한다.
- 사용자는 첫 화면에서 "지금 뭘 해야 하는지"를 바로 이해할 수 있어야 한다.
- 모바일에서 자주 보는 앱 패턴을 불필요하게 뒤집지 않는다.
- 이미 설득력 있게 작동하는 interaction model은 문제 수정 과정에서도 함부로 갈아엎지 않는다.
- 레이아웃은 좁은 폭에서 먼저 성립하고, 넓은 폭은 그 다음이다.

## 규칙

### 1. Whole-page horizontal scroll 금지

- 모바일에서 페이지 전체 wrapper에 가로 스크롤을 두지 않는다.
- 예외가 필요하면 workpack README와 authority report에 명시 승인 근거를 남긴다.
- page-level horizontal scroll과 localized horizontal scroll은 구분해서 본다.
- localized scroll 자체는 허용 가능하지만, 그것이 화면 전체 확장처럼 느껴지면 실패다.
- 가로 이동이 꼭 필요한 정보 구조라면:
  - 해당 영역만 별도 scroll container로 제한
  - 시각적 affordance를 제공
  - swipe 대상과 페이지 자체를 명확히 구분

### 2. Scroll containment가 보여야 한다

- 사용자가 어느 영역을 스크롤하는지 직관적으로 알 수 있어야 한다.
- sticky header, sheet, modal, tab panel, carousel, planner grid 같은 구조는
  스크롤 경계와 우선 스크롤 축이 분명해야 한다.

### 3. Primary CTA는 첫 화면에서 읽혀야 한다

- 핵심 행동은 시각 위계상 가장 강해야 한다.
- CTA가 secondary action들과 같은 무게로 섞이면 안 된다.
- 사용자가 가장 자주 누를 행동은 thumb-friendly 위치에 둔다.

### 3a. 컨트롤은 대상과 가까워야 한다

- 범위 이동, 필터, 정렬, 모드 전환처럼 콘텐츠 집합을 바꾸는 컨트롤은 그 콘텐츠 바로 가까이에 둔다.
- 설명 카드, 통계 칩, 장식성 섹션이 컨트롤과 실제 콘텐츠 사이를 길게 벌리면 안 된다.
- "무엇을 움직이는 버튼인지"를 사용자가 눈으로 즉시 연결할 수 있어야 한다.

### 4. 정보 구조는 일반적인 앱 기대를 존중한다

- planner, detail, list, modal, bottom sheet는 보편적인 모바일 앱 패턴과 크게 어긋나지 않게 설계한다.
- 익숙한 앱이면 모달이나 섹션 내부 스크롤로 처리할 내용을
  굳이 전체 페이지 구조로 바꿔 학습 비용을 높이지 않는다.

### 4a. 검증된 interaction model은 보존한다

- 기존 table, list, sheet, tab 같은 구조가 이미 제품의 mental model로 자리잡았다면, 레이아웃 문제를 고친다는 이유만으로 card stack이나 full-page flow로 바꾸지 않는다.
- interaction model을 바꿔야 한다면 단순 polish가 아니라 `high-risk-ui-change`로 취급한다.
- authority review는 문제를 지적할 수는 있어도, 사용자 승인 없이 interaction model 교체를 사실상 강제하는 근거가 되어서는 안 된다.

### 4b. 같은 단위 정보는 가능한 한 한 덩어리로 본다

- 같은 날짜, 같은 주문, 같은 레시피처럼 사용자가 하나의 단위로 인식하는 정보는 모바일에서 가능한 한 한 카드 또는 한 섹션 안에 모은다.
- 같은 단위를 여러 카드, 여러 축, 여러 스크롤 컨테이너로 쪼개면 스캔 비용이 급격히 올라간다.
- 반복 empty copy가 화면을 채우기 시작하면 단위 묶기나 요약 방식을 다시 설계한다.

### 5. 작은 모바일 sentinel을 기준으로 검토한다

- 기본 검토 폭 외에 더 좁은 mobile sentinel을 반드시 본다.
- 더 작은 viewport에서는 레이아웃 비율과 정보 구조를 유지한 채 폰트, 여백, 이미지, 카드 크기가 함께 비례 축소되어야 한다.
- 좁은 폭에서 아래 문제가 생기면 blocker다.
  - CTA 잘림
  - 핵심 정보 줄바꿈 붕괴
  - 고정 요소가 컨텐츠를 가림
  - 터치 타겟 축소

### 6. Modal / Sheet / Full page 선택은 사용자의 집중 흐름으로 정한다

- 짧은 선택, 필터, 부가 입력은 modal/bottom sheet를 우선 검토한다.
- 독립적인 맥락 전환이나 긴 작성 흐름만 full page를 쓴다.
- 사용자가 원래 보던 컨텍스트를 유지해야 하는 작업은 modal/sheet가 더 적합한 경우가 많다.

### 7. 시각적 무게는 브랜드보다 사용성에 먼저 복무해야 한다

- 색감이 예쁘더라도 CTA 구분, 계층, 대비를 해치면 실패다.
- decorative color가 content readability를 누르면 안 된다.
- 색 사용은 브랜드 표현보다 행동 유도와 상태 구분을 우선한다.

## Planner 추가 규칙

- planner는 "표가 넓어서 페이지 전체가 밀리는 앱"처럼 느껴지면 안 된다.
- planner의 탐색 구조는 날짜/끼니/컬럼 조작을 쉽게 느끼게 해야 한다.
- 가로 정보가 필요하면 planner 내부 컨테이너의 역할, 현재 위치, 이동 가능성을 분명히 드러낸다.
- planner에서 문제는 표 자체보다 page-level overflow일 가능성이 크므로, overflow 수정이 곧 table mental model 폐기를 의미하지는 않는다.
- 주간 범위 표시와 주간 이동 컨트롤은 planner 본문 바로 위에 붙어 있어야 한다.
- 같은 범위 메타데이터를 여러 카드에 중복 노출하거나 `ready` 같은 개발용 상태 문구를 사용자 화면에 드러내면 안 된다.
- 모바일 기본 뷰에서는 첫 화면에서 적어도 2일 이상의 요약이 보이도록 정보 밀도를 설계한다.
- 같은 날짜의 아침/점심/간식/저녁은 가능한 한 한 day card 안에서 함께 보이게 한다.
- 긴 설명 문장, 중복 통계 칩, 과한 여백 때문에 첫 day card가 아래로 밀리면 재설계 대상이다.
- planner가 4끼 고정 구조라면, 컬럼 수 확장 대신 copy와 badge 밀도를 조정해 한 카드 안에서 모두 읽히게 한다.
- 4슬롯을 보여주기 위해 여백과 장식이 실제 식사 정보보다 더 큰 비중을 차지하면 재설계한다.

## 판단 질문

authority 리뷰 전 아래 질문에 하나라도 `아니오`면 재설계 후보다.

- 첫 화면만 보고 핵심 행동이 무엇인지 바로 알 수 있는가
- 사용자가 어디를 스크롤해야 하는지 바로 알 수 있는가
- 보통의 모바일 앱 사용자 기대와 크게 어긋나지 않는가
- 좁은 폭에서 무너지지 않는가
- 색과 위계가 행동을 더 쉽게 만드는가
