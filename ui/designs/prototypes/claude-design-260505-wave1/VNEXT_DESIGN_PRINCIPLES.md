### 1.4 공식 서비스 계약/API/DB 확인 필요 (`contract check needed`)
- **메트릭 표시 4종(좋아요수/저장수/요리완료수/조회수)** — 데이터 소스 확정 필요. 화면정의서 v1.5.x에 메트릭 타입/집계 단위 명세 부재.
- **MEAL → LEFTOVERS 자동 이행** — pantry 미사용 사용자 분기 시 leftover row 자동 생성. 도메인 트리거 + status 머신 변경.
- **레시피별 "조리법" 칩 데이터** — 카드에서 제거 요청. DB 보존은 유지하되 카드 표시만 제거(표시 변경이지만 기존 사용처 영향 확인).
- **레시피북 생성 from 저장 모달** — 저장 sheet 안에서 새 레시피북 생성 흐름. POST /recipe-books 트랜잭션 위치 확정 필요.
- **이미있음/되살리기 토글 (SHOPPING_DETAIL)** — pantry-derived exclusion item이 클라이언트 보강인지 서버 응답 필드인지 정의 필요.
- **장보기 목록 이름(생성 일자 포함) + 공유** — 이름 자동 생성 규칙 + 공유 페이로드 포맷.
- **재료 카테고리 4종(주식/채소/단백질/양념)** — 현재 데이터 분류 체계와 매핑. 칩 필터 정렬 키.
- **양 단위 토글(g/ml)** — ingredient unit field 자유입력 vs enum 검토.
- **남은 요리 진입 from 식사추가 모달** — leftover → planner attach API 경로 확정.
- **SETTINGS 끼니 컬럼 추가/수정/삭제** — 사용자별 plan-slot 정의 테이블 필요. MVP DB 스키마 변경 가능성.
- **카카오/Apple 로그인 비활성화** — provider list 축소. supabase auth provider config + UX fallback 정책.

→ 모두 vNext에서는 **UI 모양만** 만들어두고 `contract check needed` 마커를 코드 주석으로 삽입.

## 2. 공통 디자인 원칙 (mobile-first, web 변환 규칙 포함)

### 2.1 시각 정리 원칙
1. **이모티콘은 의미 운반에 기여할 때만 둔다.** 카테고리 헤더, 끼니 컬럼명, 레시피 카드 메타 prefix(`#1`, `・`)에서 일괄 제거.
2. **정보 밀도를 줄여 핵심 액션을 도드라지게 한다.** 카드 한 장에 상태 배지가 3개 이상이면 가장 중요한 1개만 남기고 나머지는 카드 뒷면(상세 진입)으로.
3. **끝 라벨("보유", "#1")은 메타가 아니라 노이즈다.** 정렬·필터의 결과로 시각화하고 라벨은 제거.
4. **별점은 우리 서비스의 신호가 아니다.** 사용자 행동 메트릭(좋아요/저장/완료/조회)이 별점을 대체.
5. **카테고리 헤더는 정렬 신호일 뿐 시각 분리자가 아니다.** 항목 정렬은 유지하되 카테고리 라벨은 보이지 않게.

### 2.2 액션 위계 원칙
6. **CTA는 `명사+동사`로 통일하고 한 화면에 1순위 1개.** 예: `저장`(저장취소 X), `장보기`(장보기 목록 만들기 X), `요리하기`, `플래너에 추가`.
7. **CTA 라벨이 곧 동작.** SETTINGS의 `로그아웃`/`회원탈퇴` 자체를 누르면 실행되며, "실행/탈퇴" 같은 보조 버튼을 두지 않는다.
8. **하단 sticky 액션바는 1~2버튼.** RECIPE_DETAIL은 `플래너에 추가` + `요리하기` 2버튼, 저장은 이미지 옆 토글로 흡수.
9. **상태 배지는 메인 카드에서 빼고 진행 화면에서만 노출.** PLANNER 슬롯 카드는 배지 없이 `+ 음식` CTA 한 개만.

### 2.3 흐름·모달 원칙
10. **모달 안에서 끝낼 수 있는 건 모달 체인으로.** 식사 추가 옵션 6종 중 5종은 모달 체인, `직접등록`만 화면 전이.
11. **시트 액션 버튼 문구는 `행위 동사 1개`로 통일.** 닫기는 `X`/스와이프 다운으로만.
12. **인라인 dropdown vs 시트 모달 구분 기준** — 옵션 4개 이하 + 단일 선택 → dropdown / 5개 이상 또는 멀티 선택/필터 → 시트.
13. **"건너뛰기"/"확인" 같은 보조 액션은 줄바꿈 없이 텍스트 링크로.** COOK_MODE 소진 재료 화면.
14. **다중 선택 + 일괄 액션 패턴 통일.** PANTRY 재료, SHOPPING_DETAIL 항목, 묶음 선택 모두 동일 칩 토글 + sticky 하단 액션 바.

### 2.4 리스트·카드 정보 밀도
15. **레시피 카드 메타 = 1줄 이내.** prefix(`#1`), 분리자(`・`) 제거하고 `날짜 끼니 인분`만.
16. **레시피북 카드 좌우 정보 위치 swap.** 레시피 개수가 시각의 좌측, 책 설명/이름이 우측. 사용자가 책을 우선 인지하도록.
17. **LEFTOVERS와 ATE_LIST는 mirror 구조.** 양쪽 모두 반대편으로 가는 버튼을 같은 위치에 둔다(이력 ↔ 진행).

### 2.5 네비게이션·shell 원칙
18. **하단 4탭(홈/플래너/팬트리/마이) 모든 메인 화면에서 유지.** RECIPE_DETAIL 같은 상세에서도 유지.
19. **헤더는 `브랜드 또는 화면 타이틀` 단 한 가지 정체성만.** 프로필/장바구니는 헤더에서 추방, 마이페이지/플래너 진입으로 옮긴다.
20. **상단 가시성을 노치/dynamic island와 충돌하지 않게.** 로고는 safe-area 아래로 내려서 항상 보이게.

### 2.6 입력·폼 원칙
21. **텍스트 + 단위는 항상 묶음.** 양 입력칸 옆 g/ml 토글 분리하지 않음.
22. **숫자 전용 칸은 keyboard hint inputMode 보장.**
23. **검색 input을 클릭하면 검색 화면으로 진입.** 식사추가 모달의 검색 input은 표시는 inline, 행위는 page.

### 2.7 아이콘·이모티콘 정책
24. **시스템 의미가 합의된 아이콘만 둔다.** 휴지통(삭제), 점3개(more), `+`(추가), 체크(상태 변경), 검색돋보기(검색).
25. **장식용 이모지는 메인 화면에서 제거하고 빈 상태/페일오버 일러스트로만 허용.**

### 2.8 앱 ↔ 웹 변환 원칙 (이번 vNext의 핵심)
26. **모바일 sticky 하단 액션바 → 웹은 RIGHT-RAIL sticky 카드.** 데스크톱 RECIPE_DETAIL은 우측 1fr sticky 패널에 `플래너 추가/요리하기`를 카드로.
27. **모바일 BottomSheet → 웹은 centered dialog (480~620px).** 폭은 옵션 수에 비례.
28. **모바일 인라인 dropdown → 웹도 동일 dropdown 유지.** 시트로 승격하지 않음.
29. **모바일 가로 스크롤 카루셀 → 웹은 grid 또는 더 적은 카드의 row.** 카루셀 자체는 웹에서 지양.
30. **모바일 단일 화면 모달 체인 → 웹은 nested dialog 대신 좌측 step 사이드바 + 우측 본문(MENU_ADD/SHOPPING_FLOW 패턴).**
31. **모바일 BottomTab → 웹은 TopNav.** 같은 항목 4개를 좌측 정렬 가로 nav로.
32. **모바일 다중선택 후 sticky 하단 일괄 액션 → 웹은 선택 항목 수가 0이 아닐 때 상단 toolbar 슬라이드인.**
33. **앱 전용 패턴(요리모드 단일 화면 스크롤, 끼니 컬럼 + 음식 dashed 박스)은 웹에서 동일 의도를 유지하되 vertical/grid 비율을 더 widely 쓴다.**

### 2.9 토큰·컴포넌트 영향 (이번 vNext에서는 토큰 자체 변경 X)
- 기존 wave1 토큰(`--brand` 민트, `--cook-*` 등) 그대로 유지.
- **신규 의미 토큰만 도입**: `--metric-like`(red), `--metric-save`(brand), `--metric-cook`(accent), `--metric-view`(text-3) — RECIPE_DETAIL 메트릭 4종.
- 이외 색은 기존 wave1 팔레트 안에서 해결.

### 2.10 contract-check 마킹 규칙
- 모든 contract-check 항목은 코드에 다음 주석으로 삽입:
  ```jsx
  {/* CONTRACT_CHECK: <짧은 설명> — vNext에서는 UI shape만 */}
  ```
- 같은 주석을 화면별 변경 내역 표에 1:1로 매핑.

## 3. "앱 직접 적용" vs "웹 변환 적용" 기본 정책

| 영역 | 앱(모바일) | 웹(데스크톱) |
|------|-----------|-------------|
| 헤더 단순화(프로필/장바구니 제거) | ✅ 직접 | ✅ 직접 (TopNav도 동일하게 정돈) |
| 로고 safe-area 회피 | ✅ 직접 | (해당 없음 — 웹은 노치 없음) |
