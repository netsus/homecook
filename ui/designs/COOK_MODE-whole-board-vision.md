# COOK_MODE Whole Board Vision - 한 장 조리 보드 초안

> 목적: 기본 레시피는 전체 재료와 전체 조리법을 한 화면에서 크게 확인하고, 긴 레시피는 전체 보드를 스크롤해 확인하는 요리모드 초안
> 기준: `docs/화면정의서-v1.5.14.md` §14, `docs/요구사항기준선-v1.7.7.md` §1-3, `docs/design/mobile-ux-rules.md`
> 프로토타입: `ui/designs/prototypes/cook-mode-whole-board/index.html`

## 핵심 정정

이 초안의 핵심은 "현재 단계 1개를 크게 보여주는 것"이 아니다.

요리 중 사용자가 원하는 것은 스크린샷을 확대해 세워두듯이, **전체 재료와 전체 조리 순서를 한 번에 보는 것**이다. 따라서 이전/다음 단계 이동, 단계 rail, 현재 단계 전용 카드는 기본 화면에서 제거한다.

## 디자인 이름

**한 장 조리 보드**

- 앱: 재료 전체와 조리법 전체를 세로 한 화면에 압축 배치한 `레시피 포스터`.
- 웹/태블릿: 재료 전체와 조리법 전체를 넓은 화면에 나란히 둔 `대형 조리 보드`.
- 프로토타입 페이지: 웹 화면 가로폭이 줄지 않도록 앱 초안과 웹 초안을 위아래로 배치한다.
- 공통: 사용자가 손을 씻거나 화면을 만지지 않아도 읽을 수 있도록 수량과 단계 번호를 크게 둔다.
- 공통: 요리모드 안에서 인분 조절은 하지 않는다. 확정된 인분만 표시한다.
- 공통: 완료/취소, 화면 꺼짐 방지, 소진 재료 sheet/API 계약은 유지한다.
- 공통: 요리 컨텍스트는 `플래너 요리` 같은 내부 분류명 대신 기존 서비스처럼 날짜와 끼니를 표시한다.

## 앱 화면

### 구조

1. 상단 고정 정보
   - 뒤로/취소 아이콘
   - 레시피명
   - 확정 인분
   - 화면 안 꺼짐 상태
2. 전체 재료 보드
   - 2열 그리드
   - 재료명보다 수량을 더 강하게 표시
   - `component_label`이 있으면 작은 섹션 라벨로 묶되, 화면을 길게 만들지 않는다.
3. 전체 조리 순서 보드
   - 모든 단계를 한 화면에 보이는 밀도로 배치
   - 단계 번호는 크게, instruction은 짧고 굵게
   - 불세기/시간은 기본 화면에서 표시하지 않는다.
   - 조리 순서별 필요 재료는 표시하지 않는다. 전체 재료가 같은 화면에 있고, 단계별 재료 추출이 누락/오인식될 수 있기 때문이다.
4. 하단 고정 액션
   - 취소
   - 요리 완료

### 스크롤 정책

- 기본 레시피는 390 x 844 앱 화면 기준으로 재료 6-8개, 조리 단계 4-6개가 첫 화면에 들어오는 밀도를 목표로 한다.
- 긴 레시피는 단계 이동 UI를 추가하지 않고, 전체 보드 자체에 세로 스크롤을 허용한다.
- 스크롤이 생겨도 재료와 조리 순서라는 정보 구조는 유지한다. 사용자는 현재 단계별 페이지를 넘기는 것이 아니라 전체 레시피 보드를 훑는다.

## 웹/태블릿 화면

### 구조

- 좌측: 전체 재료 보드
- 우측: 전체 조리 순서 보드
- 상단: 레시피명, 인분, 날짜/끼니, 화면 안 꺼짐 상태
- 하단: 취소, 요리 완료

웹은 앱보다 화면이 넓으므로 전체 단계를 더 크게 보여준다. 사용자는 노트북이나 태블릿을 조리대 끝에 두고 전체 흐름을 한 번에 읽는다.

## 시각 원칙

- 배경은 조리대처럼 어두운 graphite 계열을 사용해 주방 조명에서 눈부심을 줄인다.
- 수량과 주요 완료 CTA는 현재 브랜드 파란색 `#00A1FF`를 사용한다.
- 수량 숫자는 멀리서 읽히되 너무 무겁지 않게 `700` 안팎의 weight를 사용한다.
- 기존 노란 수량 강조안은 보류한다. 브랜드 파란 수량이 앱 전체 CTA/선택 상태와 더 통일된다.
- 조리방법은 기존 method color helper와 같은 의미의 색상으로 작은 bar/pill에만 사용한다.
- 조리방법 태그는 instruction과 같은 줄에 섞지 않고, 조리법 텍스트 바로 위에 둔다.
- 불세기와 조리시간은 조리 순서 row에서 표시하지 않는다. 한 화면 정보 밀도와 추출 신뢰성을 우선한다.
- 조리 순서 번호는 작은 정사각형 파란 박스로 표시한다.
- 배경은 기존 서비스 다크모드 배경색인 `#111923`을 사용한다.
- `화면 안 꺼짐` 상태는 설정의 요리모드 화면 꺼짐 방지가 켜져 있고 브라우저에서 Wake Lock 요청을 사용하는 상태를 뜻한다. 구현 시 설정 OFF/미지원/요청 실패 상태를 구분해 copy나 표시 여부를 정한다.
- 섹션 오른쪽 보조 문구(`꺼내둘 총량`, `한 화면에 보기`)는 제거해 정보 밀도를 줄인다.
- 테두리와 면은 실제 앱 토큰에 맞춰 카드 10-12px, 큰 패널 14px 안팎의 radius를 사용한다. 조리대 도구 느낌은 유지하되 기존 서비스 화면과 너무 다르게 각지지 않게 한다.

## 계약 영향

이 초안은 다음을 바꾸지 않는다.

- `meals.status` 전이 규칙
- 플래너 경유/독립 요리 분리
- 요리모드 내 인분 조절 금지
- `{ success, data, error }` API wrapper
- 완료 시 소진 재료 sheet와 `consumed_ingredient_ids`
- 화면 꺼짐 방지 설정

추후 구현 시 public contract 변경 없이 가능한 범위:

- 현재 단계 index 상태 제거 또는 보조 상태로 격하
- `ingredients_used`, `heat_level`, `duration_text`를 전체 단계 row에 압축 표시
- 긴 레시피에서 클라이언트 표시 밀도만 조정

contract-evolution 검토가 필요한 범위:

- 단계별 재료 사용량을 API가 항상 구조화해서 보장해야 하는 경우
- 요리 중 체크/진행률을 서버에 저장해야 하는 경우
- 음성, 타이머, 자동 스크롤 같은 새 기능을 넣는 경우

## 초안 판정 질문

- 앱에서 전체 재료와 전체 조리법이 한 화면에 보이는가: 예
- 이전/다음 단계 이동 없이 요리할 수 있는가: 예
- 재료 수량이 멀리서도 먼저 보이는가: 예
- 웹에서 전체 흐름이 한눈에 보이는가: 예
- 기존 완료/팬트리/남은요리 계약을 깨지 않는가: 예
