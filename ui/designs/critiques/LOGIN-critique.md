## 마이너 이슈 (권장 수정)

| # | 위치 | 문제 | 제안 |
|---|------|------|------|
| 1 | LoginRequiredModal 와이어프레임 | 닫기(X) 버튼이 와이어프레임에 표현되지 않음. 인터랙션 노트에 ESC/배경 탭은 명시됐으나, 모달 헤더 영역의 X 버튼 또는 드래그 핸들이 와이어프레임 박스에 없어 구현 시 빠질 위험이 있음 | 모달 상단에 X 아이콘(44×44px 터치 타겟) 또는 드래그 핸들을 와이어프레임에 추가. 화면정의서 §0-3은 "[로그인] [취소]" 버튼만 명시하므로 취소 버튼이 닫기를 충족하더라도 시각적 명시 필요 |
| 2 | Step 2 와이어프레임 (닉네임 입력 비활성 상태) | 에러 메시지 "2자 이상 입력해주세요"가 초기 빈 상태에서도 노출되는지 아닌지가 와이어프레임 텍스트로는 불명확. 와이어프레임 박스에 메시지가 표시돼 있어 초기 진입 시부터 에러로 오해될 수 있음 | "2자 이상 입력해주세요"는 한 번이라도 입력 후 지웠을 때(dirty 상태)에만 표시하는 조건을 컴포넌트 상세 설명에 명시. 초기 빈 상태 와이어프레임에서는 해당 문구를 생략하거나 "(에러 시만 노출)" 주석 추가 |
| 3 | Step 2 → return-to-action 인터랙션 노트 | 신규 회원(`is_new_user=true`) 닉네임 저장 성공 후 `returnToAction` 확인 후 이동한다고 명시됐으나, 신규 회원은 플로우맵 ② 기준으로 "자동 생성: meal_plan_columns ×3, recipe_books ×3" 이후 복귀하는 것이 정확함. 서버 자동 생성 완료 전 복귀 시 UI 상태 불일치 가능성 | 인터랙션 노트에 "신규 회원은 서버 응답(PATCH /auth/profile 201) 수신 후 returnToAction 처리"임을 명시. 로딩 스피너 상태에서 사용자가 이탈하지 못하도록 pointer-events-none 처리가 이미 있어 방향은 맞음 |
| 4 | Step 1 에러 배너 와이어프레임 | 에러 상태 와이어프레임에서 에러 배너가 STATUS BAR 바로 아래 최상단에 배치됐는데, 컴포넌트 상세(§에러 배너)에는 "로고 영역 위, 화면 상단"으로 설명됨. 두 표현이 불일치. 정상 상태 와이어프레임에는 상단 여백(--space-12)이 있는데, 에러 배너가 그 여백 안에 들어가는지 아니면 여백을 밀어내는지 불명확 | 에러 배너 위치를 "status bar 직하 → 상단 여백 제거 → 배너 → 로고 영역" 순서로 와이어프레임에서도 일치시키거나, 배너가 "로고 위 --space-8 아래"임을 명확히 표기 |
| 5 | Step 2 키보드 레이아웃 대응 | 닉네임 인풋 포커스 시 소프트 키보드가 올라오면 "시작하기" 버튼이 가려질 수 있음. 이 케이스가 와이어프레임과 컴포넌트 상세에 누락됨. design-generator 스스로 "검토 필요 항목"으로 플래그를 달았으나 설계 수준에서 처리 방향이 결정되지 않은 채 구현으로 넘어갈 경우 UX 품질 저하 위험 | 키보드 오버랩 대응 와이어프레임을 추가하거나, 컴포넌트 상세에 "키보드 표시 시 버튼은 키보드 바로 위에 고정(position: sticky / padding-bottom 동적 조정)" 정책을 명시 |
| 6 | 에러 배너 배경색 토큰 | 에러 배너 배경이 `rgba(255,108,60,0.08)`로 하드코딩됨. 이는 `--brand(#FF6C3C)`의 8% 알파 값이나, 확정 토큰 목록에는 이 색상 표현 방식이 없음. 컴포넌트 상세와 와이어프레임 주석이 각각 다른 표기 방식(`bg-[#fff3f0]` vs `rgba(255,108,60,0.08)`)을 사용해 구현 시 혼선 가능 | 표기를 `--brand` 기반으로 통일. 예: `bg-[color-mix(in_srgb,_var(--brand)_8%,_transparent)]` 또는 Tailwind에서 `bg-[#FF6C3C]/8`로 단일화 |

## 체크리스트 결과

### A. 요구사항 정합성

- [x] 화면 정의서(§4)의 모든 컴포넌트가 와이어프레임에 포함됐는가

- [x] 문서에 없는 컴포넌트/필드/기능이 추가됐는가
  - 이용약관/개인정보처리방침 하단 링크: 화면정의서 §4에 명시 없음. 다만 설계 문서 내에 "세션 3 와이어프레임에 포함, 서비스 필수 요소"로 보완 판단을 명시함. 법적 필수 요소이므로 추가 허용으로 판정
- [x] 로그인 게이트 대상 액션이 모두 처리됐는가 (즉시 이동 X -> 모달 -> return-to-action)
  - 요구사항 §2-3의 3단계 패턴 충실히 구현. 모달 [로그인하기] 탭 시 LOGIN으로 이동 + returnToAction 포함, 취소 시 이전 화면 유지 명시
- [x] read-only 상태가 올바르게 반영됐는가
  - LOGIN 화면은 read-only 대상 아님. 해당 없음
- [x] 삭제된 엔드포인트 `DELETE /recipes/{id}/save`가 UI에 등장하지 않는가

## 추가 의견

### design-generator가 플래그한 항목 검토 결과

1. **구글 버튼 `--surface` + `--line` 적용**: 구글 브랜드 가이드라인 준수가 우선이므로 현행 설계가 올바름. `--brand` 적용 불필요.

2. **앱 로고 에셋 미확정**: 텍스트 폴백 설계는 적절한 방어 설계임. SVG 에셋 확정 후 교체만 하면 됨.

3. **Step 2 키보드 레이아웃**: 마이너 이슈 5로 등록하여 구현 전 정책 확정 권고.

4. **LoginRequiredModal 아이콘 SVG 교체 시 터치 타겟**: 아이콘 자체는 인터랙션 없는 장식 요소이므로 44×44px 터치 타겟 기준 적용 대상이 아님. 단, 주변 버튼의 터치 타겟은 이미 확보됨.
