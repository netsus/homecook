## 체크리스트 결과

### A. 요구사항 정합성
- [x] 화면 정의서의 모든 컴포넌트가 와이어프레임에 포함됐는가 -- SS 15: 남은요리 리스트(최근순), 아이템 카드(레시피명, 요리완료일, [다먹음], [플래너에 추가]), Empty 상태 모두 포함. 카드 구성이 정의서 명세와 1:1 대응.
- [x] 문서에 없는 컴포넌트/필드/기능이 추가됐는가 -- PlannerAddSheet는 화면정의서 SS 15에 "[플래너에 추가] -> 날짜/끼니 선택 + 계획 인분 입력"으로 명시되어 있고, SS 3 PlannerAddPopup과 동일한 구조를 재사용하므로 정의서 범위 내. 썸네일 미포함 결정(467행)도 정의서를 존중한 판단.
- [x] 로그인 게이트 대상 액션이 모두 처리됐는가 -- 화면 자체가 로그인 필수이며, 세션 만료 예외에 대해 Unauthorized 상태(180-198행)가 모달 오버레이 + return-to-action으로 처리됨. 로그인 게이트 패턴(즉시 이동 X -> 모달 -> return-to-action)을 정확히 준수.
- [x] read-only 상태가 올바르게 반영됐는가 -- 이 화면에는 read-only 모드가 해당하지 않음. 모든 카드가 액션 가능 상태.
- [x] 삭제된 엔드포인트 `DELETE /recipes/{id}/save`가 UI에 등장하지 않는가 -- 등장하지 않음.

### B. 공통 상태 커버리지
- [x] Loading 상태 (스켈레톤/인디케이터) 포함 -- 스켈레톤 카드 3개(97-125행)가 텍스트 2줄 + 버튼 2개 placeholder로 구성됨. 실제 카드 구조와 일치하는 스켈레톤.
- [x] Empty 상태 (안내 + CTA) 포함 -- "남은 요리가 없어요" + "요리를 완료하면 여기에 저장돼요" + [플래너로 돌아가기] CTA(129-152행). 정의서 SS 15의 Empty 문구 "남은 요리가 없어요. 요리를 완료하면 여기에 저장돼요."와 정확히 일치.
- [x] Error 상태 (안내 + [다시 시도]) 포함 -- "남은요리를 불러오지 못했어요" + [다시 시도] CTA(156-176행).
- [x] read-only 상태가 필요한 화면에서 수정 UI 비노출 -- 해당 없음.

### C. 내비게이션 & 플로우
- [x] 하단 탭 4개 (홈/플래너/팬트리/마이페이지) 구조 일관성 -- 4탭 고정, 플래너 활성 탭(338행). PLANNER_WEEK의 하위 화면이므로 플래너 탭 활성이 올바름.
- [x] 유저 Flow맵과 진입/이탈 경로 일치 -- 유저플로우 SS 6: `PLANNER_WEEK -> [남은요리] -> LEFTOVERS`, `[다먹음] -> ATE_LIST`, `[플래너에 추가] -> 날짜/끼니/인분 -> Meal 생성`. 모두 인터랙션 노트(374-389행)에 반영됨.
- [x] 뒤로가기 동작 명시 여부 -- AppBar 좌측 화살표 + 브라우저 back -> PLANNER_WEEK 복귀 명시(376행).
- [x] 플로우 단절 지점 없는가 -- 기본 상태: [다먹음] -> ATE_LIST, [플래너에 추가] -> PlannerAddSheet. Empty: [플래너로 돌아가기] -> PLANNER_WEEK. Error: [다시 시도] -> 재호출. 모든 상태에서 다음 행동 경로가 존재. 바텀시트 닫기 동작(X, 바깥 영역, 아래 스와이프)도 명시(386행). 단절 없음.

### D. UX 품질
- [x] 터치 타겟 최소 44px 준수 -- 뒤로가기 화살표 44x44px(260행), [다먹음] min 44px(84행, 286행), [플래너에 추가] min 44px(91행, 293행), PlannerAddSheet 내 +/- 44x44px(362행), CTA 44px/48px. 모든 인터랙티브 요소에서 44px 이상 명시.
- [x] 모바일 퍼스트 (375px 기준) 레이아웃 -- 와이어프레임이 375px 기준으로 작성됨(28행).
- [x] 작은 모바일 sentinel에서도 구조가 유지되는가 -- 320px 대응 섹션(446-461행)에서 패딩 축소, 버튼 flex-wrap 미허용(한 행 유지), min-width 조정, blocker 체크리스트 포함. 두 버튼이 320px에서 같은 행에 배치 가능한지 계산이 필요하지만(509행), 설계에서 이를 검토 요청으로 명시했고 min-width 축소로 대응 방향이 제시됨.
- [x] 핵심 액션이 시각적으로 명확한가 -- [플래너에 추가]가 `--brand` filled primary, [다먹음]이 outlined secondary. CTA 위계가 서비스 핵심 사이클(남은요리 -> 플래너 재등록)에 부합(477행). Primary CTA가 카드 우측에 배치되어 thumb-friendly zone 활용.
- [x] whole-page horizontal scroll을 유도하지 않는가 -- 스크롤 정책(403-411행)에서 "페이지 전체: 가로 스크롤 금지" 명시. PlannerAddSheet 날짜 chip rail만 localized 가로 스크롤.
- [x] scroll containment가 명확한가 -- AppBar sticky 상단, 탭바 sticky 하단, 본문 세로 스크롤. PlannerAddSheet 날짜 chip rail은 바텀시트 내부 localized scroll(410행). 스크롤 정책 테이블에서 각 영역의 스크롤 방향과 고정 여부가 명확히 정의됨.
- [x] 장보기 D&D (sort_order) UI가 명확한가 -- 해당 없음.
- [x] 팬트리 제외 섹션 2-영역 구조가 올바른가 -- 해당 없음.
- [x] AI스러운 제네릭 UI 사용 여부 -- 글로우, 과도한 그라디언트, AI 장식 요소 없음. 카드 리스트 + 바텀시트 조합으로 모바일 네이티브 패턴 준수.

### E. 도메인 규칙 정합성
- [x] `meals.status` 전이 (registered -> shopping_done -> cook_done) 표현 정확성 -- 이 화면은 `leftover_dishes`를 다루며 `meals`와 직접 관련이 없다. [플래너에 추가] 시 `POST /meals`로 새 Meal을 생성(status='registered')하는 것은 올바른 전이. Meal 생성 시 `is_leftover=true, leftover_dish_id` 포함(248행, 368행).
- [x] 독립 요리는 meals 상태를 바꾸지 않는 것이 UI에서 명확한가 -- 이 화면은 남은요리 관리 화면이며, 독립 요리로 생성된 남은요리도 동일하게 표시된다. meals 상태 변경과 무관한 화면이므로 혼동 위험 없음.
- [x] 팬트리는 수량이 아닌 보유 여부만 표시 -- 해당 없음 (이 화면에 팬트리 정보 없음).
- [x] 요리 모드에서 인분 조절 UI가 없는가 -- 해당 없음 (이 화면은 요리 모드가 아님). PlannerAddSheet에 인분 조절 stepper가 있는데 이는 "계획 인분 입력"으로, 요리 모드 내부의 인분 조절과는 다르다. 정의서 SS 15에 "계획 인분 입력"이 명시되어 있으므로 적절함.
- [x] 저장 가능한 레시피북 타입 (saved/custom) 만 표시 -- 해당 없음 (이 화면에 레시피북 관련 기능 없음).

- [x] 태그/칩에 `--olive` 사용됐는가 -- PlannerAddSheet의 끼니 선택 활성 상태에 `--olive (#1f6b52)` 사용(239행, 359행). design-tokens.md의 `--olive` 값과 일치.

- [x] 카드 border-radius 16px 준수됐는가 -- `--radius-lg (16px)` 명시(68행, 276행). 컴포넌트 규칙과 일치.

## 부록: 검토 필요 항목에 대한 의견

설계 파일 507-514행에서 design-critic에게 검토를 요청한 항목에 대해:

| 요청 항목 | 의견 |
|----------|------|
| [다먹음] 후 ATE_LIST 즉시 이동 vs LEFTOVERS 유지 | 마이너 #2에서 해석 B(LEFTOVERS 유지) 권장. 연속 처리 패턴에서 화면 전환 비용이 크다. |
| 썸네일 없는 카드의 시각적 밀도 | 적절함. 정의서에 썸네일 언급이 없고, 남은요리는 "관리" 목적이므로 텍스트 중심 카드가 맞다. 카드가 밋밋하게 느껴지면 레시피명 좌측에 작은 아이콘(냄비/접시, 24px)을 추가하는 것으로 충분. 썸네일까지는 불필요. |
| 320px에서 두 버튼 같은 행 배치 | 계산: 320px - 12px*2(컨테이너) - 12px*2(카드 패딩) = 272px 사용 가능. [다먹음](약 72px) + gap(8px) + [플래너에 추가](약 120px) = 200px. 72px 여유로 충분히 한 행 배치 가능. blocker 아님. |
| [다먹음] 낙관적 UI fade-out + undo | 마이너 #3에서 undo snackbar 권장. |
| 빈 상태 일러스트 | 설계에서 "작은 중립 이미지(냄비 또는 접시)"로 기술한 것은 적절. 과도한 일러스트보다 48px 정도의 line icon이 Baemin vocabulary와 조화. |
| 카드가 1개일 때 화면 밀도 | 허용 범위. 리스트 화면에서 1개 카드는 자연스러운 상태이며, 의도적으로 여백을 채우려 하면 오히려 어색해진다. |
| PlannerAddSheet 기본 선택 날짜 | 오늘(today)을 기본 선택으로 하는 것이 직관적. "다음 빈 슬롯"은 계산 비용이 크고, 남은요리를 오늘 먹으려는 의도가 가장 높다. |
| [다먹음] 확인 모달 필요 여부 | 확인 모달은 불필요. undo snackbar(마이너 #3)가 더 적합한 패턴. |
