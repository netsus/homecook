## 체크리스트 결과

### A. 요구사항 정합성
- [x] 화면 정의서의 모든 컴포넌트가 와이어프레임에 포함됐는가 -- SS 19: Account 섹션(닉네임, 소셜 제공자, 프로필 이미지, ⚙️ 버튼), 탭 영역(레시피북/장보기 기록), Tab 1 UI(시스템 섹션 + 커스텀 섹션 + ⋯ 메뉴 + [+ 새 레시피북]), Tab 2 UI(최신순, SHOPPING_DETAIL 이동) 모두 포함. recipe_count 표시도 반영됨.
- [x] 문서에 없는 컴포넌트/필드/기능이 추가됐는가 -- 완료/진행 중 뱃지는 화면정의서에 명시적 언급이 없으나, is_completed 필드가 API에 존재하고 사용자에게 유용한 정보이므로 합리적 추가. 삭제 확인 다이얼로그도 UX 안전장치로 적절.
- [x] 로그인 게이트 대상 액션이 모두 처리됐는가 -- 전체 화면이 로그인 필수. Unauthorized 상태에서 모달 오버레이 + return-to-action 패턴 준수.
- [x] read-only 상태가 올바르게 반영됐는가 -- "해당 없음" 판정이 정확함. 장보기 기록의 read-only는 SHOPPING_DETAIL에 위임.
- [x] 삭제된 엔드포인트가 UI에 등장하지 않는가 -- 등장하지 않음.

### B. 공통 상태 커버리지
- [x] Loading 상태 (스켈레톤/인디케이터) 포함 -- 프로필 + 카드 스켈레톤이 실제 구조를 반영하여 작성됨.
- [x] Empty 상태 (안내 + CTA) 포함 -- 레시피북 탭 (커스텀 없음 + [새 레시피북] CTA 유지), 장보기 기록 탭 ("저장된 장보기 기록이 없어요" + [플래너로 이동] CTA). 시스템 레시피북은 항상 존재하므로 완전 빈 탭은 없다는 점이 명확히 기술됨.
- [x] Error 상태 (안내 + [다시 시도]) 포함 -- "데이터를 불러오지 못했어요" + [다시 시도] CTA.
- [x] Unauthorized 상태 포함 -- 로그인 게이트 모달 + return-to-action.

### C. 내비게이션 & 플로우
- [x] 하단 탭 4개 구조 일관성 -- 4탭 고정, 마이페이지 활성 탭.
- [x] 진입/이탈 경로 일치 -- 하단 탭 진입, SETTINGS(17c)/RECIPEBOOK_DETAIL(17b)/SHOPPING_DETAIL(기존) 이탈 모두 명시.
- [x] 탭 전환 동작 명시 여부 -- 레시피북(기본) ↔ 장보기 기록 전환이 인터랙션 노트에 포함됨.
- [x] 플로우 단절 지점 없는가 -- 모든 상태에서 다음 행동 경로가 존재. Empty에서 CTA 제공, Error에서 재시도 가능.

### D. UX 품질
- [x] 터치 타겟 최소 44px 준수 -- ⚙️ 44x44, ⋯ 44x44, 탭 바 44px 높이, 카드 전체 탭 가능(48px 이상), [다시 시도] 44px, 삭제 다이얼로그 버튼 44px. 모든 인터랙티브 요소에서 44px 이상 명시.
- [x] 모바일 퍼스트 (375px 기준) 레이아웃 -- 와이어프레임이 375px 기준.
- [x] 작은 모바일 sentinel에서도 구조가 유지되는가 -- 320px 대응 섹션에서 패딩 축소, blocker 체크리스트 포함.
- [x] 핵심 액션이 시각적으로 명확한가 -- [+ 새 레시피북]은 점선 CTA 카드로 어포던스 제공, ⋯ 메뉴는 커스텀만 노출, 탭 전환은 --brand 밑줄로 명확.
- [x] whole-page horizontal scroll을 유도하지 않는가 -- 스크롤 정책에서 "가로 스크롤 금지" 명시. 가로 스크롤 요소 없음.
- [x] scroll containment가 명확한가 -- 프로필 스크롤 포함, 탭 바 sticky, 하단 탭바 sticky. 스크롤 정책 테이블에서 각 영역이 명확히 정의됨.
- [x] AI스러운 제네릭 UI 사용 여부 -- 글로우, 과도한 그라디언트 없음. 프로필 + 탭 + 리스트 패턴은 네이티브 앱 마이페이지의 전형적 구조.

### E. 도메인 규칙 정합성
- [x] 시스템 레시피북 불변성이 UI에서 명확한가 -- ⋯ 메뉴 미노출로 UI 차단, 서버 403은 fallback. 설계 결정 사항에서 명시적으로 기술.
- [x] recipe_count source of truth가 반영됐는가 -- README에 book_type별 집계 정책이 명시되어 있고, 설계에서 count 표시 위치가 정의됨.
- [x] 커스텀 CRUD 정책이 올바른가 -- 생성(POST), 이름 변경(PATCH 커스텀만), 삭제(DELETE 커스텀만 + 확인 다이얼로그) 모두 반영.
- [x] 장보기 기록 read-only 정책이 올바른가 -- 목록 조회만 가능, 수정/삭제 액션 없음. SHOPPING_DETAIL로 이동 시 read-only mode.
- [x] 저장 가능한 레시피북 타입 제한 -- 이 화면에서는 CRUD만 다루므로 해당 없음. saved/custom 제한은 04-recipe-save에서 구현됨.

### F. 디자인 토큰 준수 (design-tokens.md 기준)
- [x] CTA 버튼에 `--brand` 사용됐는가 -- [다시 시도]: --brand. [+ 새 레시피북]: --brand 텍스트. 삭제: --danger 사용 (적절한 예외).
- [x] 카드 배경에 `--surface (#ffffff)` 명시됐는가 -- 모든 카드에서 --surface 명시.
- [x] 태그/칩에 `--olive` 사용됐는가 -- 완료 뱃지에 --olive 사용.
- [x] 보조 텍스트에 적절한 토큰 사용됐는가 -- 메타 텍스트에 --text-3 (#868E96) 일관 사용. 소셜 제공자, recipe_count, 날짜 범위 등.
- [x] 카드 border-radius 16px 준수됐는가 -- --radius-lg (16px) 일관 명시.
- [x] 수평 여백 16px(모바일) 기준 준수됐는가 -- --space-4 (16px) 기본, 320px에서 --space-3 (12px) 축소.
- [x] 확정 토큰 외 임의 색상이 사용됐는가 -- 마이너 #1의 `rgba(31,107,82,0.1)` 1건 외 임의 색상 없음. --danger (#E03E3E)는 삭제 액션용으로 적절한 시맨틱 색상.

## 부록: 검토 필요 항목에 대한 의견

설계 파일의 design-critic 검토 필요 항목에 대해:

| 요청 항목 | 의견 |
|----------|------|
| 프로필 섹션 스크롤 vs sticky | 적절함. 80px 프로필 영역이 항상 sticky면 콘텐츠 영역이 줄어든다. 탭 바만 sticky면 현재 탭 인지 + 전환이 모두 가능. |
| 시스템 레시피북 아이콘 | 텍스트 아이콘(연필/북마크/하트)은 서비스 톤과 무난하게 조화. Stage 4에서 Baemin vocabulary에 맞는 line icon으로 교체 가능. |
| [+ 새 레시피북] 점선 카드 | 적절함. 점선 + --brand 텍스트는 "추가 가능" 어포던스를 전달하는 익숙한 패턴(Google Keep, Trello 등). 일반 버튼보다 리스트 흐름에 자연스럽다. |
| 장보기 뱃지 색상 구분 | olive=완료, brand=진행 중은 직관적. olive는 PLANNER_WEEK에서도 완료/활성 의미로 사용되므로 일관성이 있다. |
| 320px 시스템 레시피북 | 계산: 320px - 12px*2(컨테이너) - 12px*2(카드 패딩) = 272px. "내가 추가한 레시피"(약 140px) + spacer + "12"(약 20px) = 약 160px. 충분한 여유. blocker 아님. |
| 이름 변경 인라인 vs 모달 | 인라인 텍스트 필드 권장 (마이너 #2). 컨텍스트 유지가 모바일에서 중요. |
| 커스텀 레시피북 10개 이상 | 허용 범위. 리스트 화면이므로 스크롤로 충분히 대응 가능. 검색/필터는 이 시점에서 과도한 기능이다. |
| 프로필 이미지 fallback | 이니셜 아바타(--brand-soft 배경 + --brand 텍스트)는 충분히 친근한 디자인. Stage 4에서 Baemin 스타일 일러스트 아바타로 교체 가능하나 Stage 1 blocker 아님. |
