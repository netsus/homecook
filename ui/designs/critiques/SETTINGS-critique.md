## 체크리스트 결과

### A. 요구사항 정합성
- [x] 화면 정의서의 모든 컴포넌트가 와이어프레임에 포함됐는가 -- SS 19-D: 요리모드 화면 꺼짐 방지 토글(WakeLockToggle), 로그아웃(LogoutButton + LogoutConfirmDialog), 회원탈퇴(WithdrawButton + WithdrawConfirmDialog), 닉네임 변경(NicknameRow + NicknameEditSheet) 모두 포함.
- [x] 문서에 없는 컴포넌트/필드/기능이 추가됐는가 -- Screen Wake Lock API 비지원 환경 대응이 추가되었으나, 이는 합리적인 엣지 케이스 처리로 적절. 문서 범위를 넘지 않음.
- [x] 로그인 게이트 대상 액션이 모두 처리됐는가 -- SETTINGS 전체가 로그인 필수 화면. 진입 자체가 MYPAGE(로그인 필수)의 하위이므로 별도 게이트 불필요. 설계에서 모든 인터랙션 행에 "로그인 필요: Y"로 정확히 표기.
- [x] read-only 상태가 올바르게 반영됐는가 -- SETTINGS에서는 read-only 대상 컴포넌트가 없으므로 해당 없음. 적절한 판정.
- [x] 삭제된 엔드포인트 `DELETE /recipes/{id}/save`가 UI에 등장하지 않는가 -- 등장하지 않음.

### B. 공통 상태 커버리지
- [x] Loading 상태 (스켈레톤/인디케이터) 포함 -- 토글 스켈레톤 + 닉네임 스켈레톤이 실제 카드 구조를 반영한 ASCII 와이어프레임으로 기술됨. 토글 개별 로딩(disabled + shimmer)도 별도 명시.
- [x] Empty 상태 (안내 + CTA) 포함 -- SETTINGS는 설정 항목이 고정(4개)이므로 Empty 상태가 구조적으로 발생하지 않는다. 해당 없음으로 적절.
- [x] Error 상태 (안내 + [다시 시도]) 포함 -- 별도 Error 와이어프레임 제공: "설정을 불러오지 못했어요" + 오류 아이콘 + [다시 시도] CTA(--brand, --radius-md, 44px). 토글 API 실패 시 롤백 + 토스트, 닉네임 변경 실패 시 에러 메시지 + 시트 유지, 회원탈퇴 실패 시 다이얼로그 유지 + 토스트 등 개별 API 에러도 모두 커버.
- [x] read-only 상태가 필요한 화면에서 수정 UI 비노출 -- 해당 없음.

### C. 내비게이션 & 플로우
- [x] 하단 탭 4개 구조 일관성 -- SETTINGS는 navigation push 화면이므로 탭바 미표시. 설계 결정 사항 #2에서 근거가 명시됨. push 화면에서 탭바를 숨기는 것은 일반적 모바일 앱 패턴과 일치.
- [x] 유저 Flow맵과 진입/이탈 경로 일치 -- 진입: MYPAGE 계정 섹션 gear 버튼 (SS 19-D 일치). 이탈: 뒤로가기 -> MYPAGE (navigation pop), 로그아웃 -> HOME, 회원탈퇴 -> HOME. 유저플로우 v1.3.1 SS 11의 SETTINGS 매핑(저장/관리 여정 소속)과 일치.
- [x] 뒤로가기 동작 명시 여부 -- AppBar 좌측 뒤로가기 버튼: "탭 -> MYPAGE로 뒤로가기 (navigation pop)" 명확히 기술.
- [x] 플로우 단절 지점 없는가 -- 모든 상태(기본/로딩/에러)에서 뒤로가기 가능. 로그아웃/회원탈퇴 후 HOME 이동. 닉네임 시트 닫기 3가지 방법(X/바깥 탭/아래 스와이프) 모두 명시. 단절 지점 없음.

### D. UX 품질
- [x] 터치 타겟 최소 44px 준수 -- 뒤로가기 44x44px, 토글 44x44px 터치 영역 명시(시각적 44x24px), NicknameRow 최소 60px, LogoutButton 52px, WithdrawButton 44px 행 높이, 닉네임 시트 close 44x44px, 다이얼로그 버튼 44px 높이. 접근성 노트에서도 "모든 인터랙티브 요소 최소 44x44px" 재확인.
- [x] 모바일 퍼스트 (375px 기준) 레이아웃 -- 와이어프레임이 375px 기준으로 작성됨.
- [x] 작은 모바일 sentinel에서도 구조가 유지되는가 -- 320px 대응 섹션에서 각 컴포넌트별 대응 전략 기술: 토글 제목 2행 wrap, 닉네임 30자 말줄임, 바텀시트 여백 --space-3 축소, blocker 체크리스트 4항목 포함.
- [x] 핵심 액션이 시각적으로 명확한가 -- 닉네임 변경 CTA(--brand filled), 로그아웃 확인(--brand filled), 회원탈퇴 확인(--danger filled)으로 각 맥락에 맞는 primary CTA 색상 사용.
- [x] whole-page horizontal scroll을 유도하지 않는가 -- 스크롤 정책에서 "가로 스크롤 금지" 명시. 가로 스크롤을 유발할 요소가 없음.
- [x] scroll containment가 명확한가 -- 앱바 sticky, 설정 항목 세로 스크롤, 바텀시트/다이얼로그는 오버레이 모달. 스크롤 정책 테이블에서 각 영역이 명확히 정의됨.
- [x] AI스러운 제네릭 UI 사용 여부 -- 글로우, 과도한 그라디언트 없음. 토글 + 리스트 + 바텀시트 + 다이얼로그 패턴은 네이티브 앱 설정 화면의 전형적 구조(iOS 설정, Android Material).

### F. 디자인 토큰 준수 (design-tokens.md 기준)
- [x] CTA 버튼에 `--brand (#ED7470)` 사용됐는가 -- 닉네임 변경 [변경하기] CTA, 로그아웃 [로그아웃] 확인 버튼, Error [다시 시도] 버튼, 토글 ON 상태 모두 --brand 사용. 토글 ON 색상은 #ED7470으로 정확히 일치. 회원탈퇴 [탈퇴하기]는 --danger (#E03E3E) 사용(위험 행동이므로 적절한 예외).

## 부록: 검토 필요 항목에 대한 의견

설계 파일의 design-critic 검토 필요 항목에 대해:

| 요청 항목 | 의견 |
|----------|------|
| 로그아웃 버튼 "탭 가능" 어포던스 | 마이너 #1로 다룸. 현재 --surface + --text-2 중앙 정렬 텍스트는 iOS 설정 앱의 "로그아웃" 패턴과 유사하여 사용자가 버튼으로 인식할 가능성이 높다. pressed 피드백(--surface-fill)이 명시되어 있으므로 현재 설계로도 충분히 동작한다. |
| 회원탈퇴 --text-3 + 밑줄이 너무 약하지 않은가 | 적절한 균형. 회원탈퇴는 "찾으면 쓸 수 있지만 실수로 누르지 않는" 무게가 올바르다. 밑줄이 "탭 가능" 시그널을 제공하고, 확인 다이얼로그에서 --danger 색상으로 위험성을 명확히 강조한다. 이 이중 안전장치 패턴은 Kakao, Naver, LINE 등 국내 서비스의 회원탈퇴 UX와 일치한다. |
| 닉네임 바텀시트 키보드-CTA 관계 | 마이너 #2로 다룸. 설계 의도는 올바르나 구현 전략이 미기술. iOS Safari에서 visual viewport 이슈가 있으므로 Stage 4에서 VisualViewport API 또는 env(keyboard-inset-height) 활용을 권장한다. |
| 토글 설명 텍스트 충분성 | "요리 중 화면이 꺼지지 않아요"는 기능을 정확히 설명한다. 토글 제목("요리모드 화면 꺼짐 방지")과 설명이 각각 "무엇을"과 "결과"를 분담하므로 적절하다. |
| 로그아웃 후 HOME vs LOGIN 이동 | HOME 이동이 올바르다. HOME은 비로그인 사용자도 사용 가능한 화면이며, 로그아웃 후 즉시 LOGIN으로 보내면 "다시 로그인하라는 압박"으로 느껴질 수 있다. HOME 이동은 "로그아웃했지만 서비스를 계속 둘러볼 수 있다"는 자연스러운 경험을 제공한다. |
| 회원탈퇴 추가 확인 단계 (텍스트 재입력) | 현재 1단계 확인 다이얼로그면 충분하다. "탈퇴"/"회원탈퇴" 텍스트 재입력은 GitHub 레포지토리 삭제처럼 돌이킬 수 없는 공유 자원 삭제에 적합하지, 개인 계정 탈퇴에는 과도하다. --danger 색상 + "모든 데이터가 삭제되며 되돌릴 수 없어요" 경고문이 충분한 마찰을 제공한다. |
| Screen Wake Lock API 비지원 시 토글 숨김 vs 비활성 | 현재 설계(표시하되 탭 시 토스트)가 올바르다. 토글을 숨기면 설정 항목이 3개로 줄어 "원래 있어야 할 기능이 없다"는 혼란을 줄 수 있다. 서버에 설정값을 저장하되 클라이언트 미적용 방식은 다른 기기에서 로그인 시 자동 적용이 가능하다는 이점도 있다. |
| 설정 항목 4개 -- 세로 여백 과도 여부 | 현재 간격(섹션 간 --space-6/8, 로그아웃-회원탈퇴 간 --space-4)은 적절하다. 항목 수가 적다고 여백을 극단적으로 줄이면 각 항목이 붙어서 구분이 어려워진다. iOS/Android 네이티브 설정 화면도 항목이 적을 때 넉넉한 여백을 사용하며, 사용자에게 "이게 전부"라는 명확한 신호를 준다. |
